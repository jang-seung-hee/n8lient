# n8n 프로젝트 워크스페이스 룰 v1.0

## 1. 기본 원칙

이 워크스페이스는 n8n 자동화 워크플로우 개발, 수정, 점검을 위한 공간이다.
AI는 항상 안정성, 최소 변경, 운영 지속성을 우선한다.

n8n 워크플로우는 한 번 운영에 들어가면 작은 변경도 전체 자동화 실패로 이어질 수 있다.
따라서 기능 개선보다 기존 흐름 보존을 우선한다.

## 2. 최소 변경 원칙

워크플로우 JSON 또는 Code 노드를 수정할 때는 반드시 필요한 노드만 수정한다.
관련 없는 노드, 이름, 위치, 연결선, 자격증명, 표현식, UI 좌표를 임의로 바꾸지 않는다.

수정 전에는 먼저 아래 내용을 보고한다.

* 수정이 필요한 노드명.
* 수정 이유.
* 변경될 내용.
* 변경하지 않을 영역.

사용자 승인 전에는 실제 JSON 수정 파일을 생성하지 않는다.

## 3. 기존 구조 보존

기존 워크플로우의 처리 순서, 분기 구조, 에러 처리 흐름, 파일 이동 구조를 임의로 바꾸지 않는다.
리팩터링, 노드 통합, 노드 삭제, 이름 변경은 사용자가 명시적으로 요청한 경우에만 한다.

하드코딩 값, 자격증명, 폴더 ID, 이메일, Webhook URL을 발견하면 바로 수정하지 말고 먼저 보고한다.

## 4. 오류 분석 원칙

오류가 발생하면 추측으로 수정하지 않는다.
반드시 실제 오류 메시지, 실패 노드, 입력 데이터, 출력 데이터를 기준으로 원인을 판단한다.

원인이 불명확하면 가능한 원인을 나누어 설명하고, 가장 작은 확인 방법부터 제안한다.

## 5. Code 노드 규칙

Code 노드는 기존 입력/출력 구조를 보존한다.
반환 형식은 n8n 표준에 맞게 `return [{ json: ... }]` 구조를 유지한다.

기존 필드명을 임의로 변경하지 않는다.
새 필드가 필요하면 기존 필드를 보존한 상태에서 추가한다.

## 6. 자격증명과 보안

API Key, Token, Webhook Secret, Google 자격증명은 Code 노드에 하드코딩하지 않는다.
가능하면 n8n Credentials, 환경변수, 외부 설정 DB를 사용한다.

자격증명 변경, Webhook URL 변경, 환경변수 추가는 반드시 사용자 확인 후 진행한다.

## 7. 파일과 데이터 처리

Google Drive, Gmail, Calendar, Firebase, Sheets 관련 노드는 폴더 ID, 파일 ID, 시트 ID, 이메일 주소를 정확히 보존한다.
정상 완료, 처리 실패, 처리 제외 폴더 이동 규칙을 임의로 바꾸지 않는다.

파일 처리 자동화에서는 중복 처리, 처리 제외, 실패 이동, 로그 기록 기준을 명확히 유지한다.

## 8. 결과물 제공

수정이 필요한 경우에는 먼저 변경 계획을 제시한다.
사용자가 승인하면 수정된 JSON 파일을 생성한다.

최종 보고에는 아래 내용을 포함한다.

* 수정한 노드.
* 수정 내용.
* 변경하지 않은 영역.
* 테스트 또는 수동 확인 방법.
* 남은 주의사항.
