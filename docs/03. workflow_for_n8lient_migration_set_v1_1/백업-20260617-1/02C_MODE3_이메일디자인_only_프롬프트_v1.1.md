# MODE 3 — 이메일 리포트 디자인 Only 프롬프트 v1.0

이 문서는 이미 동작 검증된 n8n 워크플로우의 이메일 리포트 디자인만 개선하는 디자인 패치 모드다.
N8Lient 커스터마이징, Gateway 연동 구조 변경, Processor 변경, callback 변경은 수행하지 않는다.

## 필수 첨부

```text
1. 수정 대상 n8n 워크플로우 JSON
2. 이메일 프레임 템플릿 HTML
3. 이메일 디자인 규칙 Design.md
4. 이메일 디자인 작업 프롬프트
5. 콘텐츠 패턴 예시 HTML 또는 샘플 파일은 선택
```

파일명은 고정하지 않는다. 파일의 역할과 내용 기준으로 식별한다.

## 작업 목표

- 기존 워크플로우의 데이터 처리 로직은 변경하지 않는다.
- 기존 이메일 본문에 들어가던 정보와 의미는 유지한다.
- 공통 헤더, 상태바, 푸터는 이메일 프레임 템플릿을 적용한다.
- 본문은 Design.md 기준으로 문서형 리포트 구조로 재구성한다.
- 실제 n8n에 import 가능한 JSON 파일을 생성한다.

## 수정 허용 범위

```text
- Notify / Email 영역의 HTML 프레임
- CONTENT_HTML 본문 구조
- 상태바/헤더/푸터 placeholder 매핑
- 링크 표시 방식
- 모바일 대응 HTML 스타일
- 기존 결과 데이터를 보기 좋게 재배치하는 코드
- 이메일 전송용 임시 MD 첨부 처리
- 이미 이메일 영역으로 전달되는 입력 원본 binary의 이메일 임시 첨부 처리
```

## 수정 금지 범위

```text
- Webhook / Trigger 구조
- Payload Normalize / Input Validate 로직
- 파일 수신/정규화 로직
- Processor / Gemini / 외부 API 처리 로직
- processorResult 구조
- Result Policy Router
- Gateway callback payload
- DB/Storage 저장 정책
- Google Drive Optional Export 구조
- inputSchema / configSchema / retentionPolicy 계약
- Credential / Token / Secret / API Key 관련 설정
```

## MD 및 입력 원본파일 첨부 관련 주의

이메일 디자인 개선 과정에서 MD 첨부를 다룰 수 있다.
단, 디자인 Only 모드에서 허용되는 MD 첨부는 이메일 발송용 임시 결과물에 한정한다.

이미 기존 워크플로우의 Notify / Email 영역으로 입력 원본 binary(file/image/audio 등)가 전달되고 있는 경우, 해당 원본 binary를 이메일에 임시 첨부하도록 정리할 수 있다. 그러나 디자인 Only 모드에서는 Webhook, 파일 수신/정규화, Processor, Result Policy Router, callback 구조를 바꾸기 위해 원본 첨부 기능을 새로 확장하지 않는다.

MD 첨부와 입력 원본파일 첨부는 이메일 전송용 임시 첨부일 뿐이다. 이 작업 때문에 resultRefs, originalFileRefs, Firebase Storage, Google Drive 저장, callback payload 구조를 변경하지 않는다.

## 작업 절차

1. n8n JSON에서 이메일 리포팅 관련 노드를 찾는다.
2. 기존 이메일 본문에 들어가는 데이터와 의미를 목록화한다.
3. 이메일 프레임 템플릿 HTML, Design.md, 이메일 디자인 작업 프롬프트의 역할을 확인한다.
4. 수정 대상 노드와 수정 이유를 먼저 보고한다.
5. 사용자가 승인하기 전에는 JSON을 수정하거나 새 JSON을 생성하지 않는다.
6. 승인 후 Notify / Email 영역만 수정한다.
7. 최종 JSON의 import 가능성을 점검한다.

## 1차 보고 형식

수정 전 1차 보고에는 아래를 포함한다.  
사용자가 승인하기 전에는 JSON을 수정하지 않는다.

### 노드 변경 검토표 (필수)

- **대상**: 워크플로우 노드
- **작업 시점**: JSON 수정이 필요할 때
- n8n 노드 **순번(01, 02, …)** 순으로 **빠진 노드 없이** 전 행을 작성한다.
- **산출물 우선순위** (1차 보고·승인 전):
  1. **1안 (필수 시도)**: 노드 변경 검토표를 **Excel 파일(.xlsx)** 로 생성해 사용자가 다운로드할 수 있게 제공한다. 프로젝트 `scripts/` 또는 사용자 지정 폴더에 저장하고, 저장 경로와 파일명을 채팅에 안내한다.
  2. **2안**: .xlsx 생성이 불가하면 **단일 HTML 파일**로 생성해 다운로드·열기 가능하게 제공한다.
  3. **3안**: 파일 생성도 불가하면 **채팅에 표(Markdown/HTML 표)** 로 전체 노드를 표시한다.
- **표 컬럼**: `순번 | 노드명 | 변경 전 | 변경 후 | 변경 내용`
- **파일명 예시**: `{워크플로우명}_노드변경검토표_1차보고.xlsx` 또는 `.html`
- 수정 불필요 행: 배경색 없음. 변경 전·후는 「동일」 또는 「-」
- **주황색 배경**은 Notify / Email 관련 노드 중 실제 수정 대상만 표시한다. 그 외 업무·Processor·callback 노드는 수정 없음(일반 행)으로 둔다. (Excel: Fill `#FFE0B2`, HTML: `<tr style="background-color:#FFE0B2">`)
- 표(또는 파일) 하단/본문 마지막에 **승인 요청 문장** 1줄
- **1안·2안**을 사용한 경우, 채팅에는 파일 제공 사실과 요약(수정 필요 노드 개수 등)만 적고 **전체 표를 중복 붙이지 않는다**.

### 이메일 디자인 요약 (검토표 외)

```text
1. 현재 워크플로우의 이메일 리포팅 구조
2. 기존 이메일 본문에 포함되는 정보
3. 디자인 참고 파일 확인 결과
4. MD 첨부 및 입력 원본파일 임시 첨부 처리 여부
5. Credential 재연결 필요 여부
6. 남은 위험 요소
```

## 이메일 디자인 핵심 규칙 요약

아래 규칙은 첨부된 Design.md와 충돌하지 않는 범위에서 적용한다. 충돌 시 첨부된 최신 Design.md를 우선한다.

```text
- 이메일 본문은 카드형 뉴스레터가 아니라 문서형 리포트로 구성한다.
- 공통 프레임 헤더와 같은 제목을 본문에서 반복하지 않는다.
- 큰 카드 박스, 그림자 박스, URL 전체 노출, 장식용 아이콘/이모지는 사용하지 않는다.
- 링크는 의미 있는 텍스트 링크로 표현한다.
- 본문 역할 라벨과 태그 스타일을 구분한다.
- 모바일에서는 큰 여백과 중첩 테이블을 피한다.
```
