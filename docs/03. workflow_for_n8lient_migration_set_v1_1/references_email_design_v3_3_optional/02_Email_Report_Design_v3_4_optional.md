# Dansoft AI Automation Email Report Design.md v3.4

## 1. 목적
이 문서는 AI가 n8n 워크플로우의 이메일 본문을 생성하거나 수정할 때 따라야 하는 공통 디자인 규칙이다.

중요한 전제:
- 실제 워크플로우의 데이터 처리 로직은 변경하지 않는다.
- 헤더, 상태바, 푸터는 HTML 프레임 템플릿이 담당한다.
- 본문은 이 문서의 원칙과 `04_Content_Pattern_Examples_v3_4_optional.html`의 실제 패턴을 함께 기준으로 생성한다.
- 최종 사용 가능한 HTML 구조의 기준은 `04_Content_Pattern_Examples_v3_4_optional.html`이다.
- 사용자가 예시로 언급한 실제 제목명은 고정 규칙이 아니다. 이 문서에서는 보편적인 명칭인 `본문 제목`, `설명문`, `섹션 표제`, `항목 제목`, `본문 역할 라벨`, `메타 정보`, `태그`, `텍스트 링크`, `순번 목록`, `블릿 목록`, `수평바` 등을 사용한다.

---

## 2. 전체 방향
이메일 본문은 카드형 뉴스레터가 아니라 문서형 리포트로 구성한다.

다만 문서형이라고 해서 디자인을 포기하지 않는다. 다음 요소로 시각적 완성도를 만든다.

- 공통 프레임 헤더
- 상태바
- 스타일이 있는 섹션 표제
- 명확한 항목 제목
- 본문 역할 라벨
- 메타 정보
- 작은 태그 배지
- NEW / IMPORTANT / WARNING 같은 상태 배지
- 순번 목록과 블릿 목록
- 텍스트 링크
- 얇은 구분선과 굵은 수평바
- 충분한 줄간격

---

## 3. 디자인 사용 제한 원칙
AI는 본문을 만들 때 `04_Content_Pattern_Examples_v3_4_optional.html`에 실제로 들어 있는 HTML 패턴만 사용한다.

규칙:
- 샘플 파일에 없는 레이아웃, 박스, 카드, 테이블, 컬럼, 장식 요소를 새로 만들지 않는다.
- `02_Email_Report_Design_v3_4_optional.md`는 원칙 문서이고, 실제 사용 가능한 HTML 요소의 최종 기준은 `04_Content_Pattern_Examples_v3_4_optional.html`이다.
- 필요한 정보가 많더라도 새로운 표나 박스 구조를 임의로 만들지 않는다.
- 정보가 많으면 섹션 표제, 항목 제목, 메타 정보, 본문 역할 라벨, 순번 목록, 블릿 목록, 태그, 상태 배지, 텍스트 링크, 구분선, 수평바를 조합한다.
- 샘플 파일에 없는 구조가 필요해 보이면 새 구조를 만들지 말고 기존 텍스트/목록 구조로 풀어서 작성한다.

---

## 4. 금지 규칙
본문 `CONTENT_HTML` 안에서 다음 요소는 사용하지 않는다.

1. 큰 카드 박스
2. 기사별 둥근 테두리 박스
3. 그림자 박스
4. 배경색이 넓게 깔린 콘텐츠 박스
5. 링크 URL을 감싸는 큰 박스
6. URL 전체 문자열 노출
7. 섹션명 앞의 장식용 아이콘 또는 이모지
   - 예: 시계, 지구, 톱니바퀴, 나침반, 불꽃 등
8. 2단/3단 멀티컬럼 레이아웃
9. 모바일에서 폭을 좁히는 중첩 구조
10. 공통 프레임 헤더와 같은 제목을 본문 최상단에서 반복 표시하는 것
11. 본문 안의 모든 표 구조
    - `<table>`, `<thead>`, `<tbody>`, `<tr>`, `<th>`, `<td>` 사용 금지
12. CSS grid, flex 기반 복합 레이아웃
13. position, float 기반 배치

주의:
- 프레임 템플릿 내부의 이메일 호환용 `<table>` 구조는 예외이다.
- 이 금지 규칙은 AI가 생성하는 `{{CONTENT_HTML}}` 내부에 적용된다.

---

## 5. 배경과 폭
- 이메일 전체 배경은 흰색을 기본으로 한다.
- PC에서는 왼쪽 정렬, 최대 폭 900px을 기준으로 한다.
- 모바일에서는 가능한 한 전체 폭을 사용한다.
- 모바일 본문 좌우 여백은 최소화한다.
- 외곽 회색 배경은 사용하지 않는다.

---

## 6. 본문 제목 사용 규칙
본문 제목은 리포트의 최상위 표제이다. 단, 공통 프레임 헤더에 이미 동일한 제목이 표시되는 경우 본문 제목을 다시 출력하지 않는다.

권장 판단:
- 프레임 헤더 제목과 본문 제목이 같으면 본문 제목은 생략한다.
- 본문은 바로 `요약`, `흐름`, `상세 내용` 같은 첫 섹션으로 시작한다.
- 프레임 헤더가 없는 독립 HTML에서만 본문 제목을 사용한다.

권장 스타일은 `04_Content_Pattern_Examples_v3_4_optional.html`의 `Report Title` 패턴을 사용한다.

---

## 7. 설명문 / 슬로건 사용 규칙
설명문 또는 슬로건도 공통 프레임 헤더의 부제와 중복되면 본문에서 생략한다.

권장 스타일은 `04_Content_Pattern_Examples_v3_4_optional.html`의 `Report Lead Text` 패턴을 사용한다.

---

## 8. 섹션 표제
섹션 표제는 `요약`, `주요 항목`, `상세 내용`, `대화 흐름`, `오류 정보` 같은 큰 구역의 제목이다.

규칙:
- 섹션 표제 앞에 장식용 아이콘을 붙이지 않는다.
- 상단 라인, 작은 영문/한글 라벨, 굵은 제목으로 위계를 만든다.
- 배경색이 있는 큰 박스형 섹션 헤더는 사용하지 않는다.
- 섹션 표제 바로 위에 별도의 회색 구분선을 중복 배치하지 않는다.
- 색상 변형이 필요하면 샘플 파일에 있는 섹션 표제 변형만 사용한다.

---

## 9. 항목 제목
항목 제목은 각 기사, 통화 흐름 항목, 오류 항목, 세부 요약 항목의 제목이다.

규칙:
- 항목 제목은 섹션 표제보다 작게 둔다.
- 번호는 필요할 때만 사용한다.
- 제목 앞 장식용 아이콘은 사용하지 않는다.
- NEW / IMPORTANT / WARNING 같은 상태 표시는 제목 앞 아이콘이 아니라 인라인 상태 배지로 처리한다.

---

## 10. 본문 역할 라벨
`핵심 포인트`, `요약`, `원문`, `비즈니스 관점`, `한계점`, `실행 결과`처럼 문단의 역할을 알려주는 표지는 태그보다 한 단계 높은 본문 역할 라벨로 처리한다.

규칙:
- 태그와 같은 연한 pill 스타일을 기본으로 쓰지 않는다.
- 진한 색상의 작은 라벨을 사용한다.
- 모서리 곡선은 최소화한다.
- 큰 박스나 큰 배경 영역을 만들지 않는다.
- 라벨 앞에 장식용 아이콘을 붙이지 않는다.
- 색상 변형은 샘플 파일의 `Role Label Variants`만 사용한다.

---

## 11. 메타 정보
출처, 중요도, 게시일, 통화일시, 실행 시간, 워크플로우명 같은 보조 데이터는 메타 정보로 처리한다.

규칙:
- 본문보다 작고 옅게 표현한다.
- 중요한 데이터는 숨기지 않는다.
- 데이터가 많은 경우 표로 만들지 말고 여러 줄의 메타 정보 또는 키-값 텍스트로 분산한다.
- `항목: 값 · 항목: 값` 형태를 기본으로 한다.

---

## 12. 태그와 상태 배지
태그는 작은 인라인 배지로만 사용한다.

규칙:
- 태그는 작게 유지한다.
- 태그를 큰 박스 영역으로 만들지 않는다.
- 태그가 많으면 여러 줄로 자연스럽게 흐르게 둔다.
- `핵심 포인트`, `요약`, `원문` 같은 본문 역할 라벨과 같은 스타일을 쓰지 않는다.
- 색상 변형은 샘플 파일의 `Tag Variants`만 사용한다.
- NEW, IMPORTANT, WARNING 같은 상태는 샘플 파일의 `Status Badge Variants`만 사용한다.

권장 용도:
- NEW: 새로 추가된 항목
- IMPORTANT: 중요도가 높거나 사용자가 반드시 확인해야 하는 항목
- WARNING: 오류, 누락, 실패, 주의가 필요한 항목
- REQUIRED: 필수 입력 또는 필수 설정
- OPTIONAL: 선택 입력 또는 선택 설정

---

## 13. 목록 스타일
표를 대신해 순번 목록과 블릿 목록을 우선 사용한다.

순번 목록 권장 용도:
- 처리 순서
- 실행 단계
- 우선순위
- 할 일 목록
- 비교 항목의 구조화

블릿 목록 권장 용도:
- 요약
- 특징
- 주의사항
- 로그 요지
- 고객 안내 항목

규칙:
- 목록은 `<ol>`, `<ul>`, `<li>`만 사용한다.
- 목록 안에 표, 카드, 박스, 컬럼을 넣지 않는다.
- 목록 항목은 짧고 명확하게 작성한다.
- 숫자 비교도 표 대신 `항목명: 값` 형태의 목록으로 풀어쓴다.

---

## 14. 링크
링크는 URL 문자열을 그대로 노출하지 않는다. 링크 색상은 푸른색 계열을 기본으로 한다.

규칙:
- 링크 텍스트는 의미 있게 작성한다.
  - 예: `원문 링크 보기`, `Google Drive 원본 열기`, `상세 보고서 열기`, `캘린더 일정 열기`
- URL 전체 문자열은 본문에 노출하지 않는다.
- 링크 박스를 만들지 않는다.
- 본문 링크 색상은 `#2563eb`을 기본으로 한다.

---

## 15. 구분선과 수평바
섹션과 항목 사이에는 얇은 구분선 또는 굵은 수평바를 사용할 수 있다.

규칙:
- 큰 박스보다 구분선과 여백을 우선한다.
- 같은 섹션 안의 항목 사이에는 얇은 회색 구분선을 사용할 수 있다.
- 큰 구역 전환에는 굵은 수평바를 사용할 수 있다.
- 섹션의 마지막 항목 뒤에는 불필요한 구분선을 넣지 않는다.
- 다음 섹션 표제의 상단 라인이 있으면 그 라인 하나만 섹션 구분선으로 사용한다.
- 수평바는 샘플 파일의 `Horizontal Bar Variants`만 사용한다.

---

## 16. 표 사용 규칙
본문에서는 표를 사용하지 않는다.

금지 대상:
- `<table>`
- `<thead>`
- `<tbody>`
- `<tr>`
- `<th>`
- `<td>`
- 표처럼 보이도록 만든 2열/3열 박스 구조
- 항목과 값을 좌우로 나누는 복합 레이아웃

대체 방법:
- 숫자 비교표 → 순번 목록 또는 블릿 목록
- 정산표 → `항목: 값` 형태의 키-값 텍스트 목록
- 오류 로그 → 발생 시각 / 워크플로우명 / 오류 메시지 / 영향 범위 순서의 문단과 목록
- 설정값 목록 → 메타 정보 또는 블릿 목록
- 처리 결과 → 상태 배지 + 짧은 문단 + 목록

## 18. AI 적용 지침
AI는 다음을 반드시 지킨다.

1. 기존 워크플로우의 데이터 처리 로직은 수정하지 않는다.
2. 기존 본문의 의미와 데이터는 유지한다.
3. HTML 프레임은 `01_Email_Frame_Template_v3_4_optional_dansoft_footer.html`을 사용한다.
4. 본문은 `02_Email_Report_Design_v3_4_optional.md`의 원칙과 `04_Content_Pattern_Examples_v3_4_optional.html`의 실제 패턴 기준으로 재구성한다.
5. 실제 사용 가능한 HTML 구조의 최종 기준은 `04_Content_Pattern_Examples_v3_4_optional.html`이다.
6. 예시의 실제 제목명을 고정하지 말고, 보편적인 역할명 기준으로 디자인을 적용한다.
7. 공통 프레임 헤더와 동일한 본문 제목은 반복하지 않는다.
8. 섹션명 앞 장식용 아이콘은 제거한다.
9. `핵심 포인트`, `요약`, `원문`은 태그가 아니라 본문 역할 라벨로 처리한다.
10. 긴 URL은 의미 있는 텍스트 링크로 감춘다.
11. 본문 링크 색상은 푸른색 계열로 처리한다.
12. 모바일 여백을 줄이기 위해 큰 카드, 큰 박스, 중첩 구조를 만들지 않는다.
13. 본문 안에서는 테이블을 절대 생성하지 않는다.
