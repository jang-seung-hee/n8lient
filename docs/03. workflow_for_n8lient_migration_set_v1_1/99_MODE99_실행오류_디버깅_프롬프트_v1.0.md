# MODE 99 — N8Lient 실행 오류 디버깅 / 장애 원인 분석 프롬프트 v1.0

이 문서는 N8Lient에 등록·배포된 n8n 워크플로우를 앱에서 실행한 뒤 실패, 오작동, 결과 누락, callback 실패, 이메일 발송 실패, 첨부파일 누락 등이 발생했을 때 원인을 분석하기 위한 디버깅 전용 프롬프트다.

이 모드는 기존 Workflow for N8Lient 문서 세트와 분리된 새 체계가 아니다.  
아래 문서를 그대로 참조한다.

```text
- 최신 N8Lient 기초설계서 문서세트 01~09
- 09_엔팔라이언트_워크플로우_버전관리_운영_계약
- 02A/02B/02C 커스터마이징 및 이메일 디자인 프롬프트
- 03_MODE4 워크플로우 등록 가이드 / Import JSON 생성 프롬프트
- 최종 n8n 워크플로우 JSON
- N8Lient 워크플로우 등록 정보
- 앱의 submissions 실행 로그와 errorDetails
```

---

## 0. 목적

이 모드의 목적은 특정 에러를 감으로 수정하는 것이 아니다.

목적은 아래 순서로 장애를 좁히는 것이다.

```text
1. 어떤 실행 건인지 식별한다.
2. 어느 구간에서 실패했는지 판정한다.
3. 등록값 문제인지, 앱/Gateway 문제인지, n8n JSON 문제인지 구분한다.
4. 수정이 필요하면 어떤 작업 모드로 넘길지 결정한다.
5. 수정 전에는 반드시 원인 분석과 변경 대상 후보를 보고한다.
```

---

## 1. 최우선 원칙

```text
1. 디버깅 모드에서는 처음부터 n8n JSON을 수정하지 않는다.
2. 디버깅 모드에서는 앱 코드, Gateway 코드, Firestore Rules, Import JSON을 임의로 수정하지 않는다.
3. 먼저 실패 구간을 판정한다.
4. 원인 후보를 가능성 순서대로 제시한다.
5. 수정이 필요하면 사용자의 승인을 받은 뒤 다음 작업 모드로 연결한다.
6. Credential, Token, Secret, API Key, Authorization Header는 절대 출력하지 않는다.
7. 원본 파일 binary, settings 민감값, 전체 인증 URL은 출력하지 않는다.
8. 장애 수정이 필요한 경우 09번 버전관리 운영 계약에 따라 PATCH / MINOR / MAJOR를 판단한다.
```

```text
   - 실패 메시지

```

## 3. 자료 확인 후 사용자에게 물어볼 질문

필요 자료가 확인되면 바로 분석하지 말고, 사용자가 어떤 현상을 문제로 보고 있는지 먼저 묻는다.

```text
자료 확인이 완료되었습니다. 어떤 현상 또는 버그를 분석할까요?

예시:
- 실행 시 404가 발생합니다.
- 앱에서는 처리중인데 n8n은 성공으로 끝났습니다.
- n8n은 성공했는데 앱 결과가 갱신되지 않습니다.
- 이메일은 왔는데 MD 첨부파일이 없습니다.
- 원본 음성/이미지가 이메일에 첨부되지 않습니다.
- processed_result인데 DB 결과 본문이 없습니다.
- full_archive인데 originalFileRefs/resultRefs가 없습니다.
- Google Drive Optional Export 파일이 생성되지 않습니다.
- 사용자 화면과 회사관리자 화면의 결과가 다르게 보입니다.
```

## 4. 실패 구간 분류 기준

분석 시 아래 구간 중 하나 이상으로 분류한다.

```text
APP_UI
- 실행 버튼, 입력 폼, 파일 선택, 결과 화면 표시 문제

API_ROUTE_VALIDATE
- /api/automation/execute의 입력 검증, schema 검증, 인증/권한 문제

API_ROUTE_GATEWAY_CALL
- App API Route가 Gateway 호출에 실패한 문제

GATEWAY_VALIDATE
- Gateway의 사용자 승인, 회사/개인 설정 병합, contractRetentionLimit, retentionPolicy 계산 문제

GATEWAY_STORAGE
- Gateway의 파일 수신, 임시 파일 처리, Storage 업로드/보관 정책 문제

GATEWAY_N8N_CALL
- Gateway가 n8n Webhook 호출 중 실패
- 404, 401/403, timeout, 500 등

N8N_WEBHOOK_NORMALIZE
- Webhook payload 파싱, multipart payload 파싱, binary 정규화 문제

N8N_INPUT_VALIDATE
- n8n 내부 방어 검증 문제
- input.title 오판, requiredInputTypes 오판 등

N8N_PROCESSOR
- Gemini/LLM/외부 API/업무 처리 노드 실패

N8N_EMAIL_ATTACHMENT
- Notify / Email / MD 첨부 / 원본 입력파일 임시 첨부 문제

N8N_RESULT_POLICY
- Result Policy Router, DB/Storage/Optional Export 분기 문제

N8N_CALLBACK
- n8n callback HTTP Request 문제

GATEWAY_CALLBACK
- Gateway callback route 수신/검증/Firestore 업데이트 문제

FIRESTORE_UI
- Firestore 저장은 됐으나 앱 결과 화면 표시/필터/권한 문제
```

## 5. 대표 에러별 1차 판정 규칙

### 5.1 404 + GATEWAY_EXECUTE_FAILED

```text
우선 GATEWAY_N8N_CALL 문제로 본다.
확인 순서:
1. n8n 워크플로우 Active 여부
2. Webhook Path 일치 여부
3. production webhook URL과 test webhook URL 혼동 여부
4. Gateway n8n base URL 설정
5. N8Lient 등록 정보의 webhookSecretId / webhookPath
6. workflowVersion 변경 후 등록값 갱신 누락 여부
```

### 5.2 앱에서는 처리중인데 n8n은 성공

```text
우선 N8N_CALLBACK 또는 GATEWAY_CALLBACK 문제로 본다.
확인 순서:
1. n8n callback HTTP Request 응답 코드
2. payload.callbackUrl 사용 여부
3. callback Authorization Credential 연결 여부
4. submissionId 일치 여부
5. Gateway callback route 로그
6. Firestore submissions 상태 업데이트 여부
```

### 5.3 이메일은 왔지만 앱 결과가 없음

```text
retentionPolicy와 callback payload 문제로 본다.
확인 순서:
1. retentionLevel이 notify_only인지 processed_result인지 확인
2. processed_result 이상인데 processorResult가 callback에 포함됐는지 확인
3. notify_only이면 DB 결과 본문 없음이 정상인지 확인
4. Result Policy Router 분기 확인
```

### 5.4 MD 첨부 누락

```text
Notify / Email 영역 문제로 본다.
확인 순서:
1. emailAttachResultMd 또는 워크플로우 정책 확인
2. MD binary 생성 노드 존재 여부
3. Email 노드 attachments 매핑 확인
4. notify_only/processed_result/full_archive에서도 이메일 옵션이 유지되는지 확인
```

### 5.5 원본 음성/이미지/파일 이메일 첨부 누락

```text
Notify / Email 원본 입력파일 임시 첨부 문제로 본다.
확인 순서:
1. Webhook multipart binary가 n8n에 들어왔는지 확인
2. 원본 binary가 Email Attach 브랜치까지 유지되는지 확인
3. Email Original Attachment Prepare 노드 존재 여부
4. Email 노드 attachmentBinaryProperty 매핑 확인
5. 파일 크기 제한 또는 이메일 첨부 제한으로 생략됐는지 확인
```

### 5.6 title 관련 오류

```text
inputSchema/title 계약 문제로 본다.
확인 순서:
1. titleRequired 값
2. input.title이 사용자 제목인지 여부
3. titleProvided/titleSource 전달 여부
4. submissionTitle/displayTitle이 input.title로 오염됐는지 확인
5. Gateway/n8n 방어 검증에서 input.title을 무조건 필수로 보지 않는지 확인
```

## 6. 분석 순서

```text
1. submissionId, workflowKey, automationId, workflowVersion을 확인한다.
2. errorDetails.phase / source / httpStatus / hint를 먼저 본다.
3. errorDetails가 없으면 error.code / error.message와 화면 캡처를 기준으로 레거시 분석한다.
4. n8n 실행 이력이 있는지 확인한다.
   - 실행 이력 없음: Webhook Path, Active, Gateway → n8n 호출 문제 우선
   - 실행 이력 있음: 실패 노드와 callback 여부 확인
5. N8Lient 등록 정보와 n8n JSON의 Webhook Path, workflowKey, workflowVersion, n8nServerKey를 비교한다.
6. inputSchema, configSchema, retentionCapabilities, operatorRetentionPolicy와 실제 실행 조건을 비교한다.
7. Result Policy Router, Notify / Email, Callback 노드의 정합성을 확인한다.
8. 원인 후보를 가능성 순서대로 정리한다.
9. 수정이 필요한 경우 변경 대상과 영향도를 보고한다.
10. PATCH / MINOR / MAJOR 판단을 포함한다.
```

## 8. 다음 작업 모드 연결 기준

디버깅 결과에 따라 다음 모드로 연결한다.

```text
N8Lient 등록 정보가 틀린 경우
→ Mode 4 등록 가이드 / Import JSON 재생성 또는 등록값 수정 권고

n8n 표준 구조가 틀린 경우
→ Mode 2 N8Lient 커스터마이징 Only로 PATCH 수정 권고

n8n 이메일 디자인/첨부 영역만 문제인 경우
→ Mode 3 이메일 디자인 Only 또는 Notify / Email PATCH 권고

커스터마이징과 이메일 영역이 함께 틀린 경우
→ Mode 1 커스터마이징 + 디자인 동시 권고

App / Gateway 코드 문제인 경우
→ 앱/Gateway 개발 이슈로 분리하고 n8n JSON은 수정하지 않음

단순 운영 문제인 경우
→ n8n Active ON, production webhook URL 확인, Gateway 재배포, App 재배포, 등록값 수정 같은 운영 조치 우선
```

## 9. 버전관리 판단

장애 수정도 09번 버전관리 운영 계약을 따른다.

```text
PATCH:
- 버그 수정
- null/빈값 방어
- Webhook Path 등록값 수정
- Email attachment 매핑 오류 수정
- Callback payload 누락 수정
- 기존 inputSchema/callback/retentionPolicy 계약을 깨지 않는 수정

MINOR:
- 기존 구조와 호환되는 선택 기능 추가
- 이메일 원본 첨부 옵션 신규 추가
- Optional Export 옵션 추가
- 기존 고객 기본 동작을 바꾸지 않는 확장

MAJOR:
- inputSchema 구조 변경
- callback payload 구조 변경
- retentionPolicy 구조 변경
- 필수 설정값 추가
- 기존 고객 설정과 호환되지 않는 변경
```

기존 고객에게 적용할 때는 아래를 반드시 판단한다.

```text
- backwardCompatible 여부
- migrationRequired 여부
- 기존 고객 자동 적용 가능 여부
- 선택 업그레이드 필요 여부
- rollbackTarget
```

## 11. 최종 원칙

```text
디버깅 모드는 수정 모드가 아니라 진단 모드다.
수정은 원인 판정 이후에만 한다.
수정이 필요하면 기존 Mode 1~4 중 적절한 모드로 연결한다.
새 문서를 따로 놀게 만들지 말고, 기존 N8Lient 기초설계서 01~09와 Workflow for N8Lient 문서 세트를 기준으로 판단한다.
```
