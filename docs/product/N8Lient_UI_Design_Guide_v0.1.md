# N8Lient UI Design Guide v0.1

## 1. 문서 목적

이 문서는 엔팔라이언트(N8Lient) MVP의 UI 방향을 고정하기 위한 디자인 가이드이다.

엔팔라이언트는 n8n 자동화 워크플로우를 고객이 직접 실행·설정·확인하는 업무용 웹앱이다.  
디자인 목표는 장식성보다 업무 효율, 정보 밀도, 실행 실수 방지에 둔다.

기본 톤은 Cal.com 계열의 깔끔한 B2B SaaS 스타일을 참고하되, 엔팔라이언트 사용자 화면은 업무 앱 성격이 강하므로 모바일에서는 여백과 라운드를 줄인 Compact App Density를 적용한다.

---

## 2. 전체 디자인 방향

### 2.1 기본 톤

- 흰색 캔버스 기반의 깔끔한 SaaS UI를 사용한다.
- 검정 또는 짙은 회색 CTA를 기본 액션 색상으로 사용한다.
- 과한 그라데이션, 유리 효과, 강한 그림자, 과도한 둥근 카드 스타일은 사용하지 않는다.
- 사용자는 자동화를 빠르게 실행하고 결과를 확인해야 하므로, 정보 탐색보다 업무 처리 속도를 우선한다.

### 2.2 디자인 우선순위

1. 업무 흐름이 명확해야 한다.
2. 모바일에서 한 화면에 보이는 정보량을 최대화해야 한다.
3. PC 사용자 화면은 모바일 구조를 넓은 화면에 맞게 확장해야 한다.
4. 관리자/운영자 화면은 사용자용과 분리된 콘솔형 UI로 설계한다.
5. 상태값은 한눈에 구분되어야 한다.

---

## 3. 레이아웃 원칙

## 3.1 사용자용 모바일

사용자용 모바일은 하단 바텀 메뉴 중심의 앱형 구조로 설계한다.

### 기본 메뉴

```text
홈 / 실행 / 결과 / 내정보
```

### 화면 원칙

- 단일 컬럼 구조를 사용한다.
- 화면 좌우 여백을 최소화한다.
- 카드 간 간격을 줄인다.
- 모서리 라운드는 작게 사용한다.
- 한 줄에 최대한 많은 정보를 정돈해서 보여준다.
- 실행 버튼은 명확하되, 카드나 리스트는 과하게 크지 않게 만든다.
- 사용자는 모바일에서 주로 실행, 결과 확인, 재전송을 수행한다.

---

## 3.2 사용자용 PC

사용자용 PC는 모바일 사용자 화면을 넓은 화면에 맞게 확장한 형태로 설계한다.  
PC 사용자 화면을 관리자 콘솔형 사이드바 구조로 바꾸지 않는다.

### 구조

```text
상단바
좌측 데이터 분석활용 패널 + 본문
하단 바텀 메뉴
```

### 좌측 패널

좌측 패널은 자동화 실행 메뉴가 아니다.  
좌측 패널은 자동화 결과 데이터를 검색·분석·활용하기 위한 진입 메뉴이다.

```text
[데이터 분석활용]

* 통자요
* 회의록
* 아이디어 캐치
* 업무위키
```

### 본문

좌측 항목을 클릭하면 본문은 해당 자동화의 데이터 활용 화면으로 바뀐다.

예시:

```text
통자요 선택
→ 통화 기록 검색
→ 통화 요약 목록
→ 실패/처리제외
→ 추후 통화 분석 기능
```

### 하단 메뉴

PC에서도 사용자용 화면은 하단 바텀 메뉴를 유지한다.

```text
홈 / 실행 / 결과 / 내정보
```

자동화 실행은 좌측 패널이 아니라 하단 메뉴의 `실행`에서 처리한다.

---

## 3.3 회사 관리자용

회사 관리자용은 별도 페이지로 구성한다.

### 기본 구조

- 좌측 사이드바
- 상단 사용자/회사 정보
- 본문 관리 영역

### 주요 메뉴

```text
회사 정보
사용자 승인
사용자 목록
계약 자동화 목록
자동화 설정
실행 결과
실패/처리제외 관리
```

회사 관리자용은 사용자용과 같은 레이아웃을 사용하지 않는다.

---

## 3.4 운영자용

운영자용은 별도 페이지로 구성한다.

### 주요 메뉴

```text
회사 관리
자동화 템플릿 관리
계약 자동화 관리
Webhook 관리
전체 실행 로그
시스템 설정
```

운영자용은 엔팔라이언트 사업자 콘솔이다.  
전체 회사와 자동화 템플릿을 관리하는 콘솔형 UI로 설계한다.

---

# 4. Compact App Density 규칙

엔팔라이언트 모바일 화면은 일반 SaaS 마케팅 페이지보다 훨씬 조밀하게 설계한다.  
목표는 화면 안에 더 많은 업무 정보를 보여주는 것이다.

## 4.1 모바일 여백

### 권장값

```text
페이지 좌우 여백: 12px
섹션 간격: 12px ~ 16px
카드 내부 여백: 10px ~ 12px
리스트 아이템 내부 여백: 8px ~ 10px
카드 간 간격: 8px
폼 필드 간 간격: 8px
```

### 금지

```text
모바일 카드 내부 padding 24px 이상 금지
모바일 섹션 간격 32px 이상 금지
모바일에서 마케팅 페이지식 넓은 여백 사용 금지
```

---

## 4.2 모서리 라운드

모바일에서는 모서리를 과하게 둥글게 만들지 않는다.  
업무용 앱처럼 단단하고 정돈된 느낌을 유지한다.

### 권장값

```text
작은 요소: 4px
입력창/버튼: 6px
카드: 8px
큰 패널: 10px
배지: 999px 또는 4px 중 용도에 따라 선택
```

### 금지

```text
일반 카드에 16px 이상 라운드 사용 금지
모든 컴포넌트를 pill 형태로 만드는 것 금지
소비자 앱처럼 둥글둥글한 카드 스타일 금지
```

---

## 4.3 줄 간격과 텍스트 밀도

모바일에서는 문장 줄 간격을 지나치게 넓게 잡지 않는다.

### 권장값

```text
본문 line-height: 1.35 ~ 1.45
캡션 line-height: 1.25 ~ 1.35
리스트 제목 line-height: 1.25 ~ 1.35
카드 설명 line-height: 1.35
```

### 텍스트 크기

```text
페이지 제목: 18px ~ 20px / 600
섹션 제목: 15px ~ 16px / 600
리스트 제목: 14px ~ 15px / 600
본문: 13px ~ 14px / 400
보조 정보: 12px ~ 13px / 400
배지: 11px ~ 12px / 500
버튼: 13px ~ 14px / 600
```

---

## 4.4 컴팩트 배지

상태값과 부가정보는 가능한 한 배지로 한 줄에 정리한다.

### 예시

```text
[성공] 통자요 · 고객 상담 통화 · 10분 전
[처리중] 회의록 · 주간회의 · 방금 전
[실패] 업무위키 · 설정 오류 · 1시간 전
```

### 배지 규칙

```text
높이: 20px ~ 22px
좌우 padding: 6px ~ 8px
폰트 크기: 11px ~ 12px
라운드: 999px 또는 4px
텍스트는 1~4자 중심으로 짧게 유지
```

### 상태 배지

```text
성공: 초록 계열
처리중: 파랑 또는 회색 계열
실패: 빨강 계열
처리제외: 노랑 또는 회색 계열
설정오류: 빨강/주황 계열
```

색상만으로 상태를 전달하지 않는다. 반드시 텍스트를 함께 표시한다.

---

## 4.5 한 줄 정보 정렬

모바일 리스트는 가능한 한 다음 구조를 사용한다.

```text
[상태배지] 자동화명 · 제목 · 시간
```

또는

```text
제목
[상태] 자동화명 · 실행자 · 시간
```

긴 제목은 한 줄 말줄임 처리한다.

```css
white-space: nowrap;
overflow: hidden;
text-overflow: ellipsis;
```

상세 내용은 상세 화면에서 확인하게 한다.  
목록에서 모든 내용을 다 보여주려고 하지 않는다.

---

# 5. 컴포넌트 규칙

## 5.1 카드

```text
background: #ffffff 또는 #f8f9fa
border: 1px solid #e5e7eb
border-radius: 8px
padding: 모바일 10~12px / PC 16~20px
shadow: 기본 사용하지 않음
```

카드는 정보 묶음 용도이다.  
장식용 큰 카드 사용을 피한다.

---

## 5.2 버튼

### Primary Button

```text
background: #111111
color: #ffffff
height: 모바일 38px ~ 40px
border-radius: 6px
font-size: 13px ~ 14px
font-weight: 600
```

### Secondary Button

```text
background: #ffffff
border: 1px solid #e5e7eb
color: #111111
height: 모바일 36px ~ 40px
border-radius: 6px
```

버튼은 크고 시원하게 쓰되, 모바일에서 불필요하게 세로 공간을 많이 차지하지 않도록 한다.

---

## 5.3 입력 필드

```text
height: 38px ~ 40px
border-radius: 6px
padding: 8px 10px
font-size: 14px
border: 1px solid #e5e7eb
```

textarea는 기본 높이를 88px 이하로 시작한다.  
필요 시 사용자가 확장할 수 있게 한다.

---

## 5.4 리스트 아이템

실행 결과, 자동화 목록, 승인 요청 목록은 리스트형 UI를 기본으로 한다.

```text
padding: 8px 10px
border-bottom: 1px solid #f3f4f6
min-height: 44px
```

모바일에서 리스트 아이템 하나가 80px 이상을 기본 차지하지 않게 한다.

---

## 5.5 하단 바텀 메뉴

사용자용 화면의 핵심 내비게이션이다.

```text
높이: 56px ~ 60px
항목: 홈 / 실행 / 결과 / 내정보
아이콘 + 라벨 사용 가능
배경: #ffffff
상단 border: 1px solid #e5e7eb
```

PC 사용자용에서도 하단 바텀 메뉴를 유지한다.

---

## 5.6 데이터 분석활용 패널

PC 사용자용에만 기본 노출한다.  
모바일에서는 홈 또는 별도 화면으로 접근한다.

```text
width: 220px ~ 260px
background: #ffffff 또는 #f8f9fa
border-right: 1px solid #e5e7eb
padding: 12px
```

### 메뉴

```text
데이터 분석활용
- 통자요
- 회의록
- 아이디어 캐치
- 업무위키
```

이 패널은 실행 메뉴가 아니다.  
클릭 시 본문은 검색, 분석, 히스토리 등 데이터 활용 화면으로 바뀐다.

---

# 6. 색상 규칙

기본 색상은 단순하게 유지한다.

```text
Primary: #111111
Primary Active: #242424
Canvas: #ffffff
Surface Soft: #f8f9fa
Surface Card: #f5f5f5
Hairline: #e5e7eb
Hairline Soft: #f3f4f6
Ink: #111111
Body: #374151
Muted: #6b7280
Success: #10b981
Warning: #f59e0b
Error: #ef4444
Info: #3b82f6
```

Primary CTA는 검정 계열을 사용한다.  
강한 포인트 컬러를 남발하지 않는다.

---

# 7. 타이포그래피

기본 폰트는 Inter 계열 또는 시스템 산세리프를 사용한다.  
국문 가독성을 위해 Pretendard 또는 system-ui 사용을 허용한다.

```css
font-family: Pretendard, Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

Cal Sans 계열의 큰 디스플레이 타이포는 마케팅 랜딩 페이지에서는 사용할 수 있으나, 엔팔라이언트 앱 내부 화면에서는 우선하지 않는다.  
앱 내부는 정보 밀도와 가독성이 더 중요하다.

---

# 8. 반응형 기준

```text
Mobile: 0px ~ 767px
Tablet: 768px ~ 1023px
PC: 1024px 이상
Wide: 1440px 이상
```

## 모바일

```text
단일 컬럼
바텀 메뉴
카드/리스트 조밀 배치
데이터 분석활용 패널 숨김
```

## PC 사용자용

```text
상단바
좌측 데이터 분석활용 패널
본문
하단 바텀 메뉴
```

## 관리자/운영자 PC

```text
좌측 사이드바
상단 관리 바
본문 테이블/폼 중심
바텀 메뉴 사용하지 않음
```

---

# 9. 금지 사항

```text
사용자용 PC 화면을 관리자 콘솔처럼 만들지 않는다.
사용자용 PC에서 하단 바텀 메뉴를 제거하지 않는다.
좌측 데이터 분석활용 패널을 자동화 실행 메뉴로 만들지 않는다.
모바일 카드 라운드를 과하게 둥글게 만들지 않는다.
모바일 여백을 마케팅 페이지처럼 넓게 잡지 않는다.
모바일 리스트를 지나치게 큰 카드로 만들지 않는다.
상태를 색상만으로 표시하지 않는다.
Webhook, API Key, 내부 토큰을 UI에 직접 노출하지 않는다.
관리자용/운영자용/사용자용 레이아웃을 억지로 하나로 합치지 않는다.
```

---

# 10. 구현 지시 요약

개발 시 다음 원칙을 우선한다.

```text
1. 사용자용 모바일은 compact density를 적용한다.
2. 모바일 좌우 여백은 12px 수준을 기본으로 한다.
3. 모바일 카드 radius는 8px 이하를 기본으로 한다.
4. 모바일 카드 padding은 10~12px 수준을 기본으로 한다.
5. 실행 결과 목록은 상태 배지 + 제목 + 시간 정보를 한 줄에 최대한 정렬한다.
6. PC 사용자용은 모바일 구조를 확장한다.
7. PC 사용자용 좌측은 [데이터 분석활용] 패널이다.
8. 자동화 실행은 하단 메뉴의 [실행]에서 처리한다.
9. 관리자용과 운영자용은 별도 콘솔 레이아웃으로 분리한다.
10. 기능 구현 중 디자인 판단이 필요한 경우 이 문서를 우선 기준으로 삼는다.
```
