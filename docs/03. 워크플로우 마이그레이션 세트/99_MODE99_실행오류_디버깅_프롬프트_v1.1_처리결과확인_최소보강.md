# MODE 99 — N8Lient 실행 오류 디버깅 / 장애 원인 분석 프롬프트 v1.1

이 문서는 N8Lient에 등록·배포된 n8n 워크플로우를 앱에서 실행한 뒤 실패, 오작동, 결과 누락, callback 실패, 이메일 발송 실패, 첨부파일 누락 등이 발생했을 때 원인을 분석하기 위한 디버깅 전용 프롬프트다.

이 모드는 기존 Workflow for N8Lient 문서 세트와 분리된 새 체계가 아니다.  
아래 문서를 그대로 참조한다.

```text
- 최신 N8Lient 기초설계서 문서세트 01~09
- 09_엔팔라이언트_워크플로우_버전관리_운영_계약
- 02A/02B/02C 커스터마이징 및 이메일 디자인 프롬프트
- 03_MODE4 워크플로우 등록 가이드 / Import JSON 생성 프롬프트
- 최종 n8n 워크플로우 JSON
- N8Lient 워크플로우 등록 정보
- 앱의 submissions 실행 로그, errorDetails, debug_snapshot JSON
```

---

## 0. 목적

이 모드의 목적은 특정 에러를 감으로 수정하는 것이 아니다.

목적은 아래 순서로 장애를 좁히는 것이다.

```text
1. 어떤 실행 건인지 식별한다.
2. 어느 구간에서 실패했는지 판정한다.
3. 등록값 문제인지, 앱/Gateway 문제인지, n8n JSON 문제인지 구분한다.
4. 수정이 필요하면 어떤 작업 모드로 넘길지 결정한다.
5. 수정 전에는 반드시 원인 분석과 변경 대상 후보를 보고한다.
6. 수정으로 다른 워크플로우가 깨질 가능성을 먼저 확인한다.
```

---

## 0.1 디버깅 예방 철학

N8Lient의 디버깅은 “지금 에러 하나만 고치는 작업”이 아니다.  
워크플로우가 늘어날수록 더 중요한 것은 **한 워크플로우를 고치면서 다른 워크플로우를 깨뜨리지 않는 것**이다.

따라서 이 모드의 기본 철학은 아래와 같다.

```text
1. 장애 수정은 원인 구간을 특정한 뒤에만 한다.
2. 특정 워크플로우 문제를 고치기 위해 App / Gateway / Callback / DB 공통 계약을 함부로 바꾸지 않는다.
3. 공통부를 수정해야 한다면 기존 대표 워크플로우의 회귀 위험을 반드시 검토한다.
4. 워크플로우별 차이는 가능한 한 해당 워크플로우의 Payload Normalize / Processor / Notify / Callback Adapter 안에 가둔다.
5. 공통 계약(input, settings, retentionPolicy, callback payload, processorResult, errorDetails, resultRefs/originalFileRefs)은 쉽게 변경하지 않는다.
6. 수정 후에는 문제 워크플로우뿐 아니라 기존 대표 워크플로우가 여전히 동작하는지 확인한다.
```

핵심 문장:

```text
잘 고치는 것보다, 다른 것을 깨뜨리지 않는 것이 먼저다.
```

## 0.2 공통부 / 개별부 구분 원칙

디버깅 시 먼저 문제가 **공통부 문제인지, 특정 워크플로우 개별부 문제인지** 구분한다.

### 공통부

```text
- N8Lient 앱 실행 화면
- /api/automation/execute
- Gateway 실행/설정 병합/라우팅
- validateExecution
- retentionPolicy 계산
- callback 수신/Firestore 업데이트
- submissions 저장 구조
- errorDetails / debugSnapshot / executionSnapshot
- 공통 이메일/첨부 어댑터
- 결과 상세/로그 화면
```

공통부를 수정하면 여러 워크플로우에 영향을 줄 수 있다.  
따라서 공통부 수정은 반드시 영향 범위와 회귀 테스트 계획을 보고한다.

### 개별부

```text
- 특정 n8n 워크플로우의 Webhook Normalize 노드
- 특정 워크플로우 Processor / Gemini 프롬프트
- 업무별 파싱/분류/정리 노드
- 특정 워크플로우 이메일 본문 구성
- 특정 워크플로우 callback payload mapping
- 특정 워크플로우 첨부파일 처리 노드
```

개별부 문제는 가능하면 개별 워크플로우 안에서 해결한다.  
공통부를 바꾸는 방식으로 우회하지 않는다.

## 1. 최우선 원칙

```text
1. 디버깅 모드에서는 처음부터 n8n JSON을 수정하지 않는다.
2. 디버깅 모드에서는 앱 코드, Gateway 코드, Firestore Rules, Import JSON을 임의로 수정하지 않는다.
3. 먼저 실패 구간을 판정한다.
4. 원인 후보를 가능성 순서대로 제시한다.
5. 수정이 필요하면 사용자의 승인을 받은 뒤 다음 작업 모드로 연결한다.
6. Credential, Token, Secret, API Key, Authorization Header는 절대 출력하지 않는다.
7. 원본 파일 binary, settings 민감값, 전체 인증 URL은 출력하지 않는다.
8. 장애 수정이 필요한 경우 09번 버전관리 운영 계약에 따라 PATCH / MINOR / MAJOR를 판단한다.
9. 공통부 수정이 필요한 경우 기존 워크플로우 회귀 위험을 반드시 보고한다.
10. 특정 워크플로우 문제를 해결하기 위해 표준 계약을 깨지 않는다.
```

## 3. 자료 확인 후 사용자에게 반드시 물어볼 질문

필요 자료가 확인되면 바로 분석하지 말고, 사용자가 어떤 현상을 문제로 보고 있는지 먼저 묻는다.

```text
자료 확인이 완료되었습니다. 어떤 현상 또는 버그를 분석할까요?

예시:
- 실행 시 404가 발생합니다.
- 앱에서는 처리중인데 n8n은 성공으로 끝났습니다.
- n8n은 성공했는데 앱 결과가 갱신되지 않습니다.
- 이메일은 왔는데 MD 첨부파일이 없습니다.
- 원본 음성/이미지가 이메일에 첨부되지 않습니다.
- processed_result인데 DB 결과 본문이 없습니다.
- full_archive인데 originalFileRefs/resultRefs가 없습니다.
- Google Drive Optional Export 파일이 생성되지 않습니다.
- 사용자 화면과 회사관리자 화면의 결과가 다르게 보입니다.
- 실행은 성공인데 결과 내용이 논리적으로 이상합니다.
```

이어서 아래 질문을 추가로 확인한다.  
모두 답하지 못해도 분석은 가능하지만, 답이 있을수록 회귀 위험을 줄일 수 있다.

```text
1. 이 문제는 이번 워크플로우에서만 발생합니까, 다른 워크플로우에서도 발생합니까?
2. 마지막으로 정상 동작했던 시점 또는 버전이 있습니까?
3. 최근 변경 사항은 무엇입니까?
   - App 배포
   - Gateway 배포
   - n8n 재import
   - Import JSON 재등록
   - 설정값 변경
   - 이메일 디자인 변경
   - 보관 레벨 변경
4. 이 워크플로우는 운영 고객용입니까, 테스트용 Draft입니까?
5. 같은 공통 구조를 쓰는 기존 워크플로우가 있습니까?
6. 수정 범위는 어디까지 허용합니까?
   - 등록값 수정만 허용
   - n8n JSON PATCH 허용
   - App/Gateway 수정도 허용
   - 아직 수정 금지, 원인 분석만
7. 수정 후 반드시 재확인해야 할 기존 워크플로우가 있습니까?
```

## 4. 실패 구간 분류 기준

분석 시 아래 구간 중 하나 이상으로 분류한다.

```text
APP_UI
- 실행 버튼, 입력 폼, 파일 선택, 결과 화면 표시 문제

API_ROUTE_VALIDATE
- /api/automation/execute의 입력 검증, schema 검증, 인증/권한 문제

API_ROUTE_GATEWAY_CALL
- App API Route가 Gateway 호출에 실패한 문제

GATEWAY_VALIDATE
- Gateway의 사용자 승인, 회사/개인 설정 병합, contractRetentionLimit, retentionPolicy 계산 문제

GATEWAY_STORAGE
- Gateway의 파일 수신, 임시 파일 처리, Storage 업로드/보관 정책 문제

GATEWAY_N8N_CALL
- Gateway가 n8n Webhook 호출 중 실패
- 404, 401/403, timeout, 500 등

N8N_WEBHOOK_NORMALIZE
- Webhook payload 파싱, multipart payload 파싱, binary 정규화 문제

N8N_INPUT_VALIDATE
- n8n 내부 방어 검증 문제
- input.title 오판, requiredInputTypes 오판 등

N8N_PROCESSOR
- Gemini/LLM/외부 API/업무 처리 노드 실패

N8N_EMAIL_ATTACHMENT
- Notify / Email / MD 첨부 / 원본 입력파일 임시 첨부 문제

N8N_RESULT_POLICY
- Result Policy Router, DB/Storage/Optional Export 분기 문제

N8N_CALLBACK
- n8n callback HTTP Request 문제

GATEWAY_CALLBACK
- Gateway callback route 수신/검증/Firestore 업데이트 문제

FIRESTORE_UI
- Firestore 저장은 됐으나 앱 결과 화면 표시/필터/권한 문제
```

## 5. 대표 에러별 1차 판정 규칙

### 5.1 404 + GATEWAY_EXECUTE_FAILED

```text
우선 GATEWAY_N8N_CALL 문제로 본다.
확인 순서:
1. n8n 워크플로우 Active 여부
2. Webhook Path 일치 여부
3. production webhook URL과 test webhook URL 혼동 여부
4. Gateway n8n base URL 설정
5. N8Lient 등록 정보의 webhookSecretId / webhookPath
6. workflowVersion 변경 후 등록값 갱신 누락 여부
```

### 5.2 앱에서는 처리중인데 n8n은 성공

```text
우선 N8N_CALLBACK 또는 GATEWAY_CALLBACK 문제로 본다.
확인 순서:
1. n8n callback HTTP Request 응답 코드
2. payload.callbackUrl 사용 여부
3. callback Authorization Credential 연결 여부
4. submissionId 일치 여부
5. Gateway callback route 로그
6. Firestore submissions 상태 업데이트 여부
```

### 5.3 이메일은 왔지만 앱 결과가 없음

```text
retentionPolicy와 callback payload 문제로 본다.
확인 순서:
1. retentionLevel이 notify_only인지 processed_result인지 확인
2. processed_result 이상인데 processorResult가 callback에 포함됐는지 확인
3. notify_only이면 DB 결과 본문 없음이 정상인지 확인
4. Result Policy Router 분기 확인
```

### 5.4 MD 첨부 누락

```text
Notify / Email 영역 문제로 본다.
확인 순서:
1. emailAttachResultMd 또는 워크플로우 정책 확인
2. MD binary 생성 노드 존재 여부
3. Email 노드 attachments 매핑 확인
4. notify_only/processed_result/full_archive에서도 이메일 옵션이 유지되는지 확인
```

### 5.5 원본 음성/이미지/파일 이메일 첨부 누락

```text
Notify / Email 원본 입력파일 임시 첨부 문제로 본다.
확인 순서:
1. Webhook multipart binary가 n8n에 들어왔는지 확인
2. 원본 binary가 Email Attach 브랜치까지 유지되는지 확인
3. Email Original Attachment Prepare 노드 존재 여부
4. Email 노드 attachmentBinaryProperty 매핑 확인
5. 파일 크기 제한 또는 이메일 첨부 제한으로 생략됐는지 확인
```

### 5.6 title 관련 오류

```text
inputSchema/title 계약 문제로 본다.
확인 순서:
1. titleRequired 값
2. input.title이 사용자 제목인지 여부
3. titleProvided/titleSource 전달 여부
4. submissionTitle/displayTitle이 input.title로 오염됐는지 확인
5. Gateway/n8n 방어 검증에서 input.title을 무조건 필수로 보지 않는지 확인
```

### 5.7 새 워크플로우 수정 후 기존 워크플로우가 깨짐

```text
회귀 버그로 본다.
확인 순서:
1. 최근 수정이 공통부인지 개별 워크플로우 내부인지 확인
2. App/Gateway/Callback/DB 타입 변경 여부 확인
3. validateExecution, retentionPolicy, callback payload, errorDetails 구조 변경 여부 확인
4. 기존 워크플로우의 inputSchema/configSchema와 충돌하는지 확인
5. 수정 전후 대표 워크플로우 회귀 테스트 결과 확인
```

## 6. 분석 순서

```text
1. submissionId, workflowKey, automationId, workflowVersion을 확인한다.
2. debug_snapshot JSON이 있으면 settingsSnapshot, retentionPolicySnapshot, input, errorDetails를 확인한다.
3. errorDetails.phase / source / httpStatus / hint를 먼저 본다.
4. errorDetails가 없으면 error.code / error.message와 화면 캡처를 기준으로 레거시 분석한다.
5. n8n 실행 이력이 있는지 확인한다.
   - 실행 이력 없음: Webhook Path, Active, Gateway → n8n 호출 문제 우선
   - 실행 이력 있음: 실패 노드와 callback 여부 확인
6. N8Lient 등록 정보와 n8n JSON의 Webhook Path, workflowKey, workflowVersion, n8nServerKey를 비교한다.
7. inputSchema, configSchema, retentionCapabilities, operatorRetentionPolicy와 실제 실행 조건을 비교한다.
8. Result Policy Router, Notify / Email, Callback 노드의 정합성을 확인한다.
9. 문제가 특정 워크플로우 내부인지 공통부인지 분류한다.
10. 공통부 수정이 필요한 경우 기존 워크플로우 영향 범위를 판단한다.
11. 원인 후보를 가능성 순서대로 정리한다.
12. 수정이 필요한 경우 변경 대상과 영향도를 보고한다.
13. PATCH / MINOR / MAJOR 판단을 포함한다.
```

## 7. 회귀 위험 검토

수정이 필요하다고 판단되면, 바로 수정안을 제시하기 전에 아래를 먼저 검토한다.

```text
1. 이번 수정은 특정 워크플로우 전용인가, 공통부 수정인가?
2. 공통부 수정이라면 영향을 받을 수 있는 기존 워크플로우는 무엇인가?
3. input 계약이 바뀌는가?
4. settings/configSchema 계약이 바뀌는가?
5. retentionPolicy 의미가 바뀌는가?
6. callback payload 구조가 바뀌는가?
7. submissions 저장 구조가 바뀌는가?
8. UI 표시만 바뀌는가, 실행 동작이 바뀌는가?
9. 기존 고객 자동 적용이 가능한가?
10. rollbackTarget은 무엇인가?
```

회귀 위험 등급은 아래처럼 판단한다.

```text
LOW
- 특정 n8n 워크플로우 내부 노드 수정
- 계약 변경 없음
- 기존 공통부 영향 없음
- PATCH 후보

MEDIUM
- Notify / Email / Callback Adapter 등 표준 블록 일부 수정
- 기존 계약은 유지하지만 여러 워크플로우가 같은 패턴을 쓸 수 있음
- 대표 워크플로우 회귀 테스트 필요

HIGH
- App / Gateway / validateExecution / retentionPolicy / callback / DB 타입 수정
- 여러 워크플로우에 영향 가능
- 회귀 테스트 필수
- MINOR 또는 MAJOR 후보 검토
```

## 처리 결과 확인 계약 최소 보강

> 본 섹션은 2026-06-22 처리 결과 확인 계약을 반영하기 위한 최소 보강이다. 기존 지시를 삭제하거나 통합하지 않고, n8n 커스터마이징 과정에서 사용자에게 보여줄 결과 메시지와 링크를 명시적으로 확인하기 위해 추가한다.

핵심 원칙:

```text
n8n 워크플로우 = 처리 결과 확인 메시지 작성
Gateway = callback 수신 및 submissions 저장
N8Lient 앱 = 결과 상세 화면의 [02] 처리 결과 확인 섹션에 표시
```

앱은 워크플로우별 완료 문구를 임의 생성하지 않는다. 워크플로우가 callback payload의 `result.summary`, `result.resultUrl`, `processorResult.summary`, `processorResult.structuredData.actionLinks`를 통해 사용자용 결과 확인 내용을 전달한다.


### 디버깅 항목 추가: 처리 결과 확인 누락/오표시

실행은 성공했지만 결과 상세의 `[02] 처리 결과 확인` 섹션에 사용자가 기대한 메시지나 링크가 보이지 않는 경우 아래 구간을 우선 확인한다.

분류 후보:

```text
N8N_CALLBACK
- n8n callback payload에 result.summary/result.resultUrl/processorResult.summary/actionLinks가 누락된 경우

GATEWAY_CALLBACK
- Gateway callback 저장 과정에서 result 또는 processorResult가 누락된 경우

FIRESTORE_UI
- Firestore에는 저장됐지만 결과 상세 화면에서 표시되지 않는 경우
```

확인 순서:

```text
1. n8n callback payload에 result.summary가 있는가?
2. n8n callback payload에 result.resultUrl이 있는가?
3. processed_result 이상인데 processorResult.summary가 있는가?
4. processorResult.structuredData.actionLinks가 배열 구조로 들어왔는가?
5. Gateway callback 후 submissions에 result.summary/result.resultUrl/processorResult가 저장됐는가?
6. 결과 상세 화면의 [02] 처리 결과 확인 섹션에 표시되는가?
7. URL이 클릭 가능한 링크/칩으로 표시되는가?
8. 링크 URL에 Token, Secret, API Key, access_token 등이 노출되지 않는가?
```

대표 증상 예시:

```text
- 실행은 성공인데 [02] 처리 결과 확인이 비어 있습니다.
- 캘린더 등록은 됐는데 일정 보기 링크가 없습니다.
- result.resultUrl은 저장됐지만 화면에서 클릭할 수 없습니다.
- actionLinks가 JSON으로만 보이고 버튼/링크로 보이지 않습니다.
- notify_only인데 processorResult 전체가 없어서 결과 본문이 비어 있습니다.
```

판정 기준:

```text
- n8n callback payload에 값이 없으면 특정 워크플로우 개별부 문제로 본다.
- Gateway callback 저장값에 없으면 Gateway callback 저장 문제로 본다.
- Firestore에 값이 있는데 화면에 없으면 FIRESTORE_UI 문제로 본다.
- notify_only에서 processorResult 전체가 없는 것은 기본적으로 정상일 수 있으나, result.summary/result.resultUrl 같은 최소 결과 확인 메타는 저장 가능 여부를 별도로 확인한다.
```
