# [Google Drive Optional Export 고도화 개정 메모]

이 문서는 2026-06-13 기준으로 확정된 Google Drive Optional Export 패키지 정책을 추가 반영한 보존형 개정본이다.

기존 원문은 삭제하지 않는다.  
다만 Google Drive Export가 사용되는 경우에는 아래 정책을 우선 해석 기준으로 삼는다.

## 핵심 원칙

```text
엔팔라이언트 기본 저장소 = Firestore + Firebase Storage
Google Drive = 기본 저장소가 아니라 Optional Export
Optional Export = 사용자가 선택한 외부 복사/내보내기 패키지
```

Google Drive Optional Export는 결과/보관 레벨을 대체하지 않는다.  
`retentionPolicy.level`은 여전히 Gateway가 결정하고, n8n은 Gateway가 내려준 정책을 따른다.

## Google Drive Optional Export 폴더 분리

Google Drive Optional Export를 사용하는 워크플로우는 MD 결과 파일과 첨부파일 저장 위치를 분리한다.

```text
MD 파일 보관 폴더명
MD 파일 보관 폴더 ID
첨부파일 보관 폴더명
첨부파일 보관 폴더 ID
```

폴더명은 사용자가 이해하기 위한 표시값이다.  
실제 업로드 위치는 Google Drive Folder ID를 기준으로 한다.

## 표준 설정 키

```text
optionalExportProvider
googleDriveMdFolderName
googleDriveMdFolderId
googleDriveAttachmentFolderName
googleDriveAttachmentFolderId
```

조건부 필수 규칙은 다음과 같다.

```text
optionalExportProvider = none
→ Google Drive 관련 필드는 없어도 된다.

optionalExportProvider = google_drive
→ MD 폴더명, MD 폴더 ID, 첨부파일 폴더명, 첨부파일 폴더 ID를 모두 요구한다.
```

## 표준 동작

```text
1. n8n은 Gateway가 병합해 전달한 payload.settings를 사용한다.
2. MD 파일은 googleDriveMdFolderId에 업로드한다.
3. 원본 음성, 이미지, 첨부파일은 googleDriveAttachmentFolderId에 업로드한다.
4. MD 본문에는 업로드된 첨부파일의 Google Drive 링크를 포함한다.
5. resultRefs에는 MD export 결과와 첨부 export 결과를 구분해 기록한다.
```

권장 resultRefs 타입:

```text
optional_export_md
optional_export_attachment
```

## Level 2 / Level 3와의 관계

```text
processed_result + google_drive
→ 엔팔라이언트 DB에는 processorResult 저장
→ Firebase Storage 원본 보관은 기본적으로 없음
→ Google Drive Optional Export가 켜져 있으면 원본 첨부파일을 Google Drive 첨부 폴더에 외부 복사 가능

full_archive + google_drive
→ 엔팔라이언트 DB에는 processorResult 저장
→ Firebase Storage에는 원본 보관
→ Google Drive에도 MD 파일과 첨부파일을 외부 복사 가능
```

따라서 Google Drive Optional Export는 엔팔라이언트 기본 보관 정책과 별개의 외부 복사 동작이다.  
UI에서는 사용자가 혼동하지 않도록 아래 의미를 명확히 표시한다.

```text
Google Drive 내보내기를 사용하면 MD 결과 파일과 원본 첨부파일이 지정한 Google Drive 폴더에 복사됩니다.
```

## 보안 및 Credential 원칙

```text
Google Drive 폴더 ID는 settings로 받는다.
Google 계정, OAuth Token, Refresh Token, Credential ID는 settings로 받지 않는다.
Google Drive 접근 권한은 n8n Credential이 담당한다.
지정한 폴더 ID는 해당 n8n Google Drive Credential 계정이 접근 가능한 폴더여야 한다.
```

# [지식 DB화/검색 최소 계약 개정 메모]

이 문서는 2026-06-12 기준으로 확정된 엔팔라이언트 결과 DB화 및 지식 검색 최소 구조를 추가 반영한 보존형 개정본이다.

기존 원문은 삭제하지 않는다.  
다만 Level 2 `processed_result` 이상에서 DB에 저장되는 결과는 아래 최소 공통 구조를 우선 해석 기준으로 삼는다.

## 핵심 원칙

```text
Level 1 notify_only = 이메일/알림 중심. 자유 포맷 허용.
Level 2 processed_result 이상 = 최소 공통 DB 포맷 적용.
Level 3 full_archive = 최소 공통 DB 포맷 + 원본/결과 파일 참조 보관.
```

통합 검색, 벡터라이징, 자동 분류, 백링크, 지식 그래프는 워크플로우 실행 단계에서 수행하지 않는다.  
이런 후가공은 나중에 `knowledgeSearchIndex`, `embeddingIndex` 또는 별도 인덱싱/마이그레이션 워크플로우에서 처리한다.

## 최소 공통 결과 구조

Level 2 이상에서 `processorResult`는 가능한 한 아래 공통 필드를 맞춘다.

```json
{
  "title": "결과 제목",
  "summary": "짧은 요약",
  "content": "본문 텍스트",
  "mdContent": "마크다운 본문",
  "hashtags": ["태그1", "태그2"],
  "attachments": [],
  "structuredData": {},
  "warnings": []
}
```

## 검색 1차 대상

초기 검색은 다음 필드를 우선 대상으로 한다.

```text
title
summary
content
mdContent
hashtags
authorName
workflowName
companyName
createdAt
```

검색 결과 카드에는 다음 라벨을 표시할 수 있어야 한다.

```text
제목
요약
워크플로우명
작성자
회사명
생성일
첨부파일 여부
보관 레벨
```

## 원본성 결과와 후가공 결과 분리

워크플로우는 가능한 한 원본성 결과를 저장한다.

```text
좋은 예:
- 발화/회의/통화/문서 내용을 정리한 title, content, mdContent
- 사용자가 나중에 검색할 수 있는 최소 hashtags
- 첨부파일 참조

피해야 할 예:
- 자동 백링크 생성
- 지식 그래프 생성
- 벡터 임베딩 생성
- 과도한 자동 분류
- 후가공 규칙에 종속된 태그 대량 생성
```

## 저장 계층

```text
submissions = 원본형 실행 결과와 최소 공통 processorResult 저장
knowledgeSearchIndex = 향후 통합 검색/자연어 검색용 후가공 인덱스
embeddingIndex = 필요 시 Level 2 이상 결과를 대상으로 생성하는 벡터 인덱스
```

Obsidian과 Google Drive Export는 기본 지식관리 저장소가 아니다.  
엔팔라이언트의 기준 저장소는 Firestore + Firebase Storage다.  
Obsidian, Google Drive, Markdown Export는 Optional Export로만 본다.

# [보존형 개정 메모]

이 문서는 기존 원문을 삭제하거나 요약하지 않고 보존한 상태에서, 2026-06-12 기준으로 확정된 결과/보관 정책 계층 구조를 우선 해석 기준으로 추가한 보존형 개정본이다.

기존 문서의 철학, 서비스 책임 경계, 4단계 자동화 구조는 유지한다.  
다만 결과/보관 정책 관련 용어와 우선순위는 아래 최신 정책을 우선한다.

## 최신 결과/보관 정책 계층

```text
워크플로우 마스터 = 기술적 최대치
회사별 계약/매핑 = 계약 한도
회사관리자 설정 = 회사 권장값
개인사용자 설정 = 개인 선택
Gateway = 최종 계산 및 방어
n8n = Gateway가 내려준 최종 retentionPolicy에 따른 분기 처리
```

## 강제 한도

```text
workflowTemplates.retentionCapabilities
→ clientContracts.contractRetentionLimit
→ clientAutomations.contractRetentionLimit
```

워크플로우가 기술적으로 지원하지 않는 단계, 또는 회사가 계약하지 않은 단계는 선택할 수 없다.

## 실행 선택 우선순위

```text
userAutomationSettings.userRetentionPreference
→ clientAutomations.companyRetentionPolicy.recommendedLevel
→ workflowTemplates.retentionCapabilities.defaultLevel
→ 시스템 기본값 full_archive
```

회사관리자의 설정은 강제값이 아니라 회사 권장값이다.  
개인사용자가 별도 설정을 하면 개인 설정이 회사 권장값보다 우선한다.  
단, 개인 설정은 워크플로우 기술 한도와 회사별 계약 한도를 초과할 수 없다.

## 핵심 필드

```text
workflowTemplates.retentionCapabilities
clientContracts.contractRetentionLimit
clientAutomations.contractRetentionLimit
clientAutomations.companyRetentionPolicy
userAutomationSettings.userRetentionPreference
submissions.retentionPolicySnapshot
submissions.retentionPolicySnapshot.resolvedFrom
```

## 최신 계산 원칙

```text
selectableLevels =
workflowTemplates.retentionCapabilities.supportedLevels
∩ clientAutomations.contractRetentionLimit.allowedLevels
```

회사 권장값은 선택 가능 범위를 제한하지 않는다.  
Gateway는 잘못된 요청이 들어와도 최종 방어선으로 한도 내 보정 또는 오류 처리를 수행한다.

## 1. 문서 목적

이 문서는 엔팔라이언트 솔루션 헌법에서 정의한 원칙을 실제 시스템 구조로 해석한 아키텍처 기준 문서다.

헌법이 엔팔라이언트의 철학과 책임 경계를 선언한다면, 본 문서는 그 원칙이 실제 서비스, 서버, 데이터 흐름, 저장소, 워크플로우 구조 안에서 어떻게 배치되는지를 정의한다.

본 문서는 세부 구현 문서가 아니다.  
Firestore 컬렉션별 상세 필드, Gateway API의 요청/응답 JSON, Firebase Security Rules, n8n 노드 코드, UI 컴포넌트 상세 설계는 하위 문서에서 다룬다.

## 2. 엔팔라이언트 솔루션 전체 구조

엔팔라이언트 솔루션은 n8n 자동화를 고객사와 개인 사용자가 사용할 수 있도록 하는 자동화 운영 플랫폼이다.

전체 구조는 다음 계층으로 나뉜다.

```text
사용자 / 회사관리자 / 오퍼레이터
        ↓
n8lient-app
        ↓
n8lient-gateway
        ↓
n8lient-db / n8lient-storage
        ↓
n8lient-workflows
        ↓
외부 서비스
```

엔팔라이언트의 기본 호출 원칙은 다음과 같다.

```text
브라우저는 n8n을 직접 호출하지 않는다.
브라우저는 n8lient-gateway만 호출한다.
n8lient-gateway가 n8n을 서버 간 호출한다.
n8n은 처리 결과를 n8lient-gateway callback으로 반환한다.
```

## 3. 표준 서비스 구성

### 3.1 n8lient-app

`n8lient-app`은 엔팔라이언트 애플리케이션이다.

담당 영역은 다음과 같다.

```text
Google 로그인
회사 가입 요청
오퍼레이터 콘솔
회사관리자 콘솔
개인사용자 실행 화면
워크플로우 목록 조회
회사 공용 설정 화면
개인 설정 화면
실행 요청 화면
실행 결과 목록
실행 결과 상세 화면
권한 기반 다운로드 화면
```

`n8lient-app`은 비밀값을 보유하지 않는다.

### 3.2 n8lient-gateway

`n8lient-gateway`는 엔팔라이언트의 서버 통제 지점이다.

담당 영역은 다음과 같다.

```text
Firebase ID Token 검증
사용자 승인 상태 확인
clientId 검증
role 검증
워크플로우 사용권 확인
회사 공용 설정 조회
개인 사용자 설정 조회
설정 병합
retentionPolicy 결정
canonicalPayload 생성
파일 수신
보관 정책에 따른 원본 파일 저장 또는 임시 처리
Firestore submissions 생성
n8n Webhook 서버 간 호출
callback 수신
processorResult 저장
다운로드 권한 검증
외부 export 요청 중계
```

Gateway는 엔팔라이언트의 정책이 실제로 실행되는 지점이다.

따라서 다음 책임을 가진다.

```text
누가 실행할 수 있는가
어떤 워크플로우를 실행할 수 있는가
어떤 설정값이 적용되는가
어떤 보관 레벨이 적용되는가
원본 파일을 영구 보관할 것인가
결과를 누구에게 보여줄 수 있는가
```

### 3.3 n8lient-db

`n8lient-db`는 Firestore 기반 데이터 계층이다.

담당 영역은 다음과 같다.

```text
사용자 정보
고객사 정보
회사코드
가입 요청
워크플로우 마스터
회사별 워크플로우 사용권
회사 공용 설정
개인 사용자 설정
보관 정책 설정
실행 이력
결과 메타데이터
오류 상태
운영 로그 일부
```

Firestore는 실행 상태와 결과 추적의 기준 저장소다.

Firestore에 저장하면 안 되는 것은 다음과 같다.

```text
파일 원본
base64 파일 본문
binary
Google Access Token
Google Refresh Token
n8n Credential ID
API Key
Secret
```

### 3.4 n8lient-storage

`n8lient-storage`는 Firebase Storage 기반 파일 보관 계층이다.

담당 영역은 다음과 같다.

```text
원본 음성 파일
원본 이미지
PDF, DOCX, MD 등 첨부 원본
생성된 결과 파일
다운로드 대상 파일
export 전 임시 파일
```

Firebase Storage는 엔팔라이언트의 기본 파일 보관 계층이다.  
단, 실제 원본 영구 보관 여부는 `retentionPolicy.level`에 따른다.

### 3.5 n8lient-workflows

`n8lient-workflows`는 n8n 기반 업무 프로세서 계층이다.

담당 영역은 다음과 같다.

```text
표준 payload 수신
processorInput 생성 또는 수신
음성 인식
문서 분석
LLM 호출
요약
변환
구조화
processorResult 생성
retentionPolicy에 따른 결과 처리 분기
success / failed / config_error callback 전송
```

n8n이 담당하지 않는 영역은 다음과 같다.

```text
사용자 인증
회사 권한 판단
회사/개인 설정 병합
Firestore 직접 수정
최종 보관 권한 판단
사용자별 다운로드 권한 판단
Credential을 사용자별로 동적 전환하는 구조
```

n8n은 가능한 한 다음 구조를 따른다.

```text
processorInput → 업무 처리 → processorResult
```

## 4. 결과/보관 레벨 아키텍처

엔팔라이언트는 자동화별로 결과/보관 수준을 선택할 수 있다.

표준 레벨은 다음과 같다.

```text
notify_only        = 알림/로그형
processed_result   = 가공지식 저장형
full_archive       = 원본 포함 지식보관형
```

### 4.1 Level 1. notify_only

목적은 가장 빠르고 가볍게 자동화를 도입하는 것이다.

아키텍처 흐름은 다음과 같다.

```text
n8lient-app / 외부 트리거
→ n8lient-gateway
→ 최소 submission 생성
→ n8lient-workflows 처리
→ 이메일 또는 로그 중심 결과 전달
→ Gateway callback으로 최소 결과 상태 갱신
```

저장 원칙은 다음과 같다.

```text
Firestore = 최소 submission 로그, status, summary, error
Firebase Storage = 원본 영구 보관 없음
Email = 결과 보관함처럼 사용 가능
```

### 4.2 Level 2. processed_result

목적은 가공된 업무 지식을 엔팔라이언트 DB에 축적하는 것이다.

아키텍처 흐름은 다음과 같다.

```text
n8lient-gateway
→ submission 생성
→ n8lient-workflows 처리
→ processorResult callback
→ Firestore에 processorResult 저장
→ 결과 화면에서 조회
```

저장 원칙은 다음과 같다.

```text
Firestore = processorResult, structuredData, keywords, warnings
Firebase Storage = 원본 영구 보관 없음
Email = 선택 알림
```

### 4.3 Level 3. full_archive

목적은 가공 지식과 원본 파일을 함께 보관하는 것이다.

아키텍처 흐름은 다음과 같다.

```text
n8lient-gateway
→ 원본 파일 Firebase Storage 저장
→ originalFileRefs 생성
→ submission 생성
→ n8lient-workflows 처리
→ processorResult callback
→ 결과 파일이 있으면 resultRefs 저장
→ 결과 상세 화면과 권한 기반 다운로드 제공
```

저장 원칙은 다음과 같다.

```text
Firestore = processorResult, originalFileRefs, resultRefs, status, metadata
Firebase Storage = 원본 파일, 결과 파일
Google Drive = Optional Export 가능
```

## 5. 사용자 역할 기반 구조

엔팔라이언트는 세 가지 역할을 기준으로 작동한다.

```text
오퍼레이터 → 회사관리자 → 개인사용자
```

오퍼레이터는 워크플로우 기본 보관 정책을 정의한다.  
회사관리자는 허용된 범위 안에서 회사 공용 보관 정책을 설정한다.  
개인사용자는 허용된 범위 안에서 개인 실행 설정을 적용할 수 있다.

정책 우선순위는 다음과 같다.

```text
오퍼레이터 정책 > 회사관리자 정책 > 개인사용자 설정
```

실행 설정값의 적용 우선순위는 다음과 같다.

```text
개인사용자 설정 > 회사 공용 설정 > 워크플로우 기본값
```

단, 보관 레벨은 오퍼레이터와 회사관리자가 허용한 범위 안에서만 변경할 수 있다.

## 6. 회원가입 및 인증 구조

엔팔라이언트의 기본 회원가입과 로그인은 Google 인증을 표준으로 한다.

```text
Google Login
→ Firebase Authentication
→ Firebase Auth uid 발급
→ n8lient-app에서 ID Token 획득
→ n8lient-gateway에서 ID Token 검증
→ Firestore users 문서 기준으로 권한 판단
```

Google 인증은 신원 확인 수단이다.  
실제 엔팔라이언트 사용 권한은 Firestore의 users 문서에 있는 role, clientId, approvalStatus로 결정한다.

## 7. 4단계 자동화 아키텍처

### 7.1 입력/트리거 영역

입력/트리거 영역은 다양한 입력을 canonicalPayload로 정규화한다.

표준 트리거 유형은 다음과 같다.

```text
manual
scheduled
webhook
watch
retry
system
```

표준 입력 유형은 다음과 같다.

```text
text
file
audio
image
form
url
```

### 7.2 프로세서 영역

프로세서 영역은 실제 업무 결과를 만드는 영역이다.

n8n 워크플로우는 이 영역에 집중해야 한다.

### 7.3 결과/보관 영역

결과/보관 영역은 `retentionPolicy`에 따라 다음을 결정한다.

```text
processorResult 저장 여부
원본 파일 영구 보관 여부
결과 파일 저장 여부
이메일 알림 여부
Optional Export 여부
```

### 7.4 운영/예외/추적 영역

운영/예외/추적 영역은 실행 상태, 실패, 재시도, 예외, 감사 로그를 관리한다.

표준 상태는 다음과 같다.

```text
queued
processing
success
failed
skipped
config_error
```

n8n은 Firestore를 직접 수정하지 않고, callback으로 결과를 반환한다.

## 8. 표준 실행 흐름

### 8.1 manual 실행 흐름

```text
개인사용자
→ n8lient-app 실행 화면
→ n8lient-gateway
→ 권한/설정/보관 정책 확인
→ retentionPolicy에 따라 파일 저장 또는 임시 처리
→ n8lient-workflows 호출
→ processorResult callback
→ n8lient-db 결과 갱신
→ 결과 화면 표시
```

### 8.2 webhook 실행 흐름

```text
외부 시스템
→ n8lient-gateway
→ API 인증/권한 검증
→ canonicalPayload + retentionPolicy 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

### 8.3 scheduled 실행 흐름

```text
스케줄러
→ n8lient-gateway 또는 scheduled adapter
→ canonicalPayload + retentionPolicy 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

### 8.4 watch 실행 흐름

```text
외부 저장소 감지
→ watch adapter
→ 표준 input으로 정규화
→ canonicalPayload + retentionPolicy 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

Google Drive Watch는 watch adapter의 한 유형이다.

## 9. 표준 데이터 흐름

엔팔라이언트와 n8n 사이의 데이터 계약은 다음을 중심으로 한다.

```text
canonicalPayload
retentionPolicy
processorInput
processorResult
```

### 9.1 canonicalPayload

canonicalPayload는 Gateway가 만든 표준 실행 요청이다.

### 9.2 retentionPolicy

retentionPolicy는 해당 실행에서 결과와 원본을 어느 수준까지 보관할지 정의한다.

생성 주체는 Gateway다.  
소비 주체는 Gateway, n8lient-workflows, n8lient-db다.

### 9.3 processorInput

processorInput은 프로세서가 실제 업무 처리에 사용할 입력이다.

### 9.4 processorResult

processorResult는 프로세서가 만든 표준 결과다.

## 10. 저장소 아키텍처

### 10.1 Firestore

Firestore는 실행 상태와 결과 메타데이터의 기준 저장소다.

보관 레벨별 책임은 다음과 같다.

```text
notify_only = 최소 로그와 summary 중심
processed_result = processorResult 중심
full_archive = processorResult + originalFileRefs + resultRefs
```

### 10.2 Firebase Storage

Firebase Storage는 파일 보관 계층이다.

원본 영구 보관은 기본적으로 `full_archive`에서 수행한다.  
`notify_only`와 `processed_result`에서는 처리 중 임시 파일을 사용할 수 있으나, 원본을 영구 보관하지 않을 수 있다.

### 10.3 Google Drive

Google Drive는 기본 저장소가 아니다.

사용 가능 위치는 다음과 같다.

```text
Watch 입력 소스
Optional Export 저장소
고객 요청에 따른 선택 저장
PoC 단계의 임시 저장소
```

### 10.4 Email

Email은 알림 채널이며, `notify_only` 레벨에서는 사실상의 경량 보관함 역할을 할 수 있다.

단, 이메일에 Firebase Storage 공개 URL을 직접 삽입하지 않는다.

## 13. 권한 및 보안 아키텍처

엔팔라이언트는 인증과 권한을 분리한다.

인증은 Google / Firebase Authentication이 담당한다.  
권한은 Firestore의 users 문서와 role, clientId, approvalStatus를 기준으로 판단한다.

보안 원칙은 다음과 같다.

```text
브라우저는 n8lient-gateway만 호출한다.
브라우저는 n8n 토큰을 알 수 없다.
Gateway는 모든 실행 요청에서 Firebase ID Token을 검증한다.
Gateway는 clientId 기준으로 데이터 접근을 제한한다.
Firestore Rules는 클라이언트 직접 조회 범위를 제한한다.
Firebase Storage 파일은 권한 확인 없이 공개하지 않는다.
Credential, Token, Secret, API Key는 사용자 설정값으로 노출하지 않는다.
n8n Credential은 n8n 내부 고정 Credential로 관리한다.
```

## 14. 배포 및 환경 구성

### 14.1 n8lient-app

배포 위치는 Netlify 또는 웹 호스팅이다.

브라우저에 노출 가능한 공개 설정만 사용한다.

### 14.2 n8lient-gateway

배포 위치는 Google Cloud Run이다.

환경변수 성격은 다음과 같다.

```text
Firebase Admin SDK 설정
n8n 서버 주소
n8n 서버 간 호출 토큰
callback secret
upload limit
allowed origins
```

### 14.3 n8lient-db / n8lient-storage

구성 위치는 Firebase Firestore와 Firebase Storage다.

### 14.4 n8lient-workflows

구성 위치는 n8n self-hosted 또는 n8n Cloud다.

n8n 서버 `.env` 의존은 최소화한다.  
가능하면 Credential 기반 인증을 사용한다.

## 15. 운영/예외/추적 구조

운영 추적의 중심은 submission이다.

보관 레벨과 무관하게 모든 실행은 최소 submission을 남긴다.

대표 오류는 다음과 같다.

```text
REQUIRED_SETTING_MISSING
REQUIRED_INPUT_MISSING
REQUIRED_INPUT_FILE_MISSING
UNSUPPORTED_FILE_TYPE
MAX_UPLOAD_EXCEEDED
RESOURCE_PERMISSION_DENIED
STORAGE_UPLOAD_FAILED
RETENTION_POLICY_INVALID
PROCESSOR_FAILED
CALLBACK_FAILED
EXTERNAL_API_FAILED
```

n8n은 오류 발생 시 가능한 한 failed 또는 config_error callback을 반환한다.  
Gateway는 callback을 수신해 submissions 상태를 갱신한다.

## 16. 버전 및 변경 관리

엔팔라이언트는 워크플로우와 설정 구조의 변경을 추적해야 한다.

관리 대상은 다음과 같다.

```text
workflowTemplate version
configSchema version
inputSchema version
retentionPolicy version
processor version
manual version
company setting updatedAt
user setting updatedAt
```

설정 변경은 변경 이후의 새 실행부터 적용된다.  
완료된 submission의 실행 당시 설정과 결과는 변경하지 않는다.
