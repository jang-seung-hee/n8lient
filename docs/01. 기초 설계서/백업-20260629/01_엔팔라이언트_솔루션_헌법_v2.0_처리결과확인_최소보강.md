# 엔팔라이언트 솔루션 헌법 v2.0

- 문서명: 엔팔라이언트 솔루션 헌법
- 버전: v2.0 리뉴얼
- 작성일: 2026-06-16
- 문서 상태: 최상위 정책 문서
- 보강 메모: 2026-06-22 처리 결과 확인 메시지 계약 최소 보강
- 적용 범위: N8Lient 앱, Gateway, n8n 워크플로우, Firestore, Firebase Storage, Optional Export

## 1. 선언

엔팔라이언트(N8Lient)는 n8n 자동화를 고객사와 개인 사용자가 안전하고 쉽게 사용할 수 있도록 만드는 자동화 운영 플랫폼이다.

엔팔라이언트는 n8n을 대체하지 않는다. 엔팔라이언트는 n8n 워크플로우의 실행, 설정, 권한, 결과 보관, 추적, 고객사 배포를 담당하는 운영 계층이다.

엔팔라이언트의 목적은 다음과 같다.

```text
1. 잘 만든 n8n 워크플로우를 여러 고객사와 사용자에게 안정적으로 배포한다.
2. n8n의 업무 처리 로직은 가볍게 유지하고, 인증·권한·설정·보관·추적은 플랫폼이 담당한다.
3. 회사 공용 설정과 개인 사용자 설정을 함께 지원한다.
4. 자동화 결과를 보관 수준에 따라 이메일, DB, Storage로 분리 관리한다.
5. 자동화별 도입 난이도와 보안 요구에 따라 결과 보관 레벨을 선택할 수 있게 한다.
6. Gateway를 실행 정책의 최종 계산·방어 지점으로 둔다.
7. 워크플로우를 버전 단위로 관리하고, 고객사별 사용 버전 고정, 선택 업그레이드, 롤백이 가능한 운영 체계를 유지한다.
```

## 2. 솔루션 구성

엔팔라이언트 솔루션은 다음 구성 요소로 이루어진다.

```text
n8lient-app
n8lient-gateway
n8lient-db
n8lient-storage
n8lient-workflows
optional-export
```

### 2.1 n8lient-app

사용자, 회사관리자, 오퍼레이터가 사용하는 웹 애플리케이션이다.

담당 범위:

```text
Google 로그인
회사코드 기반 가입 요청
오퍼레이터 콘솔
회사관리자 콘솔
사용자 실행 화면
워크플로우 마스터 등록
계약/사용자/설정 관리
실행 결과 목록 및 검색
```

### 2.2 n8lient-gateway

브라우저와 n8n 사이의 서버 계층이다.

담당 범위:

```text
인증 검증
사용자/회사 권한 검증
워크플로우 사용 가능 여부 확인
회사 설정과 개인 설정 병합
retentionPolicy 최종 계산
submission 생성
파일 처리
n8n 서버 간 호출
callback 수신 및 결과 저장
```

Gateway는 실행 정책의 최종 방어선이다.

### 2.3 n8lient-db

Firestore 기반 데이터 계층이다.

담당 범위:

```text
사용자/회사/계약/설정/워크플로우 마스터
실행 로그
processorResult
retentionPolicySnapshot
검색 대상 메타데이터
```

### 2.4 n8lient-storage

Firebase Storage 기반 파일 저장 계층이다.

담당 범위:

```text
Level 3 원본 파일
Level 3 결과 파일
다운로드 대상 파일
```

### 2.5 n8lient-workflows

n8n 워크플로우 실행 계층이다.

담당 범위:

```text
Gateway payload 수신
Header Auth 검증
입력 정리
프로세서 실행
결과 생성
retentionPolicy에 따른 후처리
Gateway callback
```

n8n은 Firestore를 직접 수정하지 않는다. n8n은 회사/개인 설정 병합을 수행하지 않는다.

### 2.6 Optional Export

Google Drive, Obsidian, Markdown Export 같은 외부 활용 경로다.

Optional Export는 엔팔라이언트의 기본 저장소를 대체하지 않는다.

## 3. 사용자 역할

### 3.1 operator

플랫폼 운영자다.

권한:

```text
고객사 생성/관리
워크플로우 마스터 등록/수정/배포
계약 생성/관리
운영 현황 조회
초기 회사관리자 승인
워크플로우 버전관리/릴리즈/롤백 관리
보안 규칙과 운영 정책 관리
```

operator는 특정 고객사에 소속되지 않아도 된다.

### 3.2 company_admin

고객사 관리자다.

권한:

```text
소속 회사 사용자 승인/관리
회사 자동화 설정 관리
회사 정보 관리
회사 실행 결과 조회
회사 권장 보관 정책 설정
```

company_admin은 operator 권한을 부여할 수 없다. 회사관리자는 현재 정책상 회사당 1명을 기본으로 한다.

### 3.3 user

일반 사용자다.

권한:

```text
승인된 회사의 계약 자동화 사용
개인 설정 등록
자동화 실행
본인 실행 결과 조회
본인 보관 레벨 선택
```

## 4. 자동화 4단계 구조

모든 자동화는 다음 구조를 따른다.

```text
1. 입력/트리거
2. 프로세서
3. 결과/보관
4. 운영/예외/추적
```

### 4.1 입력/트리거

지원 입력:

```text
text
file
image
audio
manual
watch
schedule
webhook
```

입력 필수 여부는 워크플로우 마스터의 `inputSchema`가 정의한다.

### 4.2 프로세서

n8n 또는 외부 AI 모델이 실제 업무 처리를 수행한다.

원칙:

```text
프로세서는 업무 로직에 집중한다.
회사/개인 설정 병합은 Gateway가 한다.
권한 판단은 Gateway가 한다.
```

### 4.3 결과/보관

결과 보관은 3단계로 나뉜다.

```text
Level 1 notify_only
Level 2 processed_result
Level 3 full_archive
```

자세한 기준은 결과/보관 레벨 계약 문서에서 정의한다.

### 4.3.1 처리 결과 확인 메시지 원칙

N8Lient 앱의 결과 상세 화면에는 사용자가 워크플로우 실행 결과를 가장 먼저 확인하는 `[02] 처리 결과 확인` 섹션이 있다.

이 섹션에 표시되는 사용자용 완료 메시지, 요약, 대표 결과 URL, 후속 액션 링크는 앱이 임의로 생성하지 않는다.

```text
n8n 워크플로우 = 처리 결과 확인 메시지 작성
Gateway = callback 수신 및 submissions 저장
n8lient-app = 저장된 메시지와 링크를 [02] 처리 결과 확인에 표시
```

워크플로우별로 완료 후 사용자에게 반드시 전달해야 하는 내용이 다르므로, n8n 프로세서 전환 또는 커스터마이징 시 AI는 처리 결과 확인 메시지 초안을 작성하고, 사용자 승인 후 callback payload 생성 노드에 반영한다.

권장 callback 필드:

```text
result.summary
result.resultUrl
processorResult.summary
processorResult.structuredData.actionLinks
```

보안상 token, secret, credential, access key, authorization 값은 처리 결과 확인 메시지와 링크에 포함하지 않는다.


### 4.4 운영/예외/추적

모든 실행은 최소 submission 로그를 남긴다.

필수 추적 항목:

```text
submissionId
clientId
uid
workflowKey
automationId
status
createdAt
completedAt
errorCode
errorMessage
retentionPolicySnapshot
```

워크플로우 버전관리, 고객사별 사용 버전 고정, 릴리즈 상태, 업그레이드 및 롤백 기준은 「09_엔팔라이언트_워크플로우_버전관리_운영_계약」을 따른다.

## 5. 결과 보관 레벨 철학

### 5.1 Level 1 — notify_only

이메일 중심 경량 사용 단계다.

N8Lient에는 생성된 지식 본문이나 결과 파일을 보관하지 않는다. 단, 이메일 본문, MD 첨부파일, 입력 원본 파일의 이메일 임시 첨부 형태로 결과를 전달할 수 있다.

적합한 경우:

```text
빠른 시작
DB/Storage에 결과를 남기고 싶지 않은 경우
이메일로 결과를 받으면 충분한 경우
```

### 5.2 Level 2 — processed_result

생성 지식을 DB에 저장하고 앱에서 다시 볼 수 있게 하는 단계다.

제공 범위:

```text
Level 1 기능 전체
이메일 본문, MD 첨부파일, 입력 원본 파일 임시 첨부 기능 상속
processorResult 저장
결과 목록/상세 보기
검색
공유
MD 다운로드 동적 생성
```

### 5.3 Level 3 — full_archive

Level 2의 모든 기능에 원본 파일과 결과 파일 참조를 더한 단계다.

적합한 경우:

```text
원본 근거 확인 필요
첨부파일과 결과를 함께 관리해야 함
감사/검증/재확인이 필요
```

## 6. 보관 정책 계층

결과 보관 정책은 다음 계층으로 계산한다.

```text
워크플로우 마스터 = 기술적 최대치
회사별 계약/매핑 = 계약 한도
회사관리자 설정 = 회사 권장값
개인사용자 설정 = 개인 선택
Gateway = 최종 계산 및 방어
n8n = Gateway가 내려준 최종 retentionPolicy에 따라 분기
```

강제 한도:

```text
workflowTemplates.retentionCapabilities
→ clientContracts.contractRetentionLimit
→ clientAutomations.contractRetentionLimit
```

실행 선택 우선순위:

```text
userAutomationSettings.userRetentionPreference
→ clientAutomations.companyRetentionPolicy.recommendedLevel
→ workflowTemplates.retentionCapabilities.defaultLevel
→ 시스템 기본값
```

회사관리자 설정은 기본적으로 권장값이다. 개인사용자가 별도 설정하면 개인 설정이 우선한다. 단, 기술 한도와 계약 한도를 초과할 수 없다.

## 7. title 계약

자동화 실행 payload에서 제목 관련 필드는 다음과 같이 구분한다.

```text
input.title = 사용자가 직접 입력한 제목. 없으면 null.
titleProvided = 사용자가 제목을 직접 입력했는지 여부.
titleSource = 제목 출처.
submissionTitle = 시스템 임시 표시 제목.
displayTitle = UI 표시 제목.
processorResult.title = 프로세서가 생성한 최종 결과 제목.
```

원칙:

```text
input.title을 모든 워크플로우의 고정 필수값으로 보지 않는다.
titleRequired=true일 때만 input.title이 필수다.
titleRequired=false이면 input.title=null이 허용된다.
시스템 임시 제목을 input.title에 넣지 않는다.
최종 표시 제목은 processorResult.title을 우선한다.
```

## 8. 보안 원칙

금지 사항:

```text
브라우저에서 n8n 직접 호출
n8n에서 Firestore 직접 수정
Firestore에 파일 원본 저장
Firestore에 base64 저장
Firestore에 Credential/Token/Secret 저장
Google Drive를 기본 저장소로 전제
Storage 공개 URL 직접 노출
운영 데이터를 임의 삭제
```

원칙:

```text
Gateway가 최종 권한 검증을 수행한다.
파일 접근은 앱 또는 Gateway 다운로드 API를 통해 제어한다.
Optional Export는 기본 저장소가 아니라 외부 복사 기능이다.
```

## 9. 폐기된 정책

현행 설계에서 아래 정책은 폐기한다.

```text
raw n8n JSON을 앱이 직접 분석하는 방식
AI Assist로 앱 내부에서 n8n JSON을 자동 해석하는 방식
Google Drive를 기본 저장소로 보는 방식
모든 결과/원본 파일을 무조건 영구 보관하는 방식
input.title을 고정 필수값으로 보는 방식
n8n이 회사/개인 설정을 직접 조회하는 방식
n8n이 Firestore를 직접 수정하는 방식
```
