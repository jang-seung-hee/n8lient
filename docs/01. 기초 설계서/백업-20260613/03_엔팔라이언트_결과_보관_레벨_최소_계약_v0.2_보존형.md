# [보존형 개정 메모]

이 문서는 기존 원문을 삭제하거나 요약하지 않고 보존한 상태에서, 2026-06-12 기준으로 확정된 결과/보관 정책 계층 구조를 우선 해석 기준으로 추가한 보존형 개정본이다.

기존 문서의 철학, 서비스 책임 경계, 4단계 자동화 구조는 유지한다.  
다만 결과/보관 정책 관련 용어와 우선순위는 아래 최신 정책을 우선한다.

## 최신 결과/보관 정책 계층

```text
워크플로우 마스터 = 기술적 최대치
회사별 계약/매핑 = 계약 한도
회사관리자 설정 = 회사 권장값
개인사용자 설정 = 개인 선택
Gateway = 최종 계산 및 방어
n8n = Gateway가 내려준 최종 retentionPolicy에 따른 분기 처리
```

## 강제 한도

```text
workflowTemplates.retentionCapabilities
→ clientContracts.contractRetentionLimit
→ clientAutomations.contractRetentionLimit
```

워크플로우가 기술적으로 지원하지 않는 단계, 또는 회사가 계약하지 않은 단계는 선택할 수 없다.

## 실행 선택 우선순위

```text
userAutomationSettings.userRetentionPreference
→ clientAutomations.companyRetentionPolicy.recommendedLevel
→ workflowTemplates.retentionCapabilities.defaultLevel
→ 시스템 기본값 full_archive
```

회사관리자의 설정은 강제값이 아니라 회사 권장값이다.  
개인사용자가 별도 설정을 하면 개인 설정이 회사 권장값보다 우선한다.  
단, 개인 설정은 워크플로우 기술 한도와 회사별 계약 한도를 초과할 수 없다.

## 핵심 필드

```text
workflowTemplates.retentionCapabilities
clientContracts.contractRetentionLimit
clientAutomations.contractRetentionLimit
clientAutomations.companyRetentionPolicy
userAutomationSettings.userRetentionPreference
submissions.retentionPolicySnapshot
submissions.retentionPolicySnapshot.resolvedFrom
```

## 최신 계산 원칙

```text
selectableLevels =
workflowTemplates.retentionCapabilities.supportedLevels
∩ clientAutomations.contractRetentionLimit.allowedLevels
```

회사 권장값은 선택 가능 범위를 제한하지 않는다.  
Gateway는 잘못된 요청이 들어와도 최종 방어선으로 한도 내 보정 또는 오류 처리를 수행한다.

# 엔팔라이언트 결과/보관 레벨 최소 계약 v0.1

- 문서명: 엔팔라이언트 결과/보관 레벨 최소 계약
- 버전: v0.1
- 문서 상태: 최소 계약 초안
- 상위 문서: 엔팔라이언트 솔루션 헌법 1.1, 엔팔라이언트 아키텍처 명세서 1.1
- 목적: 자동화별 결과 전달, 가공 지식 저장, 원본 파일 보관 수준을 표준 레벨로 정의한다.

---

## 1. 기본 원칙

엔팔라이언트의 자동화 결과는 자동화별 보관 정책에 따라 저장 수준을 다르게 적용할 수 있다.

모든 실행은 운영 추적을 위해 최소한의 submission 로그를 남긴다.  
단, 자동화 결과 본문, 구조화 데이터, 원본 파일, 첨부파일의 보관 여부는 자동화별 `resultRetentionLevel` 또는 `retentionPolicy.level` 설정에 따른다.

결과/보관 레벨은 다음 3단계로 구분한다.

```text
Level 1. notify_only        = 알림/로그형
Level 2. processed_result   = 가공지식 저장형
Level 3. full_archive       = 원본 포함 지식보관형
```

---

## 2. 표준 enum

DB와 payload에서는 다음 enum을 사용한다.

```text
notify_only
processed_result
full_archive
```

UI에는 다음 한글명을 표시한다.

```text
알림/로그형
가공지식 저장형
원본 포함 지식보관형
```

---

## 3. Level 1. notify_only

### 3.1 정의

가장 가벼운 결과 처리 방식이다.

자동화 실행 결과를 화면 로그 또는 이메일로 전달하고, 엔팔라이언트에는 최소 실행 이력만 남긴다.

### 3.2 사용 목적

```text
빠른 자동화 도입
고객사 저장소 설정 부담 최소화
원본 보관이 필요 없는 단순 보고
이메일을 사실상의 결과 보관함으로 사용하는 경우
```

### 3.3 저장 대상

Firestore submissions에는 최소 정보만 저장한다.

```text
submissionId
clientId
uid
workflowKey
automationId
status
createdAt
completedAt
result.summary
error
```

processorResult 전체 본문은 저장하지 않거나, 짧은 요약만 저장한다.  
원본 음성, 이미지, 파일은 영구 보관하지 않는다.

### 3.4 결과 전달 방식

```text
화면 실행 로그
이메일 본문
이메일 첨부파일 선택 가능
```

### 3.5 적용 예시

```text
통화 요약 결과를 이메일로만 발송
회의록 초안을 이메일 첨부파일로 발송
단순 알림 자동화
```

---

## 4. Level 2. processed_result

### 4.1 정의

자동화 결과로 생성된 가공 지식은 엔팔라이언트 DB에 저장하되, 원본 파일은 보관하지 않는 방식이다.

### 4.2 사용 목적

```text
검색 가능한 자동화 결과 축적
사용자별 실행 이력 관리
결과 본문과 구조화 데이터 보관
원본 보관까지는 필요 없는 경우
```

### 4.3 저장 대상

Firestore submissions에 가공 결과를 저장한다.

```text
submissionId
status
input metadata
processorResult
result.summary
structuredData
keywords
warnings
completedAt
```

원본 음성, 이미지, 파일은 영구 보관하지 않는다.  
필요 시 처리 중 임시 파일은 사용할 수 있으나, 처리 완료 후 삭제한다.

### 4.4 결과 전달 방식

```text
엔팔라이언트 결과 화면
이메일 알림 선택 가능
검색/필터링 대상
```

### 4.5 적용 예시

```text
아이디어 정리 결과 저장
회의록 텍스트 저장
상담 요약 데이터 축적
업무 지식 카드 저장
```

---

## 5. Level 3. full_archive

### 5.1 정의

가공된 지식과 함께 원본 파일까지 보관하는 가장 높은 수준의 보관 방식이다.

### 5.2 사용 목적

```text
감사 추적
원본 재처리
증빙 보관
사용자가 원본과 결과를 함께 확인해야 하는 자동화
장기 지식 저장소 구축
```

### 5.3 저장 대상

Firestore에는 결과 메타데이터와 참조 정보를 저장한다.

```text
processorResult
originalFileRefs
resultRefs
storagePath
fileName
mimeType
sizeBytes
completedAt
```

Firebase Storage에는 원본 파일과 결과 파일을 저장한다.

```text
원본 음성
원본 이미지
원본 문서
생성된 markdown/pdf/docx 등 결과 파일
```

### 5.4 결과 전달 방식

```text
엔팔라이언트 결과 화면
원본 다운로드
결과 파일 다운로드
이메일 알림 선택 가능
Google Drive export 선택 가능
```

### 5.5 적용 예시

```text
통화 녹취 원본 + 요약 보관
회의 음성 원본 + 회의록 보관
이미지 원본 + 분석 결과 보관
법적/업무 증빙이 필요한 자동화
```

## 8. 설정 위치

보관 정책은 아래 위치에서 관리될 수 있다.

```text
workflowTemplates.defaultRetentionPolicy
clientAutomations.retentionPolicy
userAutomationSettings.retentionPolicyOverride
submissions.retentionPolicySnapshot
```

정책 우선순위는 다음과 같다.

```text
오퍼레이터 허용 범위 > 회사관리자 설정 > 개인사용자 override
```

실행 시 적용 우선순위는 다음과 같다.

```text
userAutomationSettings.retentionPolicyOverride
→ clientAutomations.retentionPolicy
→ workflowTemplates.defaultRetentionPolicy
→ 시스템 기본값
```

단, 개인사용자 override는 오퍼레이터와 회사관리자가 허용한 항목에 한해 적용된다.

## 9. Gateway 처리 기준

Gateway는 실행 요청 시 다음을 수행한다.

```text
1. workflowTemplate의 기본 보관 정책 조회
2. clientAutomation의 회사 보관 정책 조회
3. userAutomationSettings의 개인 override 조회
4. 허용 범위 검증
5. 최종 retentionPolicy 결정
6. submissions.retentionPolicySnapshot 저장
7. canonicalPayload.retentionPolicy 포함
8. level에 따라 파일 보관 여부 결정
```

파일이 있는 실행의 기준은 다음과 같다.

```text
notify_only:
- 원본 파일 영구 저장 생략 가능
- n8n 처리를 위해 file_0 전달 가능
- 처리 후 임시 파일 삭제

processed_result:
- 원본 파일 영구 저장 생략 가능
- processorResult는 저장
- n8n 처리를 위해 file_0 전달 가능

full_archive:
- 원본 파일 Firebase Storage 저장
- originalFileRefs 생성
- 원본 저장 실패 시 STORAGE_UPLOAD_FAILED 또는 RETENTION_STORAGE_FAILED 처리 가능
```

---

## 10. n8n 처리 기준

n8n은 canonicalPayload.retentionPolicy를 참고해 결과 처리 영역을 분기한다.

권장 구조는 다음과 같다.

```text
[20] Result Policy Router
[21] Notify / Email
[22] Store Processor Result
[23] Optional Export
[24] Callback
```

n8n은 다음 원칙을 따른다.

```text
processorResult는 먼저 만든다.
notify_only에서는 이메일/알림 중심으로 처리할 수 있다.
processed_result에서는 processorResult를 callback에 포함한다.
full_archive에서는 originalFileRefs와 resultRefs를 보존한다.
Optional Export 실패는 기본 처리 성공과 분리해 warning 처리할 수 있다.
```

n8n은 Firestore를 직접 수정하지 않는다.

## 12. Email 정책

Email은 알림 채널이다.

`notify_only`에서는 이메일이 사실상의 경량 보관함 역할을 할 수 있다.

이메일 첨부 정책은 다음 필드로 제어한다.

```text
emailEnabled
emailAttachResult
emailAttachOriginal
```

이메일에는 Firebase Storage 공개 URL을 직접 넣지 않는다.  
파일 접근이 필요한 경우 엔팔라이언트 권한 확인 링크를 넣는다.

---

## 13. Optional Export 정책

Optional Export는 기본 보관과 분리한다.

대상 예시는 다음과 같다.

```text
Google Drive
Google Docs
Google Sheets
Gmail
외부 Webhook
```

Optional Export 실패는 워크플로우 성격에 따라 warning으로 처리할 수 있다.

단, 고객이 해당 export 자체를 핵심 결과로 계약한 경우에는 failed 또는 config_error로 처리할 수 있다.

---

## 14. 금지 사항

아래 구조는 금지한다.

```text
retentionPolicy 없이 워크플로우가 임의로 보관 방식을 결정
보관 레벨과 무관하게 모든 원본을 무조건 영구 보관
Firestore에 파일 원본 저장
Firestore에 base64 저장
Firebase Storage 공개 URL을 이메일에 직접 삽입
Google Drive를 기본 원본 저장소로 전제
n8n이 Firestore를 직접 수정
Credential, Token, Secret, API Key를 회사/개인 설정에 노출
```
