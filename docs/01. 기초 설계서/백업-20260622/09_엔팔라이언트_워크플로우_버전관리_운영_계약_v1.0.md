# 엔팔라이언트 워크플로우 버전관리 / 운영 계약 v1.0

- 문서명: 엔팔라이언트 워크플로우 버전관리 / 운영 계약
- 버전: v1.0
- 작성일: 2026-06-17
- 문서 상태: 신규 운영 기준 문서
- 상위 문서: 엔팔라이언트 솔루션 헌법 v2.0, 아키텍처 명세서 v2.0, Gateway ↔ n8n 실행 계약 v2.0
- 적용 범위: N8Lient 앱, Gateway, n8n 워크플로우, 워크플로우 마스터, 고객사별 자동화 매핑, 릴리즈/롤백 운영

---

## 1. 목적

본 문서는 엔팔라이언트에서 운영하는 n8n 워크플로우의 버전 표기, 배포, 업그레이드, 롤백, 고객사별 버전 고정 기준을 정의한다.

엔팔라이언트는 하나의 n8n 워크플로우를 여러 고객사에 배포하는 운영 플랫폼이므로, 워크플로우 변경은 반드시 버전 단위로 추적되어야 한다.

핵심 목적은 다음과 같다.

```text
1. 운영자가 n8n 목록에서 워크플로우 버전을 즉시 구분할 수 있게 한다.
2. 고객사별로 사용 중인 워크플로우 버전을 DB에 고정한다.
3. 새 버전 배포가 기존 고객 실행에 자동 영향을 주지 않게 한다.
4. PATCH / MINOR / MAJOR 변경을 구분하여 배포 위험을 통제한다.
5. 문제 발생 시 이전 stable 버전으로 즉시 롤백할 수 있게 한다.
6. Gateway가 clientAutomation 기준으로 정확한 n8n 실행 대상을 라우팅할 수 있게 한다.
```

---

## 2. 기본 원칙

```text
1. 모든 N8Lient용 n8n 워크플로우는 버전관리한다.
2. 버전은 SemVer 방식으로 표기한다.
3. n8n 워크플로우명에는 반드시 버전을 포함한다.
4. workflowKey는 자동화 제품군을 식별한다.
5. workflowVersion은 실제 배포 버전을 식별한다.
6. 고객사별 사용 버전은 clientAutomation에 고정한다.
7. Gateway는 clientAutomation의 workflowKey + workflowVersion을 기준으로 실행 대상을 결정한다.
8. 새 버전이 등록되어도 기존 고객에게 자동 적용하지 않는다.
9. 최소 1개 이전 stable 버전은 롤백용으로 보존한다.
```

---

## 3. 버전 표기 규칙

버전은 SemVer 형식으로 관리한다.

```text
MAJOR.MINOR.PATCH
```

예:

```text
1.0.0
1.0.1
1.1.0
2.0.0
```

버전은 문자열로 저장한다.

```json
{
  "workflowKey": "idea-catcher",
  "workflowVersion": "1.0.1"
}
```

## 4. 버전 증가 기준

### 4.1 MAJOR

기존 고객 설정, 입력 구조, callback 구조, 보관 정책 구조와 호환되지 않는 변경이다.

예:

```text
inputSchema 구조 변경
callback payload 구조 변경
retentionPolicy 구조 변경
필수 설정값 추가
기존 configSchema와 동시 호환이 어려운 변경
기존 결과 저장 구조와 호환되지 않는 변경
기존 워크플로우와 동일 Gateway 라우팅으로 병행하기 어려운 변경
```

버전 예:

```text
1.1.3 → 2.0.0
```

운영 원칙:

```text
기존 고객 자동 적용 금지
신규 워크플로우 버전으로 등록
별도 마이그레이션 절차 필요
기존 버전 유지 가능해야 함
Gateway 라우팅 대상 별도 분리
```

---

### 4.2 MINOR

기존 구조와 호환되지만 기능이 추가되는 변경이다.

예:

```text
선택 기능 추가
설정 옵션 추가
Optional Export 기능 추가
이메일 첨부 옵션 추가
결과 표시 항목 추가
기본값으로 꺼져 있는 신규 기능 추가
```

버전 예:

```text
1.0.3 → 1.1.0
```

운영 원칙:

```text
신규 워크플로우 버전으로 등록
신규 고객 기본 배포 가능
기존 고객 자동 적용 금지
기존 고객은 선택 업그레이드
누락된 신규 설정값은 기본값으로 보정
```

판정 주의:

```text
기능 추가라도 기존 고객의 기본 동작이 바뀌지 않으면 MINOR로 본다.
기능 추가로 기존 고객의 결과 형식, 필수 입력, 필수 설정, callback 구조가 바뀌면 MAJOR 승격을 검토한다.
```

---

### 4.3 PATCH

기존 기능의 버그 수정, 안정화, 방어 로직 추가다.

예:

```text
null 방어
undefined 방어
빈 첨부파일 field 방어
오타 수정
노드 배치 정리
기존 로직 오류 수정
Credential 구조 유지 상태의 안전 패치
기존 결과 의미를 바꾸지 않는 이메일 HTML 수정
```

버전 예:

```text
1.0.0 → 1.0.1 → 1.0.2
```

운영 원칙:

```text
기존 고객에 적용 가능
단, 즉시 전체 자동 적용하지 않는다.
운영자 테스트 후 clientAutomation.workflowVersion을 새 PATCH로 전환한다.
변경 이력 필수 기록
문제가 생기면 이전 PATCH 또는 이전 stable로 롤백 가능해야 함
```

## 5. n8n 워크플로우 이름 규칙

n8n 워크플로우명에는 반드시 버전을 포함한다.

형식:

```text
n8lient_{워크플로우 표시명} {workflowVersion}
```

예:

```text
n8lient_JANG'S 아이디어 캐처 1.0.0
n8lient_JANG'S 아이디어 캐처 1.0.1
n8lient_JANG'S 아이디어 캐처 1.1.0
```

운영자가 n8n 목록에서 즉시 버전을 구분할 수 있어야 한다.

권장:

```text
워크플로우명 끝에 버전 표기
동일 workflowKey의 여러 버전은 이름상 구분 가능하게 유지
deprecated/retired 상태는 n8n 이름 또는 설명에 표시 가능
```

## 6. workflowKey / workflowVersion 관계

### 6.1 workflowKey

`workflowKey`는 자동화 제품군 또는 기능군을 식별하는 고정 키다.

예:

```text
idea-catcher
call-summary
meeting-minutes
expense-organizer
```

`workflowKey`는 같은 자동화 계열에서 유지한다.

### 6.2 workflowVersion

`workflowVersion`은 해당 workflowKey의 실제 배포 버전이다.

예:

```json
{
  "workflowKey": "idea-catcher",
  "workflowVersion": "1.1.0"
}
```

### 6.3 고객사별 버전 고정

고객사별 사용 버전은 `clientAutomation`에 고정한다.

예:

```json
{
  "clientId": "client_rentaltalk",
  "automationId": "auto_idea_catcher",
  "workflowKey": "idea-catcher",
  "workflowVersion": "1.0.1"
}
```

새 버전이 등록되어도 기존 `clientAutomation.workflowVersion`은 자동 변경하지 않는다.

---

## 7. 버전별 n8n 실행 라우팅

버전별로 실제 n8n 실행 대상이 명확해야 한다.

Gateway는 다음 기준으로 n8n 호출 대상을 결정한다.

```text
clientAutomation.workflowKey
+ clientAutomation.workflowVersion
→ workflowTemplateVersion 또는 routing table 조회
→ n8nServerKey / webhookPath / n8nWorkflowId 결정
→ n8n Webhook 호출
```

각 workflowVersion은 고유한 n8n 호출 대상을 가져야 한다.

권장 필드:

```json
{
  "workflowKey": "idea-catcher",
  "workflowVersion": "1.1.0",
  "n8nServerKey": "main",
  "webhookPath": "n8lient-idea-catcher-v1-1-0",
  "n8nWorkflowId": "abc123"
}
```

주의:

```text
동일 Webhook Path를 가진 서로 다른 버전의 n8n 워크플로우를 동시에 Active로 두지 않는다.
버전별 Webhook Path를 분리하거나 Gateway 내부 라우팅으로 실행 대상을 명확히 분리한다.
```

## 8. 워크플로우 마스터 저장 구조

워크플로우 마스터는 workflowKey 단일 값만으로 버전을 구분하지 않는다. 버전별 레코드를 저장할 수 있어야 한다.

권장 구조 A — 문서 ID에 버전 포함:

```text
workflowTemplates/{workflowKey}__{workflowVersion}
```

예:

```text
workflowTemplates/idea-catcher__1.0.0
workflowTemplates/idea-catcher__1.1.0
```

권장 구조 B — 패밀리/버전 분리:

```text
workflowFamilies/{workflowKey}
workflowFamilies/{workflowKey}/versions/{workflowVersion}
```

개념적으로는 B 구조가 가장 명확하다.

```text
workflowKey = 자동화 제품군
workflowVersion = 제품군의 실제 배포 버전
```

단, 현재 앱 구조가 `workflowTemplates/{workflowKey}` 단일 문서 구조라면, 버전관리 도입 시 별도 마이그레이션 계획을 세운다.

## 9. 릴리즈 상태값

각 버전은 다음 상태 중 하나를 가진다.

```text
draft       : 작성 중
testing     : 내부 테스트 중
stable      : 배포 가능
deprecated  : 사용 가능하지만 신규 배포 비권장
retired     : 사용 중단
```

상태 의미:

```text
draft
- 오퍼레이터가 작성 중인 버전
- 고객 배포 금지

testing
- 내부 테스트 또는 제한 테스트 중
- 신규 고객 기본 배포 금지

stable
- 신규 고객에게 배포 가능한 안정 버전
- 신규 고객 기본 버전 후보

deprecated
- 기존 고객 유지 가능
- 신규 고객 배포 비권장
- 선택 업그레이드 또는 마이그레이션 대상

retired
- 신규 실행 금지
- 해당 버전을 사용하는 clientAutomation이 0개일 때 적용하는 것을 원칙으로 함
```

신규 고객에게는 `stable` 버전만 배포한다.

## 10. 배포 운영 규칙

### 10.1 신규 고객

```text
현재 stable 버전을 기본 배포한다.
```

예:

```text
stable = 1.1.0
→ 신규 고객은 1.1.0 사용
```

### 10.2 기존 고객

```text
PATCH: 테스트 후 운영자 승인 전환 가능
MINOR: 자동 적용 금지, 선택 업그레이드
MAJOR: 자동 적용 금지, 별도 마이그레이션 필요
```

### 10.3 선택 업그레이드 절차

기존 고객을 새 버전으로 올릴 때는 다음 절차를 따른다.

```text
1. 현재 사용 버전 확인
2. 신규 버전과 설정 호환성 검사
3. 누락된 설정값 기본값 보정
4. 테스트 실행
5. 성공 시 clientAutomation.workflowVersion 전환
6. 실패 시 기존 버전 유지
7. 변경 이력 기록
```

## 11. 호환성 원칙

호환 로직을 모든 노드에 흩뿌리지 않는다.

호환 처리 위치:

```text
1. Gateway
2. Payload Normalize
3. Notify / Email Adapter
4. Callback Adapter
```

프로세서 영역에는 구버전 호환 분기를 넣지 않는다.

원칙:

```text
Gateway가 구형 설정을 신형 canonicalPayload로 보정한다.
n8n은 최소 방어만 한다.
입력/출력 영역은 표준 블록으로 유지한다.
워크플로우별 변화는 프로세서 영역에 집중한다.
프로세서 영역은 업무 로직을 담당하고, 버전 호환 처리를 담당하지 않는다.
```

## 13. 롤백 원칙

새 버전 배포 후 문제가 발생하면 고객사의 `clientAutomation.workflowVersion`을 이전 stable 버전으로 되돌린다.

예:

```text
1.0.1 문제 발생
→ clientAutomation.workflowVersion = 1.0.0
```

원칙:

```text
n8n 워크플로우 이전 버전은 즉시 삭제하지 않는다.
최소 1개 이전 stable 버전은 보존한다.
롤백 대상 버전의 webhookPath 또는 n8nWorkflowId가 유효해야 한다.
롤백 후 테스트 실행으로 정상 동작을 확인한다.
롤백 이력도 변경 이력에 기록한다.
```

## 14. MAJOR 마이그레이션 원칙

MAJOR 변경은 기존 고객에게 자동 적용하지 않는다.

MAJOR 업그레이드는 다음 절차를 따른다.

```text
1. 기존 버전 사용 고객 목록 확인
2. 설정 스키마 차이 분석
3. inputSchema/configSchema/callback/retentionPolicy 차이 보고
4. 마이그레이션 가능 여부 판단
5. 필요 시 설정값 변환 스크립트 또는 수동 매핑 작성
6. 테스트 고객 또는 내부 테스트 실행
7. 고객사별 선택 전환
8. 실패 시 기존 MAJOR 유지
```

MAJOR 버전은 신규 워크플로우처럼 취급한다.

---

## 15. PATCH 적용 원칙

PATCH는 호환 변경이지만 무조건 자동 반영하지 않는다.

절차:

```text
1. PATCH 버전 생성
2. 내부 테스트
3. 기존 설정으로 회귀 테스트
4. stable 표시
5. 대상 고객 clientAutomation.workflowVersion 전환
6. 문제 발생 시 rollbackTarget으로 복귀
```

PATCH라도 아래 경우에는 MINOR 또는 MAJOR 승격을 검토한다.

```text
신규 설정값이 필요함
이메일 결과 형식이 의미 있게 바뀜
callback payload 구조가 바뀜
processorResult 구조가 바뀜
기존 고객 실행 결과가 달라짐
```

---

## 16. MINOR 적용 원칙

MINOR는 기능 추가 버전이다.

기존 고객에게 자동 적용하지 않는다.

운영 원칙:

```text
신규 고객 기본 stable 후보로 사용 가능
기존 고객은 선택 업그레이드
신규 설정값은 기본값으로 보정 가능해야 함
기존 기능의 기본 동작은 유지해야 함
```

예:

```text
emailAttachOriginal 옵션 추가
Optional Export 선택 옵션 추가
결과 카드 표시 항목 추가
```

기본값이 기존 동작을 바꾸면 MAJOR 승격을 검토한다.

## 17. 금지 사항

```text
버전 없는 n8n 워크플로우 운영
n8n 워크플로우명에서 버전 누락
기존 고객의 workflowVersion 무단 자동 변경
동일 webhookPath에 여러 버전 혼재
MAJOR 변경을 PATCH로 처리
프로세서 영역에 구버전 호환 분기 난립
retired 버전을 사용하는 고객이 남아 있는데 n8n 워크플로우 삭제
이전 stable 버전 삭제 후 신규 버전만 남기는 운영
```

## 18. 최종 원칙

```text
1. 워크플로우는 무조건 버전관리한다.
2. n8n 워크플로우명에는 버전을 표시한다.
3. workflowKey는 자동화 제품군, workflowVersion은 실제 배포 버전이다.
4. 고객사별 사용 버전은 clientAutomation에 고정한다.
5. Gateway는 workflowKey + workflowVersion으로 n8n 실행 대상을 라우팅한다.
6. PATCH는 기존 고객 적용 가능하되 테스트 후 운영자 승인으로 전환한다.
7. MINOR는 신규 고객 기본 적용 가능, 기존 고객은 선택 업그레이드한다.
8. MAJOR는 별도 마이그레이션 없이 자동 적용하지 않는다.
9. 호환성 처리는 Gateway와 표준 입출력/어댑터 블록에 둔다.
10. 프로세서 영역은 워크플로우별 업무 로직만 담당한다.
11. 최소 1개 이전 stable 버전은 롤백용으로 보존한다.
```
