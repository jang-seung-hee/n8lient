# 엔팔라이언트 결과/보관 레벨 계약 v2.0

- 문서명: 엔팔라이언트 결과/보관 레벨 계약
- 버전: v2.0 리뉴얼
- 작성일: 2026-06-16
- 문서 상태: 현행 정책 기준 문서
- 상위 문서: 솔루션 헌법 v2.0, 아키텍처 명세서 v2.0

---

## 1. 목적

본 문서는 엔팔라이언트 자동화 실행 결과를 어떤 수준으로 전달, 저장, 검색, 보관할 것인지 정의한다.

결과 보관 레벨은 고객 도입 난이도, 보안 부담, 저장 비용, 지식 관리 수준을 조절하기 위한 핵심 정책이다.

---

## 2. 표준 레벨

DB와 payload에서는 다음 enum을 사용한다.

```text
notify_only
processed_result
full_archive
```

서비스 설명에서는 다음 명칭을 사용한다.

```text
1단계 알림·메일형
2단계 지식 DB 저장형
3단계 원본·첨부파일 연결형
```

---

## 3. 공통 원칙

모든 실행은 최소 submission 로그를 남긴다.

단, 생성된 지식 본문, 원본 파일, 결과 파일, 첨부파일 참조의 저장 여부는 보관 레벨에 따라 다르다.

```text
1단계 = 전송 중심. 이메일 본문, MD 첨부파일, 입력 원본 파일 임시 첨부 가능. N8Lient에 지식/파일 보관 안 함.
2단계 = 1단계 전송 기능 전체를 상속 + 생성 지식 DB 저장.
3단계 = 2단계 기능 전체를 상속 + 원본/파일 참조 보관.
```

---

## 4. Level 1 — notify_only

### 4.1 정의

가장 가볍게 시작할 수 있는 기본 단계다.

N8Lient에는 생성된 지식 본문이나 결과 파일을 DB/Storage에 보관하지 않는다. 회원가입, 실행 상태, 발송 여부, 오류 같은 최소 운영 정보만 관리한다.

결과물은 이메일 본문, MD 첨부파일, 입력 원본 파일의 이메일 임시 첨부 형태로 사용자에게 전달할 수 있다.

### 4.2 제공 범위

```text
회원가입/권한 관리
자동화 실행
최소 submission 로그
이메일 본문 전송
MD 첨부파일 전송 가능
입력 원본 파일 이메일 임시 첨부 가능
성공/실패 상태 추적
```

### 4.3 저장 대상

submissions에는 최소 로그만 저장한다.

```text
submissionId
clientId
uid
workflowKey
automationId
status
createdAt
completedAt
errorCode
errorMessage
emailSent
submissionTitle/displayTitle
retentionPolicySnapshot
```

저장하지 않는 것:

```text
processorResult.content
processorResult.mdContent
긴 결과 본문
원본 파일 참조
결과 파일 참조
Storage 파일 경로
검색용 본문 데이터
```

### 4.4 MD 첨부파일 정책

1단계에서도 이메일 전송 과정에서 MD 파일을 임시 생성하여 첨부할 수 있다.

다만 이 MD 파일은 다음에 해당하지 않는다.

```text
N8Lient DB 저장 대상 아님
Firebase Storage 보관 대상 아님
resultRefs 대상 아님
앱에서 재다운로드 대상 아님
검색 대상 아님
```

즉, 1단계의 MD 첨부는 “일회성 전송”이지 “N8Lient 보관”이 아니다.

입력 원본 파일(file/image/audio 등)도 이메일 전송용 임시 첨부로 전달할 수 있다. 이 역시 N8Lient DB 저장, Firebase Storage 보관, originalFileRefs/resultRefs, 앱 재다운로드, 검색 대상에 해당하지 않는다.

### 4.5 적합한 경우

```text
복잡한 지식 관리 없이 빠르게 자동화를 시작하고 싶은 경우
DB나 Storage에 결과 기록을 남기고 싶지 않은 경우
이메일 결과만으로 충분한 경우
보안 부담을 최소화해야 하는 경우
```

## 5. Level 2 — processed_result

### 5.1 정의

1단계의 이메일 본문, MD 첨부파일, 입력 원본 파일 임시 첨부 기능 전체에 더해, 생성된 지식을 N8Lient DB에 저장하는 단계다.

사용자는 결과 목록, 상세 보기, 공유, 검색 기능을 통해 생성 지식을 다시 조회할 수 있다.

### 5.2 제공 범위

```text
1단계 기능 전체
processorResult 저장
결과 목록
결과 상세 보기
검색 기능
공유 기능
MD 다운로드 동적 생성
```

### 5.3 저장 대상

submissions에 processorResult를 저장한다.

권장 구조:

```json
{
  "title": "결과 제목",
  "summary": "짧은 요약",
  "content": "본문 텍스트",
  "mdContent": "마크다운 본문",
  "hashtags": ["태그1", "태그2"],
  "attachments": [],
  "structuredData": {},
  "warnings": []
}
```

저장하지 않는 것:

```text
원본 파일 영구 보관
Firebase Storage 원본 경로
Storage 결과 파일 경로
originalFileRefs
resultRefs 기본 저장
```

### 5.4 MD 다운로드 정책

2단계에서 MD 다운로드는 가능하다.

원칙:

```text
DB에 저장된 processorResult.mdContent를 기반으로 요청 시점에 동적 생성한다.
MD 파일 자체를 Storage에 영구 저장하지 않는다.
동적 다운로드는 resultRefs로 보지 않는다.
```

MD 파일 자체를 Storage나 Drive에 저장하고 링크를 관리하면 3단계 또는 Optional Export 성격이 된다.

### 5.5 적합한 경우

```text
아이디어 정리
통화 요약
회의록
업무 지식화
문서 요약
나중에 검색하고 공유해야 하는 자동화
```

## 6. Level 3 — full_archive

### 6.1 정의

2단계의 모든 기능에 더해, 원본 파일과 결과 파일 참조까지 함께 보관하는 단계다.

생성된 지식의 근거가 되는 원본 음성, 이미지, 문서, 첨부파일을 함께 관리할 수 있다.

### 6.2 제공 범위

```text
2단계 기능 전체
originalFileRefs 저장
resultRefs 저장
Firebase Storage 원본 파일 저장
Firebase Storage 결과 파일 저장
원본 확인
재검토
첨부파일 기반 지식 관리
```

### 6.3 저장 대상

```text
processorResult
originalFileRefs[]
resultRefs[]
Storage original files
Storage result files
fileName
mimeType
size
provider
storagePath
```

### 6.4 적합한 경우

```text
원본 근거 확인이 중요한 업무
첨부파일과 결과물을 함께 관리해야 하는 업무
감사/검증/기록 보존이 필요한 업무
통화/회의 원본 재확인이 필요한 업무
```

### 6.5 주의사항

```text
스토리지 비용 증가
파일 접근 권한 관리 필요
원본 파일의 민감정보 위험
회사별 보관 기간과 삭제 정책 필요
```

## 7. Optional Export와의 관계

Optional Export는 보관 레벨이 아니라 부가 기능이다.

```text
Google Drive Export
Markdown Export
Obsidian Export
```

원칙:

```text
Optional Export는 retentionPolicy.level을 대체하지 않는다.
Optional Export는 N8Lient 기본 저장소가 아니다.
Optional Export는 사용자가 선택한 외부 복사/활용 기능이다.
```

예시:

```text
notify_only + email_md_attachment
= N8Lient에는 보관하지 않고 이메일로 MD 첨부 전송

processed_result + dynamic_md_download
= DB의 mdContent로 앱에서 MD 다운로드 동적 생성

processed_result + google_drive
= DB에는 processorResult 저장, Google Drive에는 외부 복사 가능

full_archive + google_drive
= N8Lient Storage에도 보관하고 Google Drive에도 외부 복사 가능
```

## 8. retentionCapabilities

워크플로우 마스터는 기술적으로 지원 가능한 최대 보관 범위를 정의한다.

필수 필드:

```text
maxLevel
defaultLevel
supportedLevels
supportsProcessorResult
supportsOriginalFileRefs
supportsResultRefs
supportsResultPolicyRouter
supportsEmailNotification
```

해석:

```text
supportsProcessorResult = Level 2 이상 가능 여부
supportsOriginalFileRefs = Level 3 원본 파일 참조 가능 여부
supportsResultRefs = 결과 파일/외부 export 참조 가능 여부
supportsResultPolicyRouter = n8n 내부 보관 정책 라우터 지원 여부
supportsEmailNotification = 이메일 보고 지원 여부
```

## 9. 계약 한도와 사용자 선택

선택 가능 레벨은 기술 한도와 계약 한도의 교집합이다.

```text
selectableLevels =
workflowTemplates.retentionCapabilities.supportedLevels
∩ clientAutomations.contractRetentionLimit.allowedLevels
```

회사 권장값은 선택 가능 범위를 제한하지 않는다.

실행 선택 우선순위:

```text
userAutomationSettings.userRetentionPreference
→ clientAutomations.companyRetentionPolicy.recommendedLevel
→ workflowTemplates.retentionCapabilities.defaultLevel
→ 시스템 기본값
```

Gateway는 최종 방어선으로 잘못된 요청을 한도 내 보정하거나 차단한다.

---

## 10. 금지 사항

```text
Level 1에서 processorResult 본문을 DB에 저장하는 것
Level 1에서 resultRefs를 저장하는 것
Level 1에서 앱 재다운로드용 파일을 보관하는 것
Level 2에서 원본 파일을 기본 영구 보관하는 것
Level 3 없이 모든 파일을 무조건 Storage에 저장하는 것
Optional Export를 기본 저장소로 설명하는 것
```
