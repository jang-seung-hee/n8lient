# 엔팔라이언트 결과데이터 권한관리 계약 v1.0

- 문서명: 엔팔라이언트 결과데이터 권한관리 계약
- 버전: v1.0
- 작성일: 2026-06-29
- 문서 상태: 신규 권한관리 기준 문서
- 상위 문서: 솔루션 헌법 v2.1, 아키텍처 명세서 v2.1, DB/Storage 계약 v2.1, 결과 DB화/지식검색 계약 v2.1
- 적용 범위: N8Lient App, Gateway callback 저장, Firestore submissions, 결과 목록/상세/검색 UI, Firebase Storage 접근 제어

---

## 1. 목적

본 문서는 N8Lient에 축적되는 자동화 실행 결과 데이터의 열람 권한을 정의한다.

N8Lient의 결과 데이터는 개인 생산성 자료이면서 동시에 회사 지식 자산이 될 수 있다. 따라서 다음 두 요구를 동시에 만족해야 한다.

```text
1. 개인이 생성한 민감한 결과는 기본적으로 보호한다.
2. 회사 차원에서 공유 가치가 있는 결과는 같은 회사 구성원이 활용할 수 있도록 공개한다.
```

이 문서는 결과를 저장할지 여부를 정하는 보관 정책 문서가 아니다. 보관 정책은 `retentionPolicy`가 담당하고, 열람 권한은 `accessMode`가 담당한다.

---

## 2. 핵심 원칙

```text
retentionPolicy = 결과를 저장할 것인가, 파일을 보관할 것인가를 결정한다.
accessMode = 저장된 결과 본문과 파일 참조를 누가 볼 수 있는지 결정한다.
```

권한 관리는 n8n 워크플로우가 아니라 N8Lient의 DB 저장/조회 계층에서 처리한다.

```text
n8n 역할
- 자동화 실행
- 결과 생성
- Gateway callback 전달

N8Lient 역할
- 결과 저장
- accessMode 결정
- 결과 조회 권한 판단
- 작성자 공개/취소 처리
- 관리자 공개 철회 처리
```

n8n은 Firestore를 직접 수정하지 않고, 사용자 권한을 판단하지 않는다.

---

## 3. 기본 권한 값

1차 구현에서는 결과 데이터의 열람 권한을 두 가지로만 구분한다.

```ts
accessMode: "private" | "company";
```

각 값의 의미는 다음과 같다.

```text
private
- 개인 보관
- 작성자 본인만 결과 본문을 열람할 수 있다.

company
- 회사 공개
- 같은 회사(clientId)에 속한 구성원이 결과 본문을 열람할 수 있다.
```

`public`이라는 명칭은 사용하지 않는다. 외부 공개로 오해될 수 있으므로 회사 내부 공개라는 의미를 명확히 하기 위해 `company`를 사용한다.

---

## 4. 워크플로우별 기본 권한 정책

각 워크플로우 또는 고객사별 자동화 매핑은 결과 생성 시 적용할 기본 권한 정책을 가진다.

```ts
resultAccessPolicy: {
  defaultAccessMode: "private" | "company";
  ownerCanChangeAccess: boolean;
  adminCanChangeAccess: boolean;
}
```

### 4.1 필드 의미

```text
defaultAccessMode
- 결과가 처음 생성될 때 적용할 기본 accessMode다.

ownerCanChangeAccess
- 작성자가 본인 결과의 공개 상태를 변경할 수 있는지 여부다.

adminCanChangeAccess
- 회사관리자가 공개된 결과를 철회할 수 있는지 여부다.
```

### 4.2 기본값 원칙

정책 값이 누락된 경우 안전한 기본값을 사용한다.

```ts
resultAccessPolicy: {
  defaultAccessMode: "private",
  ownerCanChangeAccess: false,
  adminCanChangeAccess: true
}
```

즉, 명시 정책이 없으면 결과는 개인 보관으로 저장하고, 사용자가 임의로 회사 공개하지 못하게 한다.

---

## 5. 개인 보관형 워크플로우 규칙

기본값이 `private`인 워크플로우는 다음 정책을 따른다.

```text
1. 결과 생성 시 기본 권한은 private이다.
2. 작성자 본인만 결과 본문을 열람할 수 있다.
3. 작성자가 필요하다고 판단하면 해당 결과를 company로 전환할 수 있다.
4. 작성자는 자신이 공개한 결과를 다시 private으로 되돌릴 수 있다.
5. 관리자는 공개된 결과가 부적절하다고 판단하면 private으로 공개 철회할 수 있다.
```

적용 예시:

```text
- 아이디어 캐쳐
- 개인 메모
- 개인 통화 기록
- 개인 일정 등록 결과
```

개인 보관형 워크플로우의 핵심은 다음과 같다.

```text
기본은 내 것.
공유 가치가 있는 결과만 작성자가 회사에 공개한다.
문제가 있는 공개 결과는 작성자 또는 관리자가 다시 비공개 처리한다.
```

---

## 6. 회사 공개형 워크플로우 규칙

기본값이 `company`인 워크플로우는 다음 정책을 따른다.

```text
1. 결과 생성 시 기본 권한은 company이다.
2. 같은 회사 구성원은 결과 본문을 열람할 수 있다.
3. 작성자는 임의로 private으로 변경할 수 없다.
4. 관리자만 예외적으로 특정 결과의 공개를 철회할 수 있다.
```

적용 예시:

```text
- 업무 위키
- 공지 요약
- 공유 회의록
- 회사 업무보고
- 고객 응대 기록
```

회사 공개형 워크플로우의 핵심은 다음과 같다.

```text
기본은 회사 공개.
작성자는 임의로 숨길 수 없다.
관리자는 불량 정보, 민감정보, 오분류 결과만 예외적으로 공개 철회한다.
```

---

## 7. 관리자 역할

관리자는 사전 승인자가 아니라 공개 데이터 관리자 역할을 가진다.

관리자가 할 수 있는 일:

```text
- 회사 공개 결과 목록 확인
- 부적절한 공개 결과의 공개 철회
- 공개 철회 사유 기록
```

관리자가 기본적으로 하지 않는 일:

```text
- 모든 개인 보관 결과 본문 열람
- 모든 결과의 사전 승인
```

관리자의 공개 철회는 데이터 삭제가 아니다. 원칙적으로 공개 상태만 `company`에서 `private`으로 되돌린다.

---

## 8. 결과 데이터 저장 필드

1차 구현 기준 결과 데이터에는 다음 필드를 저장한다.

```ts
submission: {
  clientId: string;
  uid: string;
  workflowKey: string;
  automationId: string;

  accessMode: "private" | "company";

  accessChangedBy?: string;
  accessChangedAt?: Timestamp;
  accessChangeReason?: string;

  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

`uid`는 실행 작성자를 의미한다. 코드 가독성을 위해 향후 `ownerUserId` alias를 둘 수 있으나, 1차 구현에서는 기존 submission 구조와의 호환을 우선한다.

---

## 9. 조회 권한 규칙

결과 본문 조회는 다음 기준으로 판단한다.

```text
accessMode = private
→ submission.uid가 현재 사용자와 같을 때만 열람 가능

accessMode = company
→ 현재 사용자의 clientId가 submission.clientId와 같으면 열람 가능
```

권한 판정은 가능한 한 하나의 함수 또는 서비스 계층으로 모은다.

```ts
function canReadSubmissionResult(user, submission) {
  if (submission.uid === user.uid) return true;

  if (submission.accessMode === "company") {
    return submission.clientId === user.clientId;
  }

  return false;
}
```

나중에 그룹/보안등급/특정 사용자 권한을 추가하더라도 이 판정 함수만 확장하는 구조를 유지한다.

---

## 10. 검색 권한 규칙

결과 목록과 검색은 accessMode를 반드시 반영한다.

```text
사용자 개인 결과 목록
- 본인 private 결과
- 같은 회사 company 결과

회사 지식 검색
- company 결과만 포함
- private 결과는 작성자 본인 검색 외에는 포함하지 않음
```

검색 인덱스 또는 후가공 인덱스에 private 결과가 포함되더라도, 조회 시점에는 반드시 accessMode와 clientId를 다시 확인한다.

---

## 11. 보관 레벨과의 관계

`accessMode`는 결과 보관 여부를 결정하지 않는다.

```text
notify_only
- processorResult 본문을 저장하지 않으므로 accessMode가 있어도 본문 열람 대상은 없다.
- 최소 로그 표시 정책에는 사용할 수 있다.

processed_result
- processorResult를 저장한다.
- 저장된 결과 본문 열람은 accessMode를 따른다.

full_archive
- processorResult, originalFileRefs, resultRefs, Storage 파일 참조를 저장한다.
- 본문과 파일 참조 열람은 accessMode를 따른다.
```

사용자 화면에서는 다음 표현을 사용한다.

```text
개인 보관
- 나만 볼 수 있습니다.

회사 공개
- 같은 회사 구성원이 볼 수 있습니다.

회사에 공개하기
- 이 결과를 회사 구성원이 볼 수 있도록 공개합니다.

공개 취소
- 회사 공개를 해제하고 나만 볼 수 있게 전환합니다.
```

## 13. 관리자 UI 문구

회사관리자 화면에서는 다음 표현을 사용한다.

```text
회사 공개 결과 관리
- 회사 구성원에게 공개된 결과를 관리합니다.

공개 철회
- 부적절하거나 민감한 공개 결과를 회사 공개 목록에서 제외합니다.
```

공개 철회 시 사유 입력은 선택으로 시작할 수 있으나, 향후 감사 로그를 위해 저장 구조는 남겨둔다.

## 14. 변경 이력 기준

결과 권한이 변경될 때는 다음 메타데이터를 남긴다.

```ts
accessChangedBy: string;
accessChangedAt: Timestamp;
accessChangeReason?: string;
```

권한 변경 유형 예시:

```text
owner_publish
- 작성자가 private 결과를 company로 공개

owner_unpublish
- 작성자가 본인 공개 결과를 private으로 되돌림

admin_withdraw
- 관리자가 company 결과를 private으로 공개 철회

system_default
- 결과 생성 시 워크플로우 기본 정책에 따라 accessMode 지정
```

초기 구현에서는 `accessChangeReason`을 선택값으로 둔다.

---

## 15. 향후 확장 방향

1차 구현에서는 `private`과 `company`만 사용한다.

향후 필요 시 다음과 같이 확장할 수 있다.

```ts
accessMode: "private" | "company" | "group" | "level" | "custom";
```

확장 예시:

```text
group
- 특정 부서 또는 팀만 열람 가능
- 예: 영업팀 공개

level
- 특정 보안등급 이상만 열람 가능
- 예: A등급 이상, S등급 전용

custom
- 특정 사용자만 지정 열람 가능
```

단, 1차 구현에서는 그룹/레벨/사용자 지정 공유는 구현하지 않는다. 현재 단계에서는 개인 보관과 회사 공개만 구현한다.

---

## 16. 최종 원칙

```text
결과 권한은 1차적으로 private / company 두 가지만 둔다.

private:
- 기본은 작성자만 열람
- 작성자가 회사 공개 가능
- 작성자가 공개 취소 가능
- 관리자가 공개 철회 가능

company:
- 기본은 회사 공개
- 작성자가 공개 취소 불가
- 관리자만 예외적으로 공개 철회 가능

권한 관리는 n8n이 아니라 N8Lient DB 저장/조회 계층에서 처리한다.
향후 group / level / custom 권한은 accessMode 확장으로 추가한다.
```
