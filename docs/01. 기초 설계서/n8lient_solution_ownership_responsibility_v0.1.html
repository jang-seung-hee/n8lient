N8Lient Solution Ownership & Responsibility Map

엔팔라이언트 솔루션
구성·소유권·역할 책임 구조

엔팔라이언트는 중앙 운영형 애플리케이션과 고객사 전용 프로세서·보관 계층을 결합한 자동화 운영 플랫폼이다. n8n은 고객사 전용 프로세서로 사용하고, 고객 데이터는 고객사 보관 계층에 저장하는 것을 기본 원칙으로 한다.

문서 버전 v0.1 작성일 2026-06-12 적용 범위: App · Gateway · Processor · Storage · Contract

1. 핵심 결론

엔팔라이언트는 n8n을 재판매하지 않는다. 고객사 전용 자동화 프로세서를 구축하고, 중앙 애플리케이션으로 운영·배포·설정·실행·조회 체계를 제공한다.

중앙 앱은 대표님이 운영하고,
프로세서와 보관 계층은 고객사 전용으로 둔다.

고객사는 자기 클라우드 또는 자기 계약 환경에 엔팔엔 프로세서와 보관 계층을 가진다. 엔팔라이언트 애플리케이션은 워크플로우 상품 등록, 회사 배정, 설정 관리, 실행 요청, 결과 조회의 운영 계층을 담당한다.

허용 방향

고객사 전용 n8n 프로세서 구축, 워크플로우 제작·운영 지원, 고객사 Storage/DB 보관, 중앙 운영 앱 제공

피해야 할 방향

n8n 접속권 판매, n8n 화이트라벨 재판매, 중앙 n8n 멀티테넌트 유료 접근권 제공

2. 솔루션 구성도

고객에게 보이는 것은 엔팔라이언트 앱이다. 실제 업무 처리는 고객사 전용 프로세서가 수행하고, 결과와 원본은 고객사 보관 계층에 남긴다.

3. 표준 실행 흐름

실행 요청은 중앙 앱에서 시작되지만, 실제 처리와 보관은 고객사 전용 계층으로 라우팅된다.

1

등록

오퍼레이터가 고객사 프로세서 endpoint, 보관 계층, 워크플로우 템플릿을 등록한다.

2

배포

고객사에 워크플로우 사용권을 배정하고, 회사관리자가 회사 공용 설정을 완료한다.

3

실행

개인사용자가 엔팔라이언트 앱에서 실행하면 Gateway가 권한과 설정을 검증한다.

4

처리

고객사 전용 n8n 프로세서가 표준 payload를 받아 업무 처리를 수행한다.

5

보관·조회

결과는 보관 레벨에 따라 고객사 DB/Storage에 남고, 앱은 권한 기반으로 보여준다.

4. 소유권·역할·책임 매트릭스

어떤 계층을 누가 소유하고, 누가 운영하며, 어떤 데이터를 가져야 하는지 명확히 구분한다.

구성 요소 소유권 운영 책임 보관 데이터 금지/주의
엔팔라이언트 애플리케이션중앙 운영 포털 서비스 제공자 워크플로우 상품 등록, 회사 배정, 사용자 화면, 운영 콘솔 계약 메타데이터, 템플릿, 라우팅 정보, 최소 상태 요약 고객 원본 파일 장기 보관 금지. n8n 접근권 판매 금지.
엔팔라이언트 Gateway중앙 또는 고객사별 선택 기본은 서비스 제공자, 고보안 고객은 고객사 전용 가능 인증, 권한, 설정 병합, 실행 통제, callback 수신 실행 중 필요한 최소 payload, 고객사 보관 계층 연결 정보 비밀값을 프론트에 노출하지 않음. 고객 데이터 장기 보관 최소화.
엔팔엔 프로세서고객사 전용 n8n 고객사 워크플로우 실행, 외부 API 호출, processorResult 생성 n8n 내부 Credential, 실행 중 binary, 프로세서 설정 Firestore 직접 수정 금지. 사용자 권한 판단 금지.
엔팔엔 보관고객사 DB/Storage 고객사 원본·결과·메타데이터 보관, 감사 추적, 다운로드 대상 관리 processorResult, originalFileRefs, resultRefs, 원본 파일, 결과 파일 공개 URL 무단 노출 금지. 권한 확인 없는 다운로드 금지.
n8n Credential외부 서비스 연결 정보 고객사 또는 고객사 위임 관리자 고객사 프로세서 내부에서 관리 Gmail, Drive, Sheets, LLM 등 연결 Credential 엔팔라이언트 설정값으로 Access Token/Refresh Token을 받지 않음.

5. 결과/보관 레벨과 고객사 보관

고객사는 자동화별로 보관 수준을 선택한다. 보관 수준이 높을수록 고객사 DB/Storage의 역할이 커진다.

Level 1. 알림/로그형

notify_only

가장 가벼운 적용 방식. 이메일 또는 실행 로그 중심으로 결과를 전달한다.

  • 최소 submission 로그
  • 이메일 보관함 활용 가능
  • 원본 영구 보관 없음

Level 2. 가공지식 저장형

processed_result

가공된 결과를 DB에 저장하여 검색과 축적이 가능해진다.

  • processorResult 저장
  • 구조화 데이터 축적
  • 원본은 처리 후 폐기 가능

Level 3. 원본 포함 보관형

full_archive

가공지식과 원본 파일을 함께 보관하는 고급형 구조다.

  • 고객사 Storage 원본 보관
  • originalFileRefs / resultRefs
  • 감사 추적과 재처리 가능

6. 사업화 관점의 경계

이 구조는 n8n 접근권 판매가 아니라 고객사 전용 자동화 프로세서 구축과 엔팔라이언트 운영 계층 제공으로 포지셔닝한다.

권장 포지션

고객사 전용 자동화 실행 환경 구축

대표님은 중앙 앱과 운영 체계를 제공하고, 고객사 클라우드에 전용 프로세서와 보관 계층을 구성해준다.

  • 워크플로우 구축·배포·운영 지원
  • 고객사 전용 n8n 프로세서 구성
  • 고객사 DB/Storage에 데이터 보관
  • 엔팔라이언트 앱 사용료·운영료·유지보수료 수취
주의 포지션

n8n 자체를 상품처럼 팔지 않는다

고객이 구매하는 것은 n8n 접근권이 아니라, 자동화 운영 포털과 고객사 전용 실행/보관 환경이다.

  • n8n UI 화이트라벨 제공 지양
  • 중앙 n8n 멀티테넌트 유료 접근권 지양
  • n8n Credential을 앱 설정값으로 노출 금지
  • 고객사 데이터와 Credential은 고객사 계층에 위치

최종 원칙

엔팔라이언트 애플리케이션은 중앙 운영 계층이다. 엔팔엔 프로세서와 엔팔엔 보관은 고객사 전용 실행·데이터 계층이다. 고객 데이터는 고객사 보관 계층에 두고, 엔팔라이언트는 자동화 운영, 권한, 설정, 실행, 결과 조회의 표준 경험을 제공한다.
