# 엔팔라이언트 결과 DB화/지식검색 계약 v2.0

- 문서명: 엔팔라이언트 결과 DB화/지식검색 계약
- 버전: v2.0 리뉴얼
- 작성일: 2026-06-16
- 문서 상태: 현행 지식 DB화 기준 문서
- 상위 문서: 결과/보관 레벨 계약 v2.0, DB/Storage 계약 v2.0

---

## 1. 목적

본 문서는 Level 2 `processed_result` 이상에서 생성된 지식을 DB에 저장하고, 사용자가 다시 열람·검색·공유할 수 있도록 하기 위한 최소 공통 구조를 정의한다.

엔팔라이언트는 처음부터 거대한 지식관리 시스템을 만들지 않는다. 다만 결과가 누적된 뒤에도 기본 검색과 후속 인덱싱이 가능하도록 최소 공통 껍데기를 맞춘다.

---

## 2. 적용 범위

적용 대상:

```text
Level 2 processed_result
Level 3 full_archive
```

Level 1 `notify_only`는 이메일/알림 중심이므로 지식 DB화 대상이 아니다.

다만 Level 1에서도 이메일 발송을 위해 MD 본문이나 MD 첨부파일을 임시 생성할 수 있다.

---

## 3. 기본 철학

```text
과도한 표준화는 피한다.
최소한의 공통 구조는 둔다.
워크플로우별 특수 데이터는 structuredData에 넣는다.
후가공성 데이터는 실행 결과와 분리한다.
통합 검색/벡터/백링크는 나중에 별도 인덱싱 단계에서 처리한다.
```

---

## 4. 최소 processorResult 구조

Level 2 이상에서 n8n은 가능한 한 다음 구조를 callback으로 반환한다.

```json
{
  "title": "결과 제목",
  "summary": "짧은 요약",
  "content": "본문 텍스트",
  "mdContent": "마크다운 본문",
  "hashtags": ["태그1", "태그2"],
  "attachments": [],
  "structuredData": {},
  "warnings": []
}
```

## 5. 필드 의미

### 5.1 title

결과 카드와 검색 결과의 제목으로 사용한다.

```text
processorResult.title을 최우선 표시 제목으로 사용한다.
비어 있으면 displayTitle, submissionTitle 순으로 fallback한다.
```

### 5.2 summary

검색 결과 카드의 짧은 설명이다.

```text
사람이 빠르게 내용을 파악할 수 있는 1~3문장 권장.
```

### 5.3 content

순수 텍스트 본문이다.

```text
검색 1차 대상.
마크다운 문법이 제거된 텍스트 권장.
```

### 5.4 mdContent

마크다운 본문이다.

```text
결과 상세 화면 표시.
MD 다운로드 동적 생성.
Optional Export MD 파일 생성에 활용.
```

### 5.5 hashtags

간단한 검색/분류용 태그다.

```text
과도한 자동 태그 생성 금지.
워크플로우별 최소 추천 태그만 사용.
```

### 5.6 attachments

결과 본문 내부에서 참조되는 첨부파일 메타데이터다.

Level 3의 `originalFileRefs`, `resultRefs`와 중복될 수 있으므로, 파일 원본 참조는 `originalFileRefs/resultRefs`를 우선한다.

### 5.7 structuredData

워크플로우별 특수 구조 데이터다.

예:

```text
회의 참석자
통화 상대방
할 일 목록
분류 결과
점수/평가 항목
```

### 5.8 warnings

처리 중 발생한 비차단 경고다.

예:

```text
첨부파일 일부 처리 실패
Optional Export 실패
음성 일부 구간 인식 불량
```

## 6. 검색 1차 대상

초기 검색은 다음 필드를 우선 대상으로 한다.

```text
processorResult.title
processorResult.summary
processorResult.content
processorResult.mdContent
processorResult.hashtags
workflowName
companyName
authorName
createdAt
```

검색 결과 카드 표시 항목:

```text
제목
요약
워크플로우명
작성자
회사명
생성일
첨부파일 여부
보관 레벨
```

---

## 7. 검색 단계

### 7.1 1차 검색

Firestore 기반 필드 검색과 필터링을 우선한다.

지원 후보:

```text
키워드 검색
워크플로우 필터
사용자 필터
날짜 필터
보관 레벨 필터
첨부파일 여부 필터
```

### 7.2 2차 검색

결과가 누적되면 별도 검색 인덱스를 구성할 수 있다.

```text
knowledgeSearchIndex
```

### 7.3 확장 검색

자연어 검색, 벡터 검색, 의미 검색은 확장 기능으로 둔다.

```text
embeddingIndex
```

워크플로우 실행 중 즉시 벡터 임베딩을 생성하는 것은 기본 정책이 아니다.

---

## 8. 원본성 결과와 후가공 결과 분리

워크플로우는 가능한 한 원본성 결과를 저장한다.

좋은 예:

```text
발화/회의/통화/문서 내용을 정리한 title, summary, content, mdContent
사용자가 검색할 수 있는 최소 hashtags
첨부파일 참조
```

피해야 할 예:

```text
자동 백링크 생성
지식 그래프 생성
벡터 임베딩 생성
과도한 자동 분류
후가공 규칙에 종속된 태그 대량 생성
```

후가공은 별도 인덱싱 워크플로우에서 처리한다.

---

## 9. MD 다운로드 정책

Level 2 이상에서는 MD 다운로드를 제공할 수 있다.

원칙:

```text
processorResult.mdContent 기반으로 요청 시점에 동적 생성한다.
동적 생성 MD는 Storage 결과 파일로 보지 않는다.
동적 생성 MD는 resultRefs로 보지 않는다.
```

MD 파일 자체를 저장하고 링크로 관리하는 경우:

```text
Firebase Storage 저장 → Level 3 성격
Google Drive 저장 → Optional Export 성격
```

---

## 10. 공유 정책

Level 2 이상에서 공유 기능을 제공할 수 있다.

최소 원칙:

```text
기본 접근 권한은 submission 권한을 따른다.
개인 사용자는 본인 결과만 공유 가능하다.
회사관리자는 소속 회사 결과를 회사 범위에서 공유할 수 있다.
공개 링크는 기본 제공하지 않는다.
```

공개 링크, 외부 공유, 만료 링크는 후속 확장 정책으로 분리한다.

---

## 11. 보안 주의사항

Level 2부터 결과 본문이 DB에 남는다.

따라서 아래 정책이 필요하다.

```text
접근 권한
보관 기간
삭제 요청
회사별 조회 범위
운영자 접근 범위
민감정보 마스킹
감사 로그
```

Level 2는 원본 파일을 보관하지 않지만, 결과 본문 자체에 개인정보나 업무기밀이 포함될 수 있다.

---

## 12. 금지 사항

```text
Level 1 결과를 검색 대상으로 다루는 것
Level 2에서 원본 파일을 기본 영구 보관하는 것
실행 중 과도한 지식 그래프/임베딩 생성
processorResult에 토큰/Secret/Credential 저장
검색 인덱스에 원본 파일 내용을 무단 복제
```
