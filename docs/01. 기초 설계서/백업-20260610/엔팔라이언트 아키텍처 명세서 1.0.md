# 엔팔라이언트 아키텍처 명세서 1.0

- 문서명: 엔팔라이언트 아키텍처 명세서
    
- 버전: v1.0
    
- 문서 상태: 아키텍처 초안
    
- 상위 문서: 엔팔라이언트 솔루션 헌법 1.0
    
- 적용 범위: 엔팔라이언트 애플리케이션, 엔팔라이언트 게이트웨이, 엔팔라이언트 스토리지, 엔팔라이언트 워크플로우, 인증/권한 구조, 실행 흐름, 운영 구조
    

---

## 1. 문서 목적

이 문서는 엔팔라이언트 솔루션 헌법에서 정의한 원칙을 실제 시스템 구조로 해석한 아키텍처 기준 문서다.

헌법이 엔팔라이언트의 철학과 책임 경계를 선언한다면, 본 문서는 그 원칙이 실제 서비스, 서버, 데이터 흐름, 저장소, 워크플로우 구조 안에서 어떻게 배치되는지를 정의한다.

본 문서는 세부 구현 문서가 아니다.  
Firestore 컬렉션별 상세 필드, Gateway API의 요청/응답 JSON, Firebase Security Rules, n8n 노드 코드, UI 컴포넌트 상세 설계는 하위 문서에서 다룬다.

본 문서의 목적은 다음과 같다.

1. 엔팔라이언트 솔루션의 전체 시스템 구조를 정의한다.
    
2. 각 서비스의 역할과 책임을 명확히 한다.
    
3. 오퍼레이터, 회사관리자, 개인사용자의 흐름을 시스템 구조와 연결한다.
    
4. 입력, 프로세서, 결과/보관, 운영/예외/추적의 4단계 구조를 실제 아키텍처에 반영한다.
    
5. n8n 워크플로우가 엔팔라이언트 플랫폼 안에서 어떤 위치를 갖는지 정의한다.
    
6. 향후 DB 규약서, Gateway API 규약서, 워크플로우 개발 규약서의 기준을 제공한다.

## 2. 엔팔라이언트 솔루션 전체 구조

엔팔라이언트 솔루션은 n8n 자동화를 고객사와 개인 사용자가 사용할 수 있도록 하는 자동화 운영 플랫폼이다.

전체 구조는 다음 계층으로 나뉜다.

```text
사용자 / 회사관리자 / 오퍼레이터
        ↓
n8lient-app
        ↓
n8lient-gateway
        ↓
n8lient-db / n8lient-storage
        ↓
n8lient-workflows
        ↓
외부 서비스
```

각 계층의 기본 의미는 다음과 같다.

- `n8lient-app`: 사용자가 보는 웹/앱 화면
    
- `n8lient-gateway`: 인증, 권한, 설정 병합, 파일 수신, n8n 호출을 담당하는 서버 API
    
- `n8lient-db`: Firestore 기반 데이터 계층
    
- `n8lient-storage`: Firebase Storage 기반 파일 보관 계층
    
- `n8lient-workflows`: n8n 기반 업무 프로세서 계층
    
- 외부 서비스: Gemini, Gmail, Google Calendar, Google Sheets, Google Drive 등
    

엔팔라이언트의 기본 호출 원칙은 다음과 같다.

```text
브라우저는 n8n을 직접 호출하지 않는다.
브라우저는 n8lient-gateway만 호출한다.
n8lient-gateway가 n8n을 서버 간 호출한다.
n8n은 처리 결과를 n8lient-gateway callback으로 반환한다.
```

## 3. 표준 서비스 구성

### 3.1 n8lient-app

`n8lient-app`은 엔팔라이언트 애플리케이션이다.

사용자가 접속하는 웹/앱 화면이며, 역할별 콘솔을 제공한다.

담당 영역은 다음과 같다.

- Google 로그인
    
- 회사 가입 요청
    
- 오퍼레이터 콘솔
    
- 회사관리자 콘솔
    
- 개인사용자 실행 화면
    
- 워크플로우 목록 조회
    
- 회사 공용 설정 화면
    
- 개인 설정 화면
    
- 실행 요청 화면
    
- 실행 결과 목록
    
- 실행 결과 상세 화면
    
- 권한 기반 다운로드 화면
    

`n8lient-app`은 비밀값을 보유하지 않는다.

금지 사항은 다음과 같다.

- n8n 서버 토큰 보유 금지
    
- Firebase Admin Key 보유 금지
    
- Callback Secret 보유 금지
    
- 브라우저에서 n8n Webhook 직접 호출 금지
    
- 브라우저에서 Firestore 권한을 우회하는 구조 금지
    

---

### 3.2 n8lient-gateway

`n8lient-gateway`는 엔팔라이언트의 서버 통제 지점이다.

사용자의 실행 요청은 반드시 Gateway를 통과해야 한다.  
Gateway는 실행 요청이 정당한지 확인하고, 필요한 설정을 병합한 뒤, n8n 워크플로우를 서버 간 호출한다.

담당 영역은 다음과 같다.

- Firebase ID Token 검증
    
- 사용자 승인 상태 확인
    
- clientId 검증
    
- role 검증
    
- 워크플로우 사용권 확인
    
- 회사 공용 설정 조회
    
- 개인 사용자 설정 조회
    
- 설정 병합
    
- canonicalPayload 생성
    
- 파일 수신
    
- Firebase Storage 원본 저장
    
- Firestore submissions 생성
    
- n8n Webhook 서버 간 호출
    
- callback 수신
    
- processorResult 저장
    
- 다운로드 권한 검증
    
- 외부 export 요청 중계
    

Gateway는 엔팔라이언트의 정책이 실제로 실행되는 지점이다.

따라서 다음 책임을 가진다.

```text
누가 실행할 수 있는가
어떤 워크플로우를 실행할 수 있는가
어떤 설정값이 적용되는가
원본 파일을 어디에 보관하는가
결과를 누구에게 보여줄 수 있는가
```

---

### 3.3 n8lient-db

`n8lient-db`는 Firestore 기반 데이터 계층이다.

담당 영역은 다음과 같다.

- 사용자 정보
    
- 고객사 정보
    
- 회사코드
    
- 가입 요청
    
- 워크플로우 마스터
    
- 회사별 워크플로우 사용권
    
- 회사 공용 설정
    
- 개인 사용자 설정
    
- 실행 이력
    
- 결과 메타데이터
    
- 오류 상태
    
- 운영 로그 일부
    

Firestore는 실행 상태와 결과 추적의 기준 저장소다.

Firestore에 저장하면 안 되는 것은 다음과 같다.

- 파일 원본
    
- base64 파일 본문
    
- binary
    
- Google Access Token
    
- Google Refresh Token
    
- n8n Credential ID
    
- API Key
    
- Secret
    

---

### 3.4 n8lient-storage

`n8lient-storage`는 Firebase Storage 기반 파일 보관 계층이다.

담당 영역은 다음과 같다.

- 원본 음성 파일
    
- 원본 이미지
    
- PDF, DOCX, MD 등 첨부 원본
    
- 생성된 결과 파일
    
- 다운로드 대상 파일
    
- export 전 임시 파일
    

파일 원본의 기본 보관소는 Firebase Storage다.

Google Drive는 기본 저장소가 아니다.  
Google Drive는 다음 경우에만 사용한다.

- Google Drive Watch 입력
    
- 고객 요청에 따른 선택 export
    
- 기존 자동화 호환을 위한 임시 저장소
    
- 개인 PoC 단계의 임시 입출력
    

사용자에게 제공되는 파일 링크는 원칙적으로 엔팔라이언트 권한 확인 링크여야 한다.  
Firebase Storage 공개 URL을 이메일이나 결과 화면에 직접 노출하지 않는다.

---

### 3.5 n8lient-workflows

`n8lient-workflows`는 n8n 기반 업무 프로세서 계층이다.

n8n은 자동화의 본체인 프로세서 역할을 수행한다.

담당 영역은 다음과 같다.

- 표준 payload 수신
    
- processorInput 생성 또는 수신
    
- 음성 인식
    
- 문서 분석
    
- LLM 호출
    
- 요약
    
- 변환
    
- 구조화
    
- processorResult 생성
    
- success / failed / config_error callback 전송
    

n8n이 담당하지 않는 영역은 다음과 같다.

- 사용자 인증
    
- 회사 권한 판단
    
- 회사/개인 설정 병합
    
- Firestore 직접 수정
    
- 장기 보관 정책 판단
    
- 사용자별 다운로드 권한 판단
    
- Credential을 사용자별로 동적 전환하는 구조
    

n8n은 가능한 한 다음 구조를 따른다.

```text
processorInput → 업무 처리 → processorResult
```

## 4. 사용자 역할 기반 구조

엔팔라이언트는 세 가지 역할을 기준으로 작동한다.

```text
오퍼레이터 → 회사관리자 → 개인사용자
```

권한과 책임은 상위 역할에서 하위 역할로 위임된다.

### 4.1 오퍼레이터

오퍼레이터는 엔팔라이언트 솔루션을 운영하는 개발회사 또는 서비스 제공자의 최종 관리자다.

주요 역할은 다음과 같다.

- 고객사 생성
    
- 회사관리자 지정
    
- 워크플로우 마스터 등록
    
- 워크플로우 버전 관리
    
- inputSchema 정의
    
- configSchema 정의
    
- 매뉴얼 등록
    
- 회사별 워크플로우 사용권 배정
    
- 기본 보관 정책 정의
    
- 오류 및 운영 로그 확인
    
- 워크플로우 업데이트 관리
    

오퍼레이터는 자동화 상품의 생산자다.

---

### 4.2 회사관리자

회사관리자는 고객사 내부의 책임자다.

주요 역할은 다음과 같다.

- 회사 사용자 승인
    
- 회사 사용자 관리
    
- 회사에 배정된 워크플로우 확인
    
- 회사 공용 설정 등록
    
- 회사 공용 설정 변경
    
- 회사 내 워크플로우 활성화/비활성화
    
- 회사 실행 결과 확인
    
- 회사 구성원에게 사용 안내
    

회사관리자는 오퍼레이터가 배정한 워크플로우만 사용할 수 있다.  
회사관리자는 n8n 내부 Credential이나 processor 로직을 수정하지 않는다.

---

### 4.3 개인사용자

개인사용자는 고객사에 소속된 실제 실행 사용자다.

주요 역할은 다음과 같다.

- 사용 가능한 워크플로우 확인
    
- 워크플로우 실행
    
- 개인 설정 등록
    
- 개인 설정 수정
    
- 자신의 실행 결과 조회
    
- 자신의 실행 이력 확인
    
- 허용된 경우 재실행 요청
    

개인사용자는 회사관리자가 허용한 범위 안에서만 개인 설정을 사용할 수 있다.

---

### 4.4 권한 위임과 설정 우선순위

정책 우선순위는 다음과 같다.

```text
오퍼레이터 정책 > 회사관리자 정책 > 개인사용자 설정
```

실행 설정값의 적용 우선순위는 다음과 같다.

```text
개인사용자 설정 > 회사 공용 설정 > 워크플로우 기본값
```

단, 개인사용자 설정은 오퍼레이터와 회사관리자가 허용한 항목에 한해 적용된다.

설정 변경은 원칙적으로 변경 이후의 새 실행부터 적용된다.  
이미 완료된 실행 이력과 결과는 사후 설정 변경으로 바꾸지 않는다.

## 5. 회원가입 및 인증 구조

엔팔라이언트의 기본 회원가입과 로그인은 Google 인증을 표준으로 한다.

사용자는 Google 계정으로 로그인한다.  
엔팔라이언트는 자체 ID/PW 기반 회원가입을 기본 제공하지 않는다.

인증 구조는 다음과 같다.

```text
Google Login
→ Firebase Authentication
→ Firebase Auth uid 발급
→ n8lient-app에서 ID Token 획득
→ n8lient-gateway에서 ID Token 검증
→ Firestore users 문서 기준으로 권한 판단
```

Google 인증은 신원 확인 수단이다.  
실제 엔팔라이언트 사용 권한은 Firestore의 users 문서에 있는 role, clientId, approvalStatus로 결정한다.

기본 가입 흐름은 다음과 같다.

1. 사용자가 Google 계정으로 로그인한다.
    
2. users 문서가 생성된다.
    
3. 사용자는 회사코드 또는 초대 링크로 가입 요청을 한다.
    
4. 회사관리자 또는 오퍼레이터가 승인한다.
    
5. 승인된 사용자는 자기 회사에 배정된 워크플로우를 사용할 수 있다.
    

승인되지 않은 사용자는 워크플로우를 실행할 수 없다.

## 6. 4단계 자동화 아키텍처

엔팔라이언트의 모든 자동화는 다음 4단계 구조를 따른다.

1. 입력/트리거 영역
    
2. 프로세서 영역
    
3. 결과/보관 영역
    
4. 운영/예외/추적 영역
    

### 6.1 입력/트리거 영역

입력/트리거 영역은 자동화 실행 요청을 받는 영역이다.

표준 트리거 유형은 다음과 같다.

- manual
    
- scheduled
    
- webhook
    
- watch
    
- retry
    
- system
    

표준 입력 유형은 다음과 같다.

- text
    
- file
    
- audio
    
- image
    
- form
    
- url
    

입력/트리거 영역은 다양한 입력을 canonicalPayload로 정규화한다.

---

### 6.2 프로세서 영역

프로세서 영역은 실제 업무 결과를 만드는 영역이다.

n8n 워크플로우는 이 영역에 집중해야 한다.

프로세서 영역은 입력 출처와 저장 위치에 직접 의존하지 않는다.

---

### 6.3 결과/보관 영역

결과/보관 영역은 결과를 어디에 저장하고 어떻게 사용자에게 보여줄지 결정하는 영역이다.

기본 원칙은 다음과 같다.

- 상태와 결과 메타데이터는 Firestore에 저장한다.
    
- 원본 파일과 결과 파일은 Firebase Storage에 저장한다.
    
- 사용자에게는 엔팔라이언트 결과 상세 링크를 제공한다.
    
- Google Drive는 선택 export 저장소로 둔다.
    

---

### 6.4 운영/예외/추적 영역

운영/예외/추적 영역은 실행 상태, 실패, 재시도, 예외, 감사 로그를 관리한다.

표준 상태는 다음과 같다.

- queued
    
- processing
    
- success
    
- failed
    
- skipped
    
- config_error
    

n8n은 Firestore를 직접 수정하지 않고, callback으로 결과를 반환한다.

## 7. 표준 실행 흐름

### 7.1 manual 실행 흐름

manual 실행은 사용자가 엔팔라이언트 화면에서 직접 실행하는 방식이다.

흐름은 다음과 같다.

```text
개인사용자
→ n8lient-app 실행 화면
→ n8lient-gateway
→ 권한/설정 확인
→ 파일이 있으면 n8lient-storage 저장
→ n8lient-workflows 호출
→ processorResult callback
→ n8lient-db 결과 갱신
→ 결과 화면 표시
```

---

### 7.2 webhook 실행 흐름

webhook 실행은 외부 시스템이 Gateway API를 호출하는 방식이다.

```text
외부 시스템
→ n8lient-gateway
→ API 인증/권한 검증
→ canonicalPayload 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

---

### 7.3 scheduled 실행 흐름

scheduled 실행은 정해진 시간이나 주기에 따라 실행되는 방식이다.

```text
스케줄러
→ n8lient-gateway 또는 scheduled adapter
→ canonicalPayload 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

---

### 7.4 watch 실행 흐름

watch 실행은 외부 저장소나 외부 소스에서 새 항목을 감지해 실행하는 방식이다.

```text
외부 저장소 감지
→ watch adapter
→ 표준 input으로 정규화
→ canonicalPayload 생성
→ n8lient-workflows 호출
→ callback
→ 결과 저장
```

Google Drive Watch는 watch adapter의 한 유형이다.

---

### 7.5 retry 실행 흐름

retry 실행은 기존 submission을 다시 처리하는 방식이다.

```text
기존 submission 선택
→ retry 요청
→ 기존 입력 또는 파일 참조 확인
→ 새 submission 생성
→ n8lient-workflows 재호출
→ callback
→ 결과 저장
```

---

## 8. 표준 데이터 흐름

엔팔라이언트와 n8n 사이의 데이터 계약은 다음 세 가지를 중심으로 한다.

### 8.1 canonicalPayload

canonicalPayload는 Gateway가 만든 표준 실행 요청이다.

역할:

- 실행 요청 전체를 표준화한다.
    
- 사용자, 회사, 워크플로우, 입력, 설정, callback 정보를 담는다.
    
- n8n으로 전달되는 기본 payload다.
    

생성 주체:

- n8lient-gateway
    

소비 주체:

- n8lient-workflows
    

---

### 8.2 processorInput

processorInput은 프로세서가 실제 업무 처리에 사용할 입력이다.

역할:

- 입력 출처와 실행 방식에 무관한 프로세서용 입력을 제공한다.
    
- manual, watch, webhook, scheduled 입력 차이를 흡수한다.
    

생성 주체:

- n8lient-gateway 또는 n8n의 Payload Normalize 영역
    

소비 주체:

- n8n Processor 영역
    

---

### 8.3 processorResult

processorResult는 프로세서가 만든 표준 결과다.

역할:

- 처리 결과를 엔팔라이언트가 저장하고 보여줄 수 있는 형태로 반환한다.
    
- 결과 요약, 본문, 구조화 데이터, 경고, 후속 조치 등을 포함한다.
    

생성 주체:

- n8lient-workflows
    

소비 주체:

- n8lient-gateway callback
    
- n8lient-db
    
- n8lient-app 결과 화면
    

---

## 9. 저장소 아키텍처

### 9.1 Firestore

Firestore는 실행 상태와 결과 메타데이터의 기준 저장소다.

담당 데이터:

- 사용자
    
- 회사
    
- 권한
    
- 워크플로우 마스터
    
- 회사별 사용권
    
- 회사 공용 설정
    
- 개인 설정
    
- 실행 이력
    
- 결과 메타데이터
    
- 오류 상태
    

Firestore에는 파일 원본을 저장하지 않는다.

---

### 9.2 Firebase Storage

Firebase Storage는 원본 파일과 결과 파일의 기본 보관소다.

담당 데이터:

- 음성 원본
    
- 이미지 원본
    
- PDF, DOCX, MD 등 파일 원본
    
- 생성된 결과 파일
    
- 다운로드 대상 파일
    

Storage 파일은 권한 확인 없이 공개하지 않는다.

---

### 9.3 Google Drive

Google Drive는 기본 저장소가 아니다.

사용 가능 위치는 다음과 같다.

- Watch 입력 소스
    
- Optional Export 저장소
    
- 고객 요청에 따른 선택 저장
    
- PoC 단계의 임시 저장소
    

Google Drive 저장 실패가 엔팔라이언트 기본 보관 실패로 이어지지 않아야 한다.

---

### 9.4 결과 조회와 다운로드

결과 조회는 엔팔라이언트 애플리케이션에서 제공한다.

파일 다운로드는 권한 확인을 거쳐야 한다.

기본 원칙은 다음과 같다.

```text
사용자는 자신에게 권한이 있는 submission의 결과만 조회할 수 있다.
파일 링크는 엔팔라이언트 권한 확인 계층을 통해 제공한다.
이메일에는 직접 파일 공개 URL을 넣지 않는다.
```

## 10. 엔팔라이언트 워크플로우 아키텍처

엔팔라이언트 워크플로우는 n8n에서 실행되는 프로세서다.

표준 노드 구간은 다음과 같다.

```text
[01] Trigger / Input
[02] Payload Normalize
[03] Input Validate
[04] File Normalize

[10] Processor Start
[11] AI / Transform
[12] Parse Result
[13] Build Processor Result

[20] Optional Export
[22] Callback

[90] Error / Exception
```

모든 워크플로우는 Sticky Note로 다음 영역을 구분해야 한다.

- N8Lient 입력/인증 영역
    
- Payload Normalize 영역
    
- Processor 영역
    
- Optional Export 영역
    
- Callback 영역
    
- Error / Exception 영역
    

n8n 워크플로우는 가능한 한 Processor 영역의 핵심 로직을 보존하고, 입력/결과/예외 영역은 표준 구조에 맞춘다.

- callback 수신 API

- 결과 조회 API

## 12. 권한 및 보안 아키텍처

엔팔라이언트는 인증과 권한을 분리한다.

인증은 Google / Firebase Authentication이 담당한다.  
권한은 Firestore의 users 문서와 role, clientId, approvalStatus를 기준으로 판단한다.

보안 원칙은 다음과 같다.

- 브라우저는 n8lient-gateway만 호출한다.
    
- 브라우저는 n8n 토큰을 알 수 없다.
    
- Gateway는 모든 실행 요청에서 Firebase ID Token을 검증한다.
    
- Gateway는 clientId 기준으로 데이터 접근을 제한한다.
    
- Firestore Rules는 클라이언트 직접 조회 범위를 제한한다.
    
- Firebase Storage 파일은 권한 확인 없이 공개하지 않는다.
    
- Credential, Token, Secret, API Key는 사용자 설정값으로 노출하지 않는다.
    
- n8n Credential은 n8n 내부 고정 Credential로 관리한다.

## 14. 운영/예외/추적 구조

운영 추적의 중심은 submission이다.

submission은 실행 요청 하나의 생애주기를 나타낸다.

기본 상태는 다음과 같다.

- queued
    
- processing
    
- success
    
- failed
    
- skipped
    
- config_error
    

오류는 표준 error code로 관리한다.

대표 오류는 다음과 같다.

- REQUIRED_SETTING_MISSING
    
- REQUIRED_INPUT_MISSING
    
- REQUIRED_INPUT_FILE_MISSING
    
- UNSUPPORTED_FILE_TYPE
    
- MAX_UPLOAD_EXCEEDED
    
- RESOURCE_PERMISSION_DENIED
    
- STORAGE_UPLOAD_FAILED
    
- PROCESSOR_FAILED
    
- CALLBACK_FAILED
    
- EXTERNAL_API_FAILED
    

n8n은 오류 발생 시 가능한 한 failed 또는 config_error callback을 반환한다.  
Gateway는 callback을 수신해 submissions 상태를 갱신한다.

## 15. 버전 및 변경 관리

엔팔라이언트는 워크플로우와 설정 구조의 변경을 추적해야 한다.

관리 대상은 다음과 같다.

- workflowTemplate version
    
- configSchema version
    
- inputSchema version
    
- processor version
    
- manual version
    
- company setting updatedAt
    
- user setting updatedAt
    

변경 원칙은 다음과 같다.

- 설정 변경은 변경 이후의 새 실행부터 적용된다.
    
- 완료된 submission의 실행 당시 설정과 결과는 변경하지 않는다.
    
- breaking change가 있으면 버전 변경 또는 마이그레이션 안내가 필요하다.
    
- 오퍼레이터는 워크플로우 업데이트 시 회사관리자가 다시 확인해야 할 설정을 명시해야 한다.

## 16. 확장 아키텍처

초기 MVP에서는 다음 서비스가 핵심 실행 서비스다.

```text
n8lient-app
n8lient-gateway
n8lient-db
n8lient-storage
n8lient-workflows
```

아래 항목은 초기에는 gateway 내부 모듈 또는 운영 스크립트로 둘 수 있다.

- n8lient-reporter
    
- n8lient-scheduler
    
- n8lient-exporter
    
- n8lient-watch-adapter
    
- n8lient-admin-tools
    

기능이 반복 사용되거나 독립 운영이 필요해질 경우 별도 서비스로 분리할 수 있다.

분리 기준은 다음과 같다.

- 실행 시간이 길어 Gateway 요청 처리에 부담이 되는 경우
    
- 독립적인 재시도 또는 큐 처리가 필요한 경우
    
- 장애 격리가 필요한 경우
    
- 로그와 권한을 별도로 관리해야 하는 경우
    
- 여러 워크플로우에서 공통으로 반복 사용되는 경우

## 17. 표준 위반 아키텍처

다음 구조는 엔팔라이언트 아키텍처 표준 위반으로 본다.

- n8n이 Firestore를 직접 수정하는 구조

- Firestore에 파일 원본을 저장하는 구조
    
- Firebase Storage 공개 URL을 권한 확인 없이 노출하는 구조
    
- Credential, Token, API Key가 회사관리자 또는 개인사용자 설정값으로 노출되는 구조
