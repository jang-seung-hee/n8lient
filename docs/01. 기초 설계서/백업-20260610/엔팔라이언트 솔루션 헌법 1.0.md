# 엔팔라이언트 솔루션 헌법 1.0

- 문서명: 엔팔라이언트 솔루션 헌법 1.0
    
- 버전: v1.0
    
- 문서 상태: 최상위 정책 초안
    
- 적용 범위: 엔팔라이언트 애플리케이션, 엔팔라이언트 게이트웨이, 엔팔라이언트 워크플로우, 엔팔라이언트 스토리지, 엔팔라이언트 운영 정책
    
- 목적: 엔팔라이언트 솔루션의 철학, 역할, 책임, 설계 원칙, 개발 기준을 정의한다.
    

---

## 1. 선언

엔팔라이언트 솔루션은 n8n 자동화를 고객사와 개인 사용자가 쉽게 사용할 수 있도록 만드는 자동화 운영 플랫폼이다.

엔팔라이언트는 n8n을 대체하지 않는다.  
엔팔라이언트는 n8n 워크플로우를 고객사와 개인 사용자의 업무 환경에 맞게 설정, 실행, 보관, 추적, 관리할 수 있도록 돕는다.

엔팔라이언트의 핵심 목적은 다음과 같다.

1. 잘 만든 n8n 워크플로우를 여러 고객사와 여러 사용자가 안정적으로 사용할 수 있게 한다.
    
2. 자동화별 핵심 처리 로직은 가볍게 유지하고, 입력·설정·권한·보관·추적은 플랫폼이 담당한다.
    
3. 회사 공용 설정과 개인 사용자 설정을 함께 지원한다.
    
4. 자동화 결과와 원본 자료를 권한 기반으로 안전하게 보관하고 조회한다.
    
5. 모든 자동화는 입력/트리거, 프로세서, 결과/보관, 운영/예외/추적의 4단계 구조를 따른다.
    

---

## 2. 엔팔라이언트 솔루션의 구성

엔팔라이언트 솔루션은 다음 구성 요소로 나뉜다.

### 2.1 엔팔라이언트 애플리케이션

엔팔라이언트 애플리케이션은 사용자, 회사관리자, 오퍼레이터가 사용하는 웹/앱 화면이다.

담당 역할은 다음과 같다.

- 로그인
    
- 회사 가입 요청
    
- 사용자 승인
    
- 고객사 관리
    
- 워크플로우 등록
    
- 워크플로우 사용권 배정
    
- 회사 공용 설정 관리
    
- 개인 설정 관리
    
- 실행 요청
    
- 실행 결과 조회
    
- 결과 상세 화면
    
- 권한 기반 다운로드
    
- 운영자 관리 화면
    

### 2.2 엔팔라이언트 게이트웨이

엔팔라이언트 게이트웨이는 애플리케이션과 n8n 사이의 서버 API 계층이다.

담당 역할은 다음과 같다.

- 사용자 인증 검증
    
- 회사 및 사용자 권한 확인
    
- 워크플로우 사용권 확인
    
- 회사 공용 설정과 개인 설정 병합
    
- canonical payload 생성
    
- 파일 수신
    
- 원본 파일 보관
    
- n8n 서버 간 호출
    
- callback 수신
    
- Firestore 실행 이력 갱신
    
- 다운로드 권한 검증
    
- 외부 API 연동 진입점 관리
    

### 2.3 엔팔라이언트 워크플로우

엔팔라이언트 워크플로우는 n8n에서 실행되는 자동화 프로세서다.

담당 역할은 다음과 같다.

- 표준 payload 수신
    
- processorInput 생성
    
- 실제 업무 처리
    
- LLM 호출
    
- 문서 생성
    
- 요약
    
- 변환
    
- 구조화
    
- processorResult 생성
    
- success / failed / config_error callback 전송
    

엔팔라이언트 워크플로우는 사용자 인증, 회사 권한 판단, 개인/회사 설정 병합, 장기 보관 정책 판단, Firestore 직접 수정, 사용자별 다운로드 권한 판단을 담당하지 않는다.

### 2.4 엔팔라이언트 스토리지

엔팔라이언트 스토리지는 원본 파일, 결과 파일, 메타데이터, 실행 로그를 보관하는 저장 계층이다.

기본 원칙은 다음과 같다.

- 원본 파일의 기본 저장소는 Firebase Storage다.
    
- 실행 상태, 결과 요약, 메타데이터, 오류 정보는 Firestore에 저장한다.
    
- Google Drive는 기본 저장소가 아니라 입력 Watch 또는 선택 export 저장소다.
    
- 이메일에는 직접 공개 파일 URL을 넣지 않고, 엔팔라이언트 권한 확인 링크를 넣는다.
    
- 사용자는 권한이 확인된 자료만 조회하거나 다운로드할 수 있다.

## 3. 4단계 자동화 원칙

엔팔라이언트의 모든 자동화는 다음 4단계 구조로 설계한다.

1. 입력/트리거 영역
    
2. 프로세서 영역
    
3. 결과/보관 영역
    
4. 운영/예외/추적 영역
    

자동화별 핵심 차이는 원칙적으로 프로세서 영역에서 발생한다.  
입력 방식, 저장 방식, 결과 조회, 예외 처리, 권한 확인은 가능한 한 엔팔라이언트 공통 기능으로 흡수한다.

---

## 4. 1단계: 입력/트리거 영역

입력/트리거 영역은 자동화 실행 요청을 받는 영역이다.

이 영역의 목적은 다양한 실행 수단을 하나의 표준 입력 구조로 정규화하는 것이다.

### 4.1 표준 트리거 유형

엔팔라이언트는 다음 트리거 유형을 표준으로 인정한다.

- manual: 사용자가 엔팔라이언트 화면에서 직접 실행
    
- scheduled: 정해진 시간이나 주기에 따라 실행
    
- webhook: 외부 시스템이 Gateway API를 호출하여 실행
    
- watch: 외부 저장소나 소스에서 새 항목을 감지하여 실행
    
- retry: 기존 실행 건을 재실행
    
- system: 운영자 또는 시스템 내부 작업으로 실행
    

Google Drive Watch는 제거 대상이 아니다.  
Google Drive Watch는 입력/트리거 영역의 입력 어댑터로 인정한다.

### 4.2 표준 입력 유형

엔팔라이언트는 입력 유형을 다음 범위로 표준화한다.

- text
    
- file
    
- audio
    
- image
    
- form
    
- url
    

새로운 입력 수단이 추가되더라도 먼저 위 표준 입력 유형 중 하나로 정규화한다.

---

## 5. 2단계: 프로세서 영역

프로세서 영역은 자동화의 본체다.

프로세서는 정규화된 입력을 받아 실제 업무 결과를 만든다.

예시는 다음과 같다.

- 음성 인식
    
- 대화 요약
    
- 일정 추출
    
- 회의록 생성
    
- 지식노트 생성
    
- 문서 변환
    
- 키워드 추출
    
- 할 일 추출
    
- 구조화된 JSON 생성
    
- Markdown 생성
    

프로세서는 입력 수단이나 저장 방식에 직접 의존하지 않아야 한다.

프로세서는 가능한 한 다음 구조를 따른다.

```text
processorInput → 업무 처리 → processorResult
```

프로세서는 Google Drive 폴더 ID, 이메일 수신자, 사용자 권한, 회사 설정 병합, 장기 보관 정책을 직접 판단하지 않는다.

---

## 6. 3단계: 결과/보관 영역

결과/보관 영역은 프로세서가 만든 결과를 어디에 저장하고 어떻게 사용자에게 보여줄지 결정하는 영역이다.

이 영역은 자동화별 핵심 업무 로직이 아니라 플랫폼의 보관 정책과 출력 정책에 속한다.

### 6.1 기본 보관 원칙

엔팔라이언트의 기본 결과 보관 정책은 다음과 같다.

- 결과 요약, 상태, 로그, 메타데이터는 Firestore에 저장한다.
    
- 원본 음성, 이미지, 첨부파일은 Firebase Storage에 저장한다.
    
- 결과 상세 화면은 엔팔라이언트 애플리케이션에서 제공한다.
    
- 다운로드는 엔팔라이언트 권한 확인 후 제공한다.
    
- 이메일 리포트에는 엔팔라이언트 결과 링크를 포함한다.
    
- Google Drive는 기본 저장소가 아니라 선택 export 저장소로 둔다.
    

### 6.2 Google Drive의 지위

Google Drive는 다음 용도로 사용할 수 있다.

- 입력 Watch 소스
    
- 개인 실험형 워크플로우의 임시 저장소
    
- 고객 요청에 따른 선택 export 저장소
    
- 기존 자동화 호환을 위한 과도기적 저장소
    

Google Drive 저장 실패가 엔팔라이언트 기본 보관 실패로 이어져서는 안 된다.  
기본 보관의 책임은 Firebase Storage와 Firestore가 담당한다.

---

## 7. 4단계: 운영/예외/추적 영역

운영/예외/추적 영역은 자동화 실행의 상태와 장애 대응을 담당한다.

처리 대상은 다음과 같다.

- success callback
    
- failed callback
    
- config_error callback
    
- 실행 로그
    
- 재시도 정보
    
- 권한 오류
    
- 파일 초과 오류
    
- 외부 API 오류
    
- 원본 보관 정책
    
- 삭제 정책
    
- 감사 로그
    

submission은 최소한 다음 상태를 가진다.

- queued
    
- processing
    
- success
    
- failed
    
- skipped
    
- config_error
    

n8n은 Firestore submissions를 직접 수정하지 않는다.  
결과 반영은 엔팔라이언트 Gateway callback API를 통해 수행한다.

---

## 8. 표준 데이터 계약

엔팔라이언트와 n8n 사이의 데이터 계약은 다음 세 가지로 표준화한다.

### 8.1 canonicalPayload

canonicalPayload는 엔팔라이언트가 실행 요청 전체를 정규화한 표준 실행 요청이다.

포함 성격은 다음과 같다.

- submissionId
    
- clientId
    
- uid
    
- workflowKey
    
- automationId
    
- trigger
    
- settings
    
- input
    
- requestedAt
    
- callbackUrl
    

### 8.2 processorInput

processorInput은 n8n 프로세서가 실제 처리에 필요한 입력만 추출한 값이다.

포함 성격은 다음과 같다.

- 입력 제목
    
- 입력 텍스트
    
- primaryFile
    
- 보조 파일 목록
    
- 입력 폼 값
    
- 원본 메타데이터
    
- 처리 옵션
    

### 8.3 processorResult

processorResult는 n8n이 처리 후 엔팔라이언트에 반환하는 표준 결과다.

포함 성격은 다음과 같다.

- title
    
- summary
    
- content
    
- mdContent
    
- structuredData
    
- keywords
    
- warnings
    
- resultType
    
- sourceRefs
    
- confidence
    
- nextAction
    

n8n 프로세서는 canonicalPayload 전체에 과도하게 의존하지 않고 processorInput 중심으로 동작해야 한다.

## 9. 웹/앱 역할 및 권한 위임 정책

엔팔라이언트 웹/앱은 세 가지 역할로 나뉜다.

1. 오퍼레이터
    
2. 회사관리자
    
3. 개인사용자
    

권한과 책임은 오퍼레이터에서 회사관리자, 회사관리자에서 개인사용자로 위임된다.

```text
오퍼레이터 → 회사관리자 → 개인사용자
```

정책 우선순위는 다음과 같다.

```text
오퍼레이터 정책 > 회사관리자 정책 > 개인사용자 설정
```

실행 설정값의 적용 우선순위는 다음과 같다.

```text
개인사용자 설정 > 회사 공용 설정 > 워크플로우 기본값
```

단, 개인사용자 설정은 오퍼레이터와 회사관리자가 허용한 항목에 한해 적용된다.

---

## 10. 오퍼레이터 역할

오퍼레이터는 엔팔라이언트 솔루션을 운영하는 개발회사 또는 서비스 제공자의 최종 관리자다.

오퍼레이터는 자동화의 생산자이며, 플랫폼의 기준을 정의한다.

### 10.1 오퍼레이터가 할 수 있는 것

- 고객사 생성 및 관리
    
- 회사관리자 지정 및 권한 관리
    
- 워크플로우 마스터 등록
    
- 워크플로우 버전 관리
    
- 워크플로우 배포, 중지, 비공개 전환
    
- inputSchema 정의
    
- configSchema 정의
    
- 필수 설정값 정의
    
- 매뉴얼 등록
    
- 회사별 워크플로우 사용권 배정
    
- 기본 보관 정책 정의
    
- 개인 설정 허용 여부 정의
    
- 결과/보관/리포팅 정책 정의
    
- 운영 로그 및 오류 추적
    

### 10.2 오퍼레이터가 해야 하는 것

- 워크플로우를 엔팔라이언트 4단계 표준에 맞게 등록해야 한다.
    
- 회사관리자가 설정해야 할 필수값과 선택값을 구분해야 한다.
    
- 회사관리자용 매뉴얼과 개인사용자용 매뉴얼을 구분해 제공해야 한다.
    
- 워크플로우 업데이트 시 기존 회사 설정과 사용자 설정에 미치는 영향을 표시해야 한다.
    
- breaking change가 있는 경우 버전 변경 또는 마이그레이션 절차를 제공해야 한다.
    

### 10.3 오퍼레이터가 하면 안 되는 것

- 회사관리자의 회사별 설정을 임의로 덮어쓰면 안 된다.
    
- 개인사용자의 개인 설정을 임의로 변경하면 안 된다.
    
- Credential, API Key, Token 같은 비밀값을 회사관리자나 개인사용자 설정값으로 노출하면 안 된다.

## 11. 회사관리자 역할

회사관리자는 고객사 내부의 책임자다.

회사관리자는 오퍼레이터가 배포한 워크플로우 사용권을 바탕으로 자기 회사에 맞는 회사 공용 설정을 등록하고, 회사 소속 사용자에게 배포한다.

### 11.1 회사관리자가 할 수 있는 것

- 자기 회사 사용자 승인 및 거절
    
- 자기 회사 사용자 목록 조회
    
- 자기 회사에 배정된 워크플로우 목록 확인
    
- 워크플로우별 회사 공용 기본 설정 등록
    
- 회사 공용 설정 수정
    
- 회사 내 워크플로우 활성화/비활성화
    
- 회사 공용 보관 위치 설정
    
- 회사 공용 보고 이메일 설정
    
- 회사 사용자 실행 결과 조회
    
- 회사 공용 매뉴얼 확인
    

### 11.2 회사관리자가 해야 하는 것

- 오퍼레이터가 정의한 필수 설정값을 회사 기준으로 등록해야 한다.
    
- 회사 공용 설정이 정상인지 테스트한 뒤 사용자에게 배포해야 한다.
    
- 회사 소속 사용자만 승인해야 한다.
    
- 회사 공용 저장소나 리소스가 필요한 경우 권한 공유를 완료해야 한다.
    
- 회사 정책상 개인 설정을 허용할지 제한할지 결정해야 한다.
    
- 워크플로우 업데이트가 발생하면 회사 설정을 다시 검토해야 한다.
    

### 11.3 회사관리자가 하면 안 되는 것

- 다른 회사의 사용자, 설정, 실행 결과를 조회하거나 수정할 수 없다.
    
- 오퍼레이터가 배정하지 않은 워크플로우를 임의로 사용할 수 없다.
    
- 워크플로우의 processor 로직이나 n8n 내부 Credential을 수정할 수 없다.
    
- 개인사용자의 개인 설정을 임의로 변경할 수 없다.

## 12. 개인사용자 역할

개인사용자는 고객사에 소속되어 실제 워크플로우를 실행하는 사용자다.

개인사용자는 회사관리자가 배포한 워크플로우를 확인하고, 회사 공용 설정을 그대로 사용할지 또는 허용된 범위 안에서 개인 설정을 등록해 사용할지 선택할 수 있다.

### 12.1 개인사용자가 할 수 있는 것

- 자신에게 허용된 워크플로우 목록 확인
    
- 워크플로우 실행
    
- 자신의 실행 결과 조회
    
- 자신의 개인 설정 등록
    
- 자신의 개인 설정 수정
    
- 회사 기본 설정 사용
    
- 개인 설정과 회사 기본 설정 중 적용 상태 확인
    
- 자신의 실행 이력 확인
    
- 재실행이 허용된 경우 재실행 요청
    

### 12.2 개인사용자가 해야 하는 것

- 회사가 승인한 계정으로만 사용해야 한다.
    
- 개인 설정을 사용할 경우 필요한 리소스 권한을 직접 준비해야 한다.
    
- 필수 입력값을 채워 실행해야 한다.
    
- 개인 저장소, 개인 이메일, 개인 옵션을 사용할 때 회사 정책을 따라야 한다.
    

### 12.3 개인사용자가 하면 안 되는 것

- 다른 사용자의 실행 결과를 조회할 수 없다.
    
- 다른 사용자의 개인 설정을 조회하거나 수정할 수 없다.
    
- 회사 공용 설정을 수정할 수 없다.
    
- configSchema 밖의 임의 설정을 추가할 수 없다.
    
- Credential, Token, API Key를 개인 설정에 등록할 수 없다.
    
- 회사관리자가 금지한 개인화 항목을 override할 수 없다.

## 13. 설정 변경 원칙

회사관리자는 회사 공용 설정을 언제든지 변경할 수 있다.  
개인사용자는 허용된 범위 안에서 개인 설정을 언제든지 변경할 수 있다.

다만 설정 변경은 원칙적으로 변경 이후의 새 실행부터 적용된다.

이미 완료된 submissions의 실행 당시 설정값과 결과는 변경하지 않는다.  
실행 당시 적용된 설정 요약은 추적 가능해야 한다.

개인 설정 중 빈 값은 회사 공용 설정을 덮어쓰지 않으며, 회사 기본값으로 fallback된다.

## 14. 워크플로우 배포 상태

워크플로우는 다음 상태를 가질 수 있다.

- draft: 오퍼레이터가 작성 중인 상태
    
- published: 회사에 배정 가능한 상태
    
- assigned: 특정 회사에 사용권이 배정된 상태
    
- configured: 회사관리자가 필수 설정을 완료한 상태
    
- enabled: 개인사용자가 실행할 수 있는 상태
    
- disabled: 일시 중지 상태
    
- deprecated: 더 이상 신규 사용을 권장하지 않는 상태
    
- archived: 보관 상태
    

개인사용자는 configured 및 enabled 상태의 워크플로우만 실행할 수 있다.

## 15. 실험형 워크플로우 PoC 원칙

엔팔라이언트 표준 편입 전 단계에서는 개인 n8n 환경에서 빠르게 실험형 워크플로우를 만들 수 있다.

이때 Google Drive Watch, 개인 Google Drive 저장, 개인 Gmail 발송 등 단순한 입출력 방식을 임시로 허용한다.

다만 향후 엔팔라이언트 편입 가능성이 있는 워크플로우는 처음부터 입력/트리거, 프로세서, 결과/보관, 운영/예외/추적 영역을 분리해야 한다.

PoC 단계의 Google Drive Watch는 입력 어댑터로 본다.  
개인 Google Drive 저장은 Optional Export로 본다.

프로세서 영역은 입력 수단과 저장 위치에 직접 의존하지 않아야 하며, 가능한 한 processorInput을 받아 processorResult를 만드는 구조로 설계한다.

하드코딩된 폴더 ID, 파일명 접두어, 수신 이메일, 처리 옵션은 나중에 settings.xxx로 분리할 수 있도록 한 곳에 모아야 한다.

## 16. n8n 워크플로우 설계 원칙

모든 엔팔라이언트 워크플로우는 4단계 구조가 노드명과 Sticky Note에 드러나야 한다.

권장 노드 구간은 다음과 같다.

```text
[01] Trigger / Input
[02] Payload Normalize
[03] Input Validate
[04] File Normalize

[10] Processor Start
[11] AI / Transform
[12] Parse Result
[13] Build Processor Result

[20] Optional Export
[22] Callback

[90] Error / Exception
```

기존 정상 동작 중인 업무 처리 노드는 불필요하게 수정하지 않는다.  
입력 노드와 결과 노드는 공통 구조에 맞춰 교체 가능하게 만든다.

---

## 17. 표준 위반 사례

다음 구조는 엔팔라이언트 표준 위반으로 본다.

- n8n이 Firestore를 직접 조회하거나 수정하는 구조
    
- n8n이 사용자 권한을 판단하는 구조
    
- n8n이 회사/개인 설정 병합을 수행하는 구조
    
- Google Drive를 기본 원본 저장소로 전제하는 구조
    
- 이메일에 Firebase Storage 공개 URL을 직접 넣는 구조
    
- 프로세서가 특정 입력 수단에 강하게 묶이는 구조
    
- 프로세서가 저장 위치나 발송 정책을 직접 판단하는 구조
    
- Credential, Token, API Key가 회사관리자 또는 개인사용자 설정값으로 노출되는 구조

## 18. 변경 관리 원칙

엔팔라이언트 애플리케이션, 게이트웨이, 스토리지, 워크플로우 수정은 본 헌법의 책임 경계를 기준으로 판단한다.

새 기능을 추가할 때는 먼저 해당 기능이 다음 중 어디에 속하는지 분류한다.

- 입력/트리거
    
- 프로세서
    
- 결과/보관
    
- 운영/예외/추적
    

기능의 위치가 불명확한 경우 구현보다 분류를 먼저 한다.  
분류 없이 기능을 추가하지 않는다.

## 19. 적용 순서

본 헌법은 다음 순서로 적용한다.

1. 정책 및 아키텍처 문서 확정
    
2. 엔팔라이언트 애플리케이션과 게이트웨이를 4단계 책임 경계에 맞게 수정
    
3. 엔팔라이언트 스토리지 정책을 Firebase Storage와 Firestore 기준으로 정비
    
4. 기존 n8n 워크플로우를 프로세서 중심 구조로 재정렬
    
5. 신규 워크플로우는 처음부터 본 헌법을 기준으로 설계
    
6. 기존 워크플로우는 중요도와 사용 빈도에 따라 순차 이관

## 20. 최종 원칙

엔팔라이언트는 자동화 플랫폼이다.  
n8n은 자동화 프로세서다.

엔팔라이언트는 입력, 설정, 권한, 보관, 결과 조회, 예외 추적을 담당한다.  
n8n은 정규화된 입력을 받아 업무 결과를 만드는 데 집중한다.

기본 저장소는 Firebase Storage와 Firestore다.  
Google Drive는 Watch 입력 또는 선택 export다.

모든 자동화는 4단계 구조를 따른다.  
모든 권한은 오퍼레이터에서 회사관리자, 회사관리자에서 개인사용자로 위임된다.  
모든 결과는 권한 기반으로 조회되어야 한다.
