# [지식 DB화/검색 최소 계약 개정 메모]

이 문서는 2026-06-12 기준으로 확정된 엔팔라이언트 결과 DB화 및 지식 검색 최소 구조를 추가 반영한 보존형 개정본이다.

기존 원문은 삭제하지 않는다.  
다만 Level 2 `processed_result` 이상에서 DB에 저장되는 결과는 아래 최소 공통 구조를 우선 해석 기준으로 삼는다.

## 핵심 원칙

```text
Level 1 notify_only = 이메일/알림 중심. 자유 포맷 허용.
Level 2 processed_result 이상 = 최소 공통 DB 포맷 적용.
Level 3 full_archive = 최소 공통 DB 포맷 + 원본/결과 파일 참조 보관.
```

통합 검색, 벡터라이징, 자동 분류, 백링크, 지식 그래프는 워크플로우 실행 단계에서 수행하지 않는다.  
이런 후가공은 나중에 `knowledgeSearchIndex`, `embeddingIndex` 또는 별도 인덱싱/마이그레이션 워크플로우에서 처리한다.

## 최소 공통 결과 구조

Level 2 이상에서 `processorResult`는 가능한 한 아래 공통 필드를 맞춘다.

```json
{
  "title": "결과 제목",
  "summary": "짧은 요약",
  "content": "본문 텍스트",
  "mdContent": "마크다운 본문",
  "hashtags": ["태그1", "태그2"],
  "attachments": [],
  "structuredData": {},
  "warnings": []
}
```

## 검색 1차 대상

초기 검색은 다음 필드를 우선 대상으로 한다.

```text
title
summary
content
mdContent
hashtags
authorName
workflowName
companyName
createdAt
```

검색 결과 카드에는 다음 라벨을 표시할 수 있어야 한다.

```text
제목
요약
워크플로우명
작성자
회사명
생성일
첨부파일 여부
보관 레벨
```

## 원본성 결과와 후가공 결과 분리

워크플로우는 가능한 한 원본성 결과를 저장한다.

```text
좋은 예:
- 발화/회의/통화/문서 내용을 정리한 title, content, mdContent
- 사용자가 나중에 검색할 수 있는 최소 hashtags
- 첨부파일 참조

피해야 할 예:
- 자동 백링크 생성
- 지식 그래프 생성
- 벡터 임베딩 생성
- 과도한 자동 분류
- 후가공 규칙에 종속된 태그 대량 생성
```

## 저장 계층

```text
submissions = 원본형 실행 결과와 최소 공통 processorResult 저장
knowledgeSearchIndex = 향후 통합 검색/자연어 검색용 후가공 인덱스
embeddingIndex = 필요 시 Level 2 이상 결과를 대상으로 생성하는 벡터 인덱스
```

Obsidian과 Google Drive Export는 기본 지식관리 저장소가 아니다.  
엔팔라이언트의 기준 저장소는 Firestore + Firebase Storage다.  
Obsidian, Google Drive, Markdown Export는 Optional Export로만 본다.

# [보존형 개정 메모]

이 문서는 기존 원문을 삭제하거나 요약하지 않고 보존한 상태에서, 2026-06-12 기준으로 확정된 결과/보관 정책 계층 구조를 우선 해석 기준으로 추가한 보존형 개정본이다.

기존 문서의 철학, 서비스 책임 경계, 4단계 자동화 구조는 유지한다.  
다만 결과/보관 정책 관련 용어와 우선순위는 아래 최신 정책을 우선한다.

## 최신 결과/보관 정책 계층

```text
워크플로우 마스터 = 기술적 최대치
회사별 계약/매핑 = 계약 한도
회사관리자 설정 = 회사 권장값
개인사용자 설정 = 개인 선택
Gateway = 최종 계산 및 방어
n8n = Gateway가 내려준 최종 retentionPolicy에 따른 분기 처리
```

## 강제 한도

```text
workflowTemplates.retentionCapabilities
→ clientContracts.contractRetentionLimit
→ clientAutomations.contractRetentionLimit
```

워크플로우가 기술적으로 지원하지 않는 단계, 또는 회사가 계약하지 않은 단계는 선택할 수 없다.

## 실행 선택 우선순위

```text
userAutomationSettings.userRetentionPreference
→ clientAutomations.companyRetentionPolicy.recommendedLevel
→ workflowTemplates.retentionCapabilities.defaultLevel
→ 시스템 기본값 full_archive
```

회사관리자의 설정은 강제값이 아니라 회사 권장값이다.  
개인사용자가 별도 설정을 하면 개인 설정이 회사 권장값보다 우선한다.  
단, 개인 설정은 워크플로우 기술 한도와 회사별 계약 한도를 초과할 수 없다.

## 핵심 필드

```text
workflowTemplates.retentionCapabilities
clientContracts.contractRetentionLimit
clientAutomations.contractRetentionLimit
clientAutomations.companyRetentionPolicy
userAutomationSettings.userRetentionPreference
submissions.retentionPolicySnapshot
submissions.retentionPolicySnapshot.resolvedFrom
```

## 최신 계산 원칙

```text
selectableLevels =
workflowTemplates.retentionCapabilities.supportedLevels
∩ clientAutomations.contractRetentionLimit.allowedLevels
```

회사 권장값은 선택 가능 범위를 제한하지 않는다.  
Gateway는 잘못된 요청이 들어와도 최종 방어선으로 한도 내 보정 또는 오류 처리를 수행한다.

# 엔팔라이언트 솔루션 헌법 1.1

- 문서 상태: 최상위 정책 초안

- 적용 범위: 엔팔라이언트 애플리케이션, 엔팔라이언트 게이트웨이, 엔팔라이언트 워크플로우, 엔팔라이언트 스토리지, 엔팔라이언트 운영 정책
- 목적: 엔팔라이언트 솔루션의 철학, 역할, 책임, 설계 원칙, 개발 기준을 정의한다.

---

## 1. 선언

엔팔라이언트 솔루션은 n8n 자동화를 고객사와 개인 사용자가 쉽게 사용할 수 있도록 만드는 자동화 운영 플랫폼이다.

엔팔라이언트는 n8n을 대체하지 않는다.  
엔팔라이언트는 n8n 워크플로우를 고객사와 개인 사용자의 업무 환경에 맞게 설정, 실행, 보관, 추적, 관리할 수 있도록 돕는다.

엔팔라이언트의 핵심 목적은 다음과 같다.

1. 잘 만든 n8n 워크플로우를 여러 고객사와 여러 사용자가 안정적으로 사용할 수 있게 한다.
2. 자동화별 핵심 처리 로직은 가볍게 유지하고, 입력·설정·권한·보관·추적은 플랫폼이 담당한다.
3. 회사 공용 설정과 개인 사용자 설정을 함께 지원한다.
4. 자동화 결과와 원본 자료를 권한 기반으로 안전하게 보관하고 조회한다.
5. 모든 자동화는 입력/트리거, 프로세서, 결과/보관, 운영/예외/추적의 4단계 구조를 따른다.
6. 자동화별 도입 난이도와 보관 책임에 따라 결과/보관 수준을 선택할 수 있게 한다.

---

## 2. 엔팔라이언트 솔루션의 구성

엔팔라이언트 솔루션은 다음 구성 요소로 나뉜다.

```text
엔팔라이언트 애플리케이션
엔팔라이언트 게이트웨이
엔팔라이언트 워크플로우
엔팔라이언트 스토리지
엔팔라이언트 운영 정책
```

### 2.1 엔팔라이언트 애플리케이션

엔팔라이언트 애플리케이션은 사용자, 회사관리자, 오퍼레이터가 사용하는 웹/앱 화면이다.

담당 역할은 다음과 같다.

```text
로그인
회사 가입 요청
사용자 승인
고객사 관리
워크플로우 등록
워크플로우 사용권 배정
회사 공용 설정 관리
개인 설정 관리
실행 요청
실행 결과 조회
결과 상세 화면
권한 기반 다운로드
운영자 관리 화면
```

### 2.2 엔팔라이언트 게이트웨이

엔팔라이언트 게이트웨이는 애플리케이션과 n8n 사이의 서버 API 계층이다.

담당 역할은 다음과 같다.

```text
사용자 인증 검증
회사 및 사용자 권한 확인
워크플로우 사용권 확인
회사 공용 설정과 개인 설정 병합
보관 정책 해석
canonical payload 생성
파일 수신
보관 정책에 따른 원본 파일 처리
n8n 서버 간 호출
callback 수신
Firestore 실행 이력 갱신
다운로드 권한 검증
외부 API 연동 진입점 관리
```

### 2.3 엔팔라이언트 워크플로우

엔팔라이언트 워크플로우는 n8n에서 실행되는 자동화 프로세서다.

담당 역할은 다음과 같다.

```text
표준 payload 수신
processorInput 생성
실제 업무 처리
LLM 호출
문서 생성
요약
변환
구조화
processorResult 생성
결과/보관 정책에 따른 결과 처리 분기
success / failed / config_error callback 전송
```

엔팔라이언트 워크플로우는 사용자 인증, 회사 권한 판단, 개인/회사 설정 병합, Firestore 직접 수정, 사용자별 다운로드 권한 판단을 담당하지 않는다.

### 2.4 엔팔라이언트 스토리지

엔팔라이언트 스토리지는 원본 파일, 결과 파일, 메타데이터, 실행 로그를 보관하는 저장 계층이다.

기본 원칙은 다음과 같다.

```text
Firestore = 상태, 권한, 설정, 실행 이력, 결과 메타데이터
Firebase Storage = 원본 파일, 결과 파일, 다운로드 대상 파일
Google Drive = Watch 입력 또는 Optional Export 저장소
Email = 알림 및 경량 보관함으로 사용 가능
```

Firebase Storage는 엔팔라이언트의 기본 파일 보관 계층이다.  
다만 모든 자동화가 원본 파일을 영구 보관해야 하는 것은 아니며, 원본 보관 여부는 결과/보관 레벨 정책에 따른다.

---

## 3. 4단계 자동화 원칙

엔팔라이언트의 모든 자동화는 다음 4단계 구조로 설계한다.

```text
1. 입력/트리거 영역
2. 프로세서 영역
3. 결과/보관 영역
4. 운영/예외/추적 영역
```

자동화별 핵심 차이는 원칙적으로 프로세서 영역에서 발생한다.  
입력 방식, 저장 방식, 결과 조회, 예외 처리, 권한 확인은 가능한 한 엔팔라이언트 공통 기능으로 흡수한다.

---

## 4. 1단계: 입력/트리거 영역

입력/트리거 영역은 자동화 실행 요청을 받는 영역이다.

이 영역의 목적은 다양한 실행 수단을 하나의 표준 입력 구조로 정규화하는 것이다.

표준 트리거 유형은 다음과 같다.

```text
manual
scheduled
webhook
watch
retry
system
```

표준 입력 유형은 다음과 같다.

```text
text
file
audio
image
form
url
```

Google Drive Watch는 제거 대상이 아니다.  
Google Drive Watch는 입력/트리거 영역의 입력 어댑터로 인정한다.

---

## 5. 2단계: 프로세서 영역

프로세서 영역은 자동화의 본체다.

프로세서는 정규화된 입력을 받아 실제 업무 결과를 만든다.

```text
processorInput → 업무 처리 → processorResult
```

프로세서는 입력 수단이나 저장 방식에 직접 의존하지 않아야 한다.

프로세서는 Google Drive 폴더 ID, 이메일 수신자, 사용자 권한, 회사 설정 병합, 장기 보관 정책을 직접 판단하지 않는다.

---

## 6. 3단계: 결과/보관 영역

결과/보관 영역은 프로세서가 만든 결과를 어디에 저장하고 어떻게 사용자에게 보여줄지 결정하는 영역이다.

이 영역은 자동화별 핵심 업무 로직이 아니라 플랫폼의 보관 정책과 출력 정책에 속한다.

### 6.1 기본 보관 원칙

엔팔라이언트의 기본 결과 보관 원칙은 다음과 같다.

```text
모든 실행은 최소 운영 추적 로그를 남긴다.
가공 결과와 원본 파일의 보관 범위는 자동화별 보관 레벨에 따라 달라진다.
결과 상세 화면은 엔팔라이언트 애플리케이션에서 제공한다.
다운로드는 엔팔라이언트 권한 확인 후 제공한다.
Google Drive는 기본 저장소가 아니라 선택 export 저장소로 둔다.
```

### 6.2 결과/보관 레벨 정책

엔팔라이언트는 자동화별로 결과/보관 수준을 선택할 수 있다.

표준 레벨은 다음 세 가지다.

```text
Level 1. notify_only        = 알림/로그형
Level 2. processed_result   = 가공지식 저장형
Level 3. full_archive       = 원본 포함 지식보관형
```

#### Level 1. 알림/로그형

가장 가벼운 결과 처리 방식이다.

자동화 결과는 화면 로그 또는 이메일로 전달하고, 엔팔라이언트에는 최소 실행 이력만 남긴다.

원본 파일과 processorResult 전체 본문은 영구 보관하지 않을 수 있다.

#### Level 2. 가공지식 저장형

자동화 결과로 생성된 가공 지식은 엔팔라이언트 DB에 저장하되, 원본 파일은 보관하지 않는 방식이다.

processorResult, 구조화 데이터, 키워드, 경고 정보 등을 저장할 수 있다.

원본 음성, 이미지, 파일은 처리 후 폐기할 수 있다.

#### Level 3. 원본 포함 지식보관형

가공된 지식과 함께 원본 파일까지 보관하는 가장 높은 수준의 보관 방식이다.

Firebase Storage에 원본 파일과 결과 파일을 저장하고, Firestore에는 originalFileRefs와 resultRefs를 남긴다.

원본 보관 실패는 실행 실패로 처리할 수 있다.

### 6.3 Google Drive의 지위

Google Drive는 다음 용도로 사용할 수 있다.

```text
입력 Watch 소스
개인 실험형 워크플로우의 임시 저장소
고객 요청에 따른 선택 export 저장소
기존 자동화 호환을 위한 과도기적 저장소
```

Google Drive 저장 실패가 엔팔라이언트 기본 보관 실패로 이어져서는 안 된다.  
단, 해당 자동화가 Google Drive export 자체를 핵심 결과로 정의한 경우에는 별도 정책으로 실패 처리할 수 있다.

---

## 7. 4단계: 운영/예외/추적 영역

운영/예외/추적 영역은 자동화 실행의 상태와 장애 대응을 담당한다.

처리 대상은 다음과 같다.

```text
success callback
failed callback
config_error callback
실행 로그
재시도 정보
권한 오류
파일 초과 오류
외부 API 오류
보관 정책 오류
삭제 정책
감사 로그
```

submission은 최소한 다음 상태를 가진다.

```text
queued
processing
success
failed
skipped
config_error
```

n8n은 Firestore submissions를 직접 수정하지 않는다.  
결과 반영은 엔팔라이언트 Gateway callback API를 통해 수행한다.

---

## 8. 표준 데이터 계약

엔팔라이언트와 n8n 사이의 데이터 계약은 다음 네 가지로 표준화한다.

```text
canonicalPayload
retentionPolicy
processorInput
processorResult
```

### 8.1 canonicalPayload

canonicalPayload는 엔팔라이언트가 실행 요청 전체를 정규화한 표준 실행 요청이다.

포함 성격은 다음과 같다.

```text
submissionId
clientId
uid
workflowKey
automationId
trigger
settings
input
retentionPolicy
requestedAt
callbackUrl
```

### 8.2 retentionPolicy

retentionPolicy는 해당 실행 결과를 어떤 수준으로 보관할지 결정하는 표준 보관 정책이다.

포함 성격은 다음과 같다.

```text
level
emailEnabled
storeProcessorResult
storeOriginalFiles
storageProvider
optionalExportProvider
```

### 8.3 processorInput

processorInput은 n8n 프로세서가 실제 처리에 필요한 입력만 추출한 값이다.

### 8.4 processorResult

processorResult는 n8n이 처리 후 엔팔라이언트에 반환하는 표준 결과다.

---

## 9. 웹/앱 역할 및 권한 위임 정책

엔팔라이언트 웹/앱은 세 가지 역할로 나뉜다.

```text
오퍼레이터
회사관리자
개인사용자
```

권한과 책임은 오퍼레이터에서 회사관리자, 회사관리자에서 개인사용자로 위임된다.

```text
오퍼레이터 → 회사관리자 → 개인사용자
```

정책 우선순위는 다음과 같다.

```text
오퍼레이터 정책 > 회사관리자 정책 > 개인사용자 설정
```

실행 설정값의 적용 우선순위는 다음과 같다.

```text
개인사용자 설정 > 회사 공용 설정 > 워크플로우 기본값
```

단, 개인사용자 설정은 오퍼레이터와 회사관리자가 허용한 항목에 한해 적용된다.

---

## 10. 오퍼레이터 역할

오퍼레이터는 엔팔라이언트 솔루션을 운영하는 개발회사 또는 서비스 제공자의 최종 관리자다.

오퍼레이터는 자동화의 생산자이며, 플랫폼의 기준을 정의한다.

오퍼레이터는 다음을 할 수 있다.

```text
고객사 생성 및 관리
회사관리자 지정 및 권한 관리
워크플로우 마스터 등록
워크플로우 버전 관리
워크플로우 배포, 중지, 비공개 전환
inputSchema 정의
configSchema 정의
기본 보관 정책 정의
결과/보관/리포팅 정책 정의
회사별 워크플로우 사용권 배정
운영 로그 및 오류 추적
```

오퍼레이터는 Credential, API Key, Token 같은 비밀값을 회사관리자나 개인사용자 설정값으로 노출하면 안 된다.

---

## 11. 회사관리자 역할

회사관리자는 고객사 내부의 책임자다.

회사관리자는 오퍼레이터가 배포한 워크플로우 사용권을 바탕으로 자기 회사에 맞는 회사 공용 설정을 등록하고, 회사 소속 사용자에게 배포한다.

회사관리자는 다음을 할 수 있다.

```text
자기 회사 사용자 승인 및 거절
자기 회사 사용자 목록 조회
자기 회사에 배정된 워크플로우 목록 확인
워크플로우별 회사 공용 기본 설정 등록
회사 공용 설정 수정
회사 내 워크플로우 활성화/비활성화
회사 공용 보관 정책 확인 및 허용된 범위 내 설정
회사 사용자 실행 결과 조회
```

회사관리자는 다른 회사의 사용자, 설정, 실행 결과를 조회하거나 수정할 수 없다.

---

## 12. 개인사용자 역할

개인사용자는 고객사에 소속되어 실제 워크플로우를 실행하는 사용자다.

개인사용자는 다음을 할 수 있다.

```text
자신에게 허용된 워크플로우 목록 확인
워크플로우 실행
자신의 실행 결과 조회
자신의 개인 설정 등록 및 수정
회사 기본 설정 사용
허용된 경우 재실행 요청
```

개인사용자는 다른 사용자의 실행 결과를 조회할 수 없다.

---

## 13. 설정 변경 원칙

회사관리자는 회사 공용 설정을 언제든지 변경할 수 있다.  
개인사용자는 허용된 범위 안에서 개인 설정을 언제든지 변경할 수 있다.

다만 설정 변경은 변경 이후의 새 실행부터 적용된다.

이미 완료된 submissions의 실행 당시 설정값과 결과는 변경하지 않는다.  
실행 당시 적용된 설정 요약과 보관 정책은 추적 가능해야 한다.

---

## 14. 워크플로우 배포 상태

워크플로우는 다음 상태를 가질 수 있다.

```text
draft
published
assigned
configured
enabled
disabled
deprecated
archived
```

개인사용자는 configured 및 enabled 상태의 워크플로우만 실행할 수 있다.

---

## 15. 실험형 워크플로우 PoC 원칙

엔팔라이언트 표준 편입 전 단계에서는 개인 n8n 환경에서 빠르게 실험형 워크플로우를 만들 수 있다.

이때 Google Drive Watch, 개인 Google Drive 저장, 개인 Gmail 발송 등 단순한 입출력 방식을 임시로 허용한다.

다만 향후 엔팔라이언트 편입 가능성이 있는 워크플로우는 처음부터 입력/트리거, 프로세서, 결과/보관, 운영/예외/추적 영역을 분리해야 한다.

PoC 단계의 Google Drive Watch는 입력 어댑터로 본다.  
개인 Google Drive 저장은 Optional Export로 본다.  
이메일 발송은 notify_only 수준의 결과 전달 방식으로 볼 수 있다.

---

## 16. n8n 워크플로우 설계 원칙

모든 엔팔라이언트 워크플로우는 4단계 구조가 노드명과 Sticky Note에 드러나야 한다.

권장 노드 구간은 다음과 같다.

```text
[01] Trigger / Input
[02] Payload Normalize
[03] Input Validate
[04] File Normalize

[10] Processor Start
[11] AI / Transform
[12] Parse Result
[13] Build Processor Result

[20] Result Policy Router
[21] Notify / Email
[22] Store Processor Result
[23] Optional Export
[24] Callback

[90] Error / Exception
```

기존 정상 동작 중인 업무 처리 노드는 불필요하게 수정하지 않는다.  
입력 노드와 결과 노드는 공통 구조에 맞춰 교체 가능하게 만든다.

---

## 17. 표준 위반 사례

다음 구조는 엔팔라이언트 표준 위반으로 본다.

```text
n8n이 Firestore를 직접 조회하거나 수정하는 구조
n8n이 사용자 권한을 판단하는 구조
n8n이 회사/개인 설정 병합을 수행하는 구조
Google Drive를 기본 원본 저장소로 전제하는 구조
이메일에 Firebase Storage 공개 URL을 직접 넣는 구조
프로세서가 특정 입력 수단에 강하게 묶이는 구조
프로세서가 저장 위치나 발송 정책을 직접 판단하는 구조
Credential, Token, API Key가 회사관리자 또는 개인사용자 설정값으로 노출되는 구조
보관 레벨과 무관하게 모든 원본을 무조건 영구 보관하는 구조
```

---

## 18. 변경 관리 원칙

엔팔라이언트 애플리케이션, 게이트웨이, 스토리지, 워크플로우 수정은 본 헌법의 책임 경계를 기준으로 판단한다.

새 기능을 추가할 때는 먼저 해당 기능이 다음 중 어디에 속하는지 분류한다.

```text
입력/트리거
프로세서
결과/보관
운영/예외/추적
```

기능의 위치가 불명확한 경우 구현보다 분류를 먼저 한다.  
분류 없이 기능을 추가하지 않는다.

## 19. 적용 순서

본 헌법은 다음 순서로 적용한다.

```text
1. 정책 및 아키텍처 문서 확정
2. 엔팔라이언트 애플리케이션과 게이트웨이를 4단계 책임 경계에 맞게 수정
3. 결과/보관 레벨 정책을 워크플로우 템플릿과 회사 설정에 반영
4. 엔팔라이언트 스토리지 정책을 Firestore와 Firebase Storage 기준으로 정비
5. 기존 n8n 워크플로우를 프로세서 중심 구조로 재정렬
6. 신규 워크플로우는 처음부터 본 헌법을 기준으로 설계
7. 기존 워크플로우는 중요도와 사용 빈도에 따라 순차 이관
```

## 20. 최종 원칙

엔팔라이언트는 자동화 플랫폼이다.  
n8n은 자동화 프로세서다.

엔팔라이언트는 입력, 설정, 권한, 보관, 결과 조회, 예외 추적을 담당한다.  
n8n은 정규화된 입력을 받아 업무 결과를 만드는 데 집중한다.

기본 저장소는 Firestore와 Firebase Storage다.  
Google Drive는 Watch 입력 또는 선택 export다.

모든 자동화는 4단계 구조를 따른다.  
모든 권한은 오퍼레이터에서 회사관리자, 회사관리자에서 개인사용자로 위임된다.  
모든 결과는 권한 기반으로 조회되어야 한다.

자동화 결과와 원본의 보관 범위는 `resultRetentionLevel`에 따라 조절한다.
