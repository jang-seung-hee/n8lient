# 엔팔라이언트 결과 DB화/지식검색 최소 계약 v0.1

- 문서명: 엔팔라이언트 결과 DB화/지식검색 최소 계약
- 버전: v0.1
- 상위 문서: 엔팔라이언트 솔루션 헌법, 아키텍처 명세서, 결과/보관 레벨 최소 계약, DB/Storage 최소 계약
- 작성일: 2026-06-12
- 목적: 워크플로우별 결과 저장 형식이 과도하게 흩어지지 않도록 최소 공통 DB 포맷과 검색 준비 구조를 정의한다.

---

## 1. 기본 철학

엔팔라이언트는 처음부터 거대한 지식관리 시스템을 만들지 않는다.

다만 여러 워크플로우의 결과가 쌓인 뒤에도 최소 검색, 필터링, 후가공 인덱싱이 가능하도록 공통 결과 껍데기는 맞춘다.

```text
과도한 표준화는 피한다.
최소한의 공통 구조는 둔다.
후가공성 데이터는 실행 결과와 분리한다.
통합 검색/벡터/백링크는 나중에 별도 인덱싱 단계에서 처리한다.
```

---

## 2. 적용 범위

이 문서는 다음 자동화에 적용한다.

```text
아이디어 캐처
통화 요약
회의록
업무위키
문서 요약
이미지/음성/파일 분석
그 외 Level 2 이상 결과를 DB에 저장하는 워크플로우
```

Level 1 `notify_only`는 이메일/알림 중심이므로 자유 포맷을 허용한다.

Level 2 `processed_result`와 Level 3 `full_archive`부터 최소 공통 DB 포맷을 적용한다.

---

## 3. 레벨별 DB화 기준

### 3.1 notify_only

```text
이메일/알림 중심
자유 포맷 허용
최소 submission 로그 저장
result.summary 저장 가능
processorResult 전체 저장 생략 가능
검색/벡터화 대상에서 기본 제외
```

### 3.2 processed_result

```text
processorResult 저장
최소 공통 검색 필드 적용
원본 파일 영구 보관 없음
향후 knowledgeSearchIndex 생성 대상 가능
```

### 3.3 full_archive

```text
processorResult 저장
originalFileRefs 저장
resultRefs 저장 가능
원본 파일 Firebase Storage 보관
향후 knowledgeSearchIndex 및 embeddingIndex 생성 대상 가능
```

---

## 4. 최소 공통 processorResult 구조

Level 2 이상에서 n8n 워크플로우는 가능한 한 아래 구조를 callback으로 반환한다.

```json
{
  "title": "결과 제목",
  "summary": "짧은 요약",
  "content": "본문 텍스트",
  "mdContent": "마크다운 본문",
  "hashtags": ["태그1", "태그2"],
  "attachments": [],
  "structuredData": {},
  "warnings": []
}
```

---

## 5. 필드 의미

### 5.1 title

검색 결과 카드의 제목으로 사용한다.

```text
필수 권장
비어 있으면 summary 또는 input.title로 fallback 가능
```

### 5.2 summary

검색 결과 카드의 짧은 설명으로 사용한다.

```text
선택이지만 권장
짧고 사람이 읽을 수 있어야 한다.
```

### 5.3 content

워크플로우 결과의 본문 텍스트다.

```text
검색 1차 대상
마크다운 문법이 제거된 순수 텍스트를 권장
```

### 5.4 mdContent

마크다운 표시용 본문이다.

```text
결과 상세 화면 표시용
Markdown Export 또는 Optional Export에 활용 가능
```

### 5.5 hashtags

간단한 해시태그 배열이다.

```text
선택 필드
워크플로우가 자연스럽게 만들 수 있을 때만 사용
무리한 자동 분류나 지식 그래프 대체 용도로 쓰지 않는다.
```

아이디어 캐처처럼 원본성 보존이 중요한 워크플로우는 `hashtags`를 빈 배열로 둘 수 있다.

### 5.6 attachments

결과와 관련된 첨부파일 참조다.

```json
[
  {
    "type": "original",
    "refType": "original",
    "index": 0,
    "fileName": "audio.mp3",
    "mimeType": "audio/mpeg",
    "sizeBytes": 123456
  }
]
```

실제 다운로드는 `originalFileRefs`, `resultRefs`와 Gateway 다운로드 API를 기준으로 처리한다.

### 5.7 structuredData

워크플로우별 자유 확장 영역이다.

```text
회의록의 참석자 목록
통화 요약의 고객명/문의유형
문서 분석의 항목별 추출값
업무위키의 문서 타입
```

공통 검색에 반드시 필요한 필드는 `structuredData` 안에만 숨기지 말고 title/content/summary/hashtags에도 반영한다.

### 5.8 warnings

처리 중 주의할 점, 누락, 불확실성을 담는다.

```text
음성 품질 낮음
화자 구분 불명확
첨부파일 일부 미처리
외부 export 실패
```

## 7. 검색 1차 대상

초기 검색은 Firestore/클라이언트 수준의 단순 검색 또는 후속 인덱스를 전제로 다음 필드를 우선 대상으로 한다.

```text
processorResult.title
processorResult.summary
processorResult.content
processorResult.mdContent
processorResult.hashtags
authorName
workflowName
companyName
createdAt
```

검색 결과 카드에는 다음 라벨을 표시한다.

```text
제목
요약
워크플로우명
작성자
회사명
생성일
첨부파일 여부
보관 레벨
```

## 8. 후가공 분리 원칙

워크플로우 실행 단계에서는 다음을 하지 않는다.

```text
벡터라이징
embedding 생성
백링크 생성
지식 그래프 생성
관련 노트 자동 연결
과도한 자동 분류
후가공 규칙에 종속된 태그 대량 생성
```

이유는 다음과 같다.

```text
후가공 규칙은 나중에 바뀔 가능성이 크다.
원본성 결과와 후가공 결과가 섞이면 DB가 오염된다.
검색 인덱스는 언제든 재생성 가능해야 한다.
```

---

## 9. 향후 인덱스 구조

통합 검색 또는 자연어 검색이 필요해지면 submissions를 읽어 별도 인덱스를 만든다.

권장 후보:

```text
knowledgeSearchIndex
embeddingIndex
```

### 9.1 knowledgeSearchIndex 예시

```json
{
  "indexId": "sub_xxx",
  "submissionId": "sub_xxx",
  "clientId": "client_xxx",
  "uid": "user_uid",
  "workflowKey": "idea-catcher",
  "automationId": "auto_xxx",
  "title": "검색 제목",
  "summary": "검색 요약",
  "contentText": "검색 대상 본문",
  "hashtags": ["태그1"],
  "authorName": "작성자",
  "workflowName": "워크플로우명",
  "createdAt": "ISO_8601",
  "sourceUpdatedAt": "ISO_8601",
  "indexVersion": "v1"
}
```

### 9.2 embeddingIndex 예시

```json
{
  "embeddingId": "sub_xxx_chunk_001",
  "submissionId": "sub_xxx",
  "clientId": "client_xxx",
  "workflowKey": "idea-catcher",
  "chunkText": "임베딩 대상 텍스트",
  "chunkIndex": 1,
  "embeddingProvider": "text-embedding-model",
  "embeddingVectorRef": "external_vector_db_id",
  "indexVersion": "v1"
}
```

## 10. n8n 워크플로우 개발 기준

n8n 워크플로우는 Level 2 이상에서 최소한 다음을 지켜야 한다.

```text
processorResult.title 생성
processorResult.summary 생성 권장
processorResult.content 또는 mdContent 생성
hashtags는 필요한 경우만 생성
attachments는 원본/결과 파일 참조를 표현할 때 사용
structuredData는 워크플로우별 자유 확장으로 사용
warnings는 불확실성/누락/실패 보조 정보로 사용
```

n8n은 knowledgeSearchIndex나 embeddingIndex를 직접 만들지 않는다.  
n8n은 원본성 실행 결과를 callback으로 반환한다.

---

## 11. Gateway 저장 기준

Gateway는 callback 수신 후 retentionPolicySnapshot.level에 따라 저장한다.

```text
notify_only:
- result.summary 중심
- processorResult 전체 저장 생략 가능

processed_result:
- processorResult 저장
- 최소 공통 필드가 있으면 그대로 저장

full_archive:
- processorResult 저장
- originalFileRefs/resultRefs와 함께 상세 조회 및 다운로드 가능
```

Gateway는 검색 인덱스를 즉시 만들지 않는다.  
검색 인덱스는 후속 작업으로 분리한다.

---

## 12. UI 표시 기준

결과 상세 화면은 processorResult가 있을 경우 다음 순서로 표시한다.

```text
title
summary
mdContent 또는 content
hashtags
attachments
structuredData
warnings
```

검색 결과 목록 또는 카드형 결과는 다음을 표시한다.

```text
title
summary
workflowName
authorName
companyName
createdAt
retentionPolicySnapshot.level
첨부파일 여부
```

## 13. Optional Export 기준

Obsidian, Google Drive, Markdown 파일 export는 기본 저장소가 아니다.

```text
Firestore = 기준 DB
Firebase Storage = 기준 파일 저장소
Optional Export = 사용자가 원할 때 외부로 내보내는 기능
```

Obsidian은 엔팔라이언트의 기본 지식관리 저장소로 삼지 않는다.  
필요하면 mdContent를 다운로드하거나 Optional Export로 활용한다.

## 14. 하지 말 것

```text
processorResult에 embedding vector 직접 저장 금지
워크플로우 실행 중 knowledgeSearchIndex 직접 생성 금지
워크플로우 실행 중 백링크/지식 그래프 생성 금지
후가공 규칙에 종속된 분류값을 원본 결과처럼 저장 금지
Firestore에 파일 binary/base64 저장 금지
Obsidian 또는 Google Drive를 기본 지식 저장소로 간주 금지
```

## 15. 최소 성공 기준

이 계약이 적용된 워크플로우는 다음을 만족해야 한다.

```text
Level 2 이상에서 processorResult.title 또는 summary가 존재한다.
Level 2 이상에서 processorResult.content 또는 mdContent가 존재한다.
검색 결과 카드에 제목, 요약, 워크플로우명, 작성자, 회사명, 생성일을 표시할 수 있다.
후가공 데이터는 submissions와 분리할 수 있다.
나중에 knowledgeSearchIndex를 재생성할 수 있다.
```
