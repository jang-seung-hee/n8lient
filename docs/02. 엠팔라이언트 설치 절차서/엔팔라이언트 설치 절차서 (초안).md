## 0. 설치 전 전제 조건

이 매뉴얼은 다음 조건이 이미 준비되어 있다는 전제로 작성한다.

- 회사 전용 n8n self-hosted 서버가 설치되어 있다.
    
- n8n 도메인이 준비되어 있다.
    
- n8n 워크플로우 JSON이 import 되어 있다.
    
- n8n 워크플로우의 Webhook path가 확인되어 있다.
    
- n8n Google Drive, Gmail, Gemini 등 필요한 Credential 연결이 완료되어 있다.
    
- n8n Webhook Header Auth Credential 설정이 가능하다.
    
- N8Lient 소스코드 저장소가 준비되어 있다.
    
- Google Cloud, Firebase, Netlify 또는 프론트 배포 환경에 접근할 수 있다.

# 1. 회사 기본 정보 정리

## 목적

설치 대상 회사의 식별값과 도메인, n8n 서버 정보를 먼저 정리한다.  
이 값들이 이후 Firebase, Gateway, N8Lient 운영자 콘솔, n8n 설정에 반복 사용된다.

## 작업

아래 정보를 표로 정리한다.

```text
회사명:
회사 코드:
clientId:
Firebase Project ID:
Google Cloud Project ID:
n8n 서버 주소:
n8nServerKey:
Gateway URL:
Frontend URL:
운영자 이메일:
회사 관리자 이메일:
```

예시:

```text
회사명: 렌탈톡톡
회사 코드: RTT2026
clientId: client_rentaltoktok_001
Firebase Project ID: n8lient-rentaltoktok
Google Cloud Project ID: n8lient-rentaltoktok
n8n 서버 주소: https://n8n.rentaltalk.kr
n8nServerKey: main
Gateway URL: 배포 후 확정
Frontend URL: 배포 후 확정
운영자 이메일: operator@example.com
회사 관리자 이메일: admin@company.com
```

## AI 지시 예시

```text
아래 회사 정보를 기준으로 N8Lient 설치에 필요한 변수 목록을 정리해줘.
clientId, companyCode, n8nServerKey, Gateway 환경변수, Netlify 환경변수에 들어갈 값을 구분해서 표로 만들어줘.

회사명:
회사 코드:
n8n 서버 주소:

```

## 팁

- `clientId`는 사람이 봐도 알아볼 수 있게 만든다.
    
- 예: `client_rentaltoktok_001`, `client_abcfood_001`
    
- 회사코드는 사용자가 가입할 때 입력하는 값이므로 너무 길거나 복잡하지 않게 만든다.
    

## 주의사항

- `companyCode`는 외부 사용자도 입력할 수 있으므로 예측하기 너무 쉬운 값은 피한다.
    
- `n8nServerKey`는 보통 `main`으로 시작한다.
    
- 여러 n8n 서버를 운영할 때만 `sub`, `clientA` 같은 키를 추가한다.

# 2. Firebase 프로젝트 생성 및 설정

## 목적

N8Lient의 로그인, 사용자 정보, 회사 설정, 실행 로그를 관리할 Firebase 프로젝트를 준비한다.

## 작업

Firebase Console에서 회사 전용 프로젝트를 생성한다.

필수 설정:

```text
1. Firebase Project 생성
2. Authentication 활성화
3. Google 로그인 Provider 활성화
4. Firestore Database 생성
5. Web App 추가
6. Firebase Web SDK 설정값 확보
7. Firebase Admin SDK 서비스 계정 키 생성
```

## 필요한 값

프론트용 공개값:

```env
NEXT_PUBLIC_FIREBASE_API_KEY=
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
NEXT_PUBLIC_FIREBASE_APP_ID=
```

Gateway 서버용 비밀값:

```yaml
FIREBASE_ADMIN_PROJECT_ID:
FIREBASE_ADMIN_CLIENT_EMAIL:
FIREBASE_ADMIN_PRIVATE_KEY:
```

## AI 지시 예시

```text
Firebase 프로젝트 설정을 진행하려고 한다.
N8Lient 설치용으로 Firebase Authentication, Firestore, Web App, Admin SDK 키를 준비해야 한다.
내가 Firebase Console 화면을 캡처하거나 설정값을 붙여넣으면, 어떤 값이 .env.local에 들어가고 어떤 값이 env.yaml에 들어가는지 구분해서 정리해줘.
```

## 팁

- Firebase Web SDK 값은 공개값이므로 프론트에 들어가도 된다.
    
- Firebase Admin SDK 키는 서버용 비밀값이다.
    
- Admin SDK 키는 Gateway가 사용자 인증과 Firestore 서버 접근을 할 때 사용한다.
    

## 주의사항

- `FIREBASE_ADMIN_PRIVATE_KEY`는 절대 Git에 올리면 안 된다.
    
- `.env.local`, `env.yaml`, `serviceAccount*.json`은 반드시 `.gitignore`에 포함한다.
    
- Firebase Authentication의 승인 도메인에 프론트 배포 도메인을 추가해야 한다.
    
- Netlify 도메인을 쓰면 `회사서비스.netlify.app` 도메인을 승인 도메인에 추가한다.

# 3. Firestore Rules 준비 및 배포

## 목적

일반 사용자, 회사 관리자, 운영자가 각자 허용된 데이터만 볼 수 있게 제한한다.

## 기본 권한 원칙

```text
일반 user:
- 자기 users 문서
- 자기 submissions
- 자기 userAutomationSettings

company_admin:
- 자기 clientId의 users
- 자기 clientId의 submissions
- 자기 clientId의 clientAutomations

operator:
- 전체 운영 데이터
```

## 작업

프로젝트의 `firestore.rules`를 회사 Firebase 프로젝트에 배포한다.

```powershell
firebase deploy --only firestore:rules --project 회사_firebase_project_id
```

## AI 지시 예시

```text
N8Lient Firestore Rules를 회사 전용 Firebase 프로젝트에 배포하려고 한다.
현재 firestore.rules를 기준으로 user, company_admin, operator 권한이 올바르게 제한되어 있는지 점검하고, 배포 명령어를 알려줘.
```

## 팁

- Gateway는 Firebase Admin SDK를 사용하므로 Firestore Rules 제한을 받지 않는다.
    
- 프론트에서 직접 읽는 화면은 Rules 영향을 받는다.
    
- Rules를 너무 세게 잠그면 화면 목록이 안 보일 수 있다.
    

## 주의사항

- 테스트 중 편하다고 `allow read: if isAuthenticated()`처럼 풀어두면 안 된다.
    
- submissions는 개인정보와 실행 내용이 들어가므로 타인 조회를 막아야 한다.
    
- 회사 관리자는 자기 회사 데이터만 봐야 한다.

# 5. 환경변수 파일 준비

## 목적

프론트용 공개 설정과 Gateway용 비밀 설정을 분리한다.

## 파일 구분

```text
.env.local
= 프론트엔드용 공개 설정

env.yaml
= Cloud Run Gateway 서버용 비밀 설정
```

## .env.local 예시

```env
# Gateway API Base URL
NEXT_PUBLIC_GATEWAY_BASE_URL=https://회사-gateway-url.run.app

# Firebase Web SDK 공개 설정
NEXT_PUBLIC_FIREBASE_API_KEY=
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
NEXT_PUBLIC_FIREBASE_APP_ID=
```

## env.yaml 예시

```yaml
# Firebase Admin SDK 설정
FIREBASE_ADMIN_PROJECT_ID: "firebase_project_id"
FIREBASE_ADMIN_CLIENT_EMAIL: "firebase_admin_client_email"
FIREBASE_ADMIN_PRIVATE_KEY: "-----BEGIN PRIVATE KEY-----\\n...\\n-----END PRIVATE KEY-----\\n"

# n8n 서버 설정
N8N_SERVER_MAIN_BASE_URL: "https://회사-n8n-domain.com"
N8N_SERVER_MAIN_TOKEN: "n8n_header_auth_token"

# n8n callback 검증 Secret
N8N_CALLBACK_SECRET: "callback_secret"

# Gateway 자체 URL
GATEWAY_BASE_URL: "https://회사-gateway-url.run.app"

# 파일 업로드 제한
MAX_UPLOAD_MB: "10"

# Gateway CORS 허용 도메인
ALLOWED_ORIGINS: "https://회사-frontend-domain.com,http://localhost:3000"
```

## AI 지시 예시

```text
아래 Firebase 설정값과 n8n 서버 정보를 기준으로 .env.local과 env.yaml을 나눠서 작성해줘.
프론트에 노출되면 안 되는 값은 env.yaml로 분리하고, 각 항목에 주석을 달아줘.

Firebase Web SDK:
Firebase Admin SDK:
n8n 서버 주소:
n8n Header Auth Token:
Callback Secret:
프론트 도메인:
Gateway 도메인:
```

## 팁

- `.env.local`은 Netlify 환경변수로 그대로 옮길 값이다.
    
- `env.yaml`은 Cloud Run 배포 시 `--env-vars-file`로 들어갈 값이다.
    
- 새 워크플로우를 추가할 때마다 `env.yaml`을 수정하지 않는다.
    
- Gateway는 `webhookSecretId`를 보고 `/webhook/{webhookSecretId}`를 자동 조합한다.
    

## 주의사항

아래 값은 `.env.local`에 절대 넣지 않는다.

```text
FIREBASE_ADMIN_PRIVATE_KEY
N8N_SERVER_MAIN_TOKEN
N8N_CALLBACK_SECRET
N8N_SERVER_MAIN_BASE_URL
```

아래 파일은 Git에 절대 올리지 않는다.

```text
.env.local
env.yaml
serviceAccount*.json
*.pem
```

# 6. Cloud Run Gateway 배포

## 목적

파일 업로드와 실행 요청을 처리할 Gateway API 서버를 Cloud Run에 배포한다.

### ⚠️ Gateway 배포 전 필수 주의사항

Gateway는 App의 유효성 검사 로직(`validateExecution.ts`)을 공유합니다. Cloud Run 배포 환경에서는 상위 디렉토리 접근이 불가능하므로, **배포 전 반드시 로컬에서 동기화 스크립트를 실행**하여 소스를 최신화해야 합니다.

1. **공유 파일 동기화**: `npm run sync:gateway` (프로젝트 루트에서 실행)
2. **동기화 상태 확인**: `npm run check:gateway-sync`
3. **Gateway 빌드 테스트**: `cd n8lient-gateway && npm run build`

## 작업

프로젝트 루트에서 배포한다.

```powershell
gcloud run deploy n8lient-gateway `
  --source ./n8lient-gateway `
  --port 8080 `
  --env-vars-file ./env.yaml `
  --allow-unauthenticated `
  --min-instances 1 `
  --region asia-northeast3 `
  --project 회사_구글클라우드_프로젝트ID
```

배포 후 Service URL을 확인한다.

```text
https://n8lient-gateway-xxxxx.run.app
```

헬스체크한다.

```powershell
curl https://회사-gateway-url.run.app/health
```

정상 응답:

```text
OK
```

## AI 지시 예시

```text
Cloud Run Gateway를 배포하려고 한다.
현재 프로젝트 루트에 env.yaml이 있고, Gateway 소스는 ./n8lient-gateway 폴더에 있다.
아래 조건으로 gcloud 배포 명령어를 만들고, 배포 후 확인해야 할 명령어까지 알려줘.

project:
region:
service name:
source:
env file:
```

## 팁

- 첫 배포 후 Gateway URL이 확정되면 `env.yaml`의 `GATEWAY_BASE_URL`에 반영하고 다시 배포한다.
    
- `--min-instances 1`은 콜드 스타트를 줄이지만 비용이 발생한다.
    
- 초기 테스트 중 비용을 아끼려면 `--min-instances 1`을 빼고 테스트할 수도 있다.
    

## 주의사항

- `--env-vars-file`은 Cloud Run 환경변수를 파일 기준으로 덮어쓴다.
    
- `env.yaml`에 필요한 값이 빠지면 기존 값도 사라질 수 있다.
    
- 민감값이 Cloud Audit Log에 노출될 수 있으므로 장기적으로는 Secret Manager 이전을 검토한다.
    
- 배포 후 `env.yaml`을 Git에 올리지 않았는지 반드시 확인한다.

# 7. 프론트엔드 배포

## 목적

사용자가 접속할 N8Lient 웹 화면을 배포한다.

## 작업

Netlify를 사용하는 경우 환경변수에 아래 값을 등록한다.

```env
NEXT_PUBLIC_GATEWAY_BASE_URL=https://회사-gateway-url.run.app
NEXT_PUBLIC_FIREBASE_API_KEY=
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
NEXT_PUBLIC_FIREBASE_APP_ID=
```

## AI 지시 예시

```text
N8Lient 프론트를 Netlify에 배포하려고 한다.
아래 .env.local 값을 기준으로 Netlify 환경변수에 넣어야 할 항목만 정리해줘.
비밀값이 섞여 있는지 점검해줘.
```

## 팁

- Netlify 환경변수는 `.env.local`과 같은 값으로 맞추면 된다.
    
- Netlify 배포 후 실제 프론트 URL을 Gateway `ALLOWED_ORIGINS`에 포함해야 한다.

## 주의사항

- Netlify에 `N8N_SERVER_MAIN_TOKEN`, `N8N_CALLBACK_SECRET`, `FIREBASE_ADMIN_PRIVATE_KEY`를 넣으면 안 된다.
    
- 환경변수를 수정한 뒤에는 재배포해야 반영된다.

# 8. n8n 워크플로우 최종 확인

## 목적

Gateway가 보낸 요청을 n8n이 안전하게 받고, 처리 후 Gateway callback으로 결과를 돌려주게 한다.

## 작업

n8n에서 워크플로우별로 확인한다.

```text
1. Webhook path 확인
2. Header Auth Credential 연결 확인
3. Google Drive/Gmail/Gemini Credential 연결 확인
4. Callback HTTP Request Credential 확인
5. Callback URL이 payload.callbackUrl을 사용하는지 확인
```

## Webhook Header Auth

```text
Credential Type: Header Auth
Header Name: X-N8N-TOKEN
Header Value: Gateway env.yaml의 N8N_SERVER_MAIN_TOKEN 값
```

## Callback Header Auth

```text
Credential Type: Header Auth
Header Name: Authorization
Header Value: Bearer {Gateway env.yaml의 N8N_CALLBACK_SECRET}
```

## AI 지시 예시

```text
n8n 워크플로우가 N8Lient Gateway와 연동 가능한지 확인하려고 한다.
아래 항목을 기준으로 점검 체크리스트를 만들어줘.

- Webhook path
- Header Auth Credential
- X-N8N-TOKEN
- Google Drive/Gmail/Gemini Credential
- Callback URL
- Authorization Bearer Secret
```

## 팁

- n8n Cloud를 써도 서버 `.env` 없이 Header Auth Credential로 처리할 수 있다.
    
- Webhook path는 N8Lient의 `workflowTemplates.webhookSecretId`와 일치해야 한다.
    
- 예: `webhookSecretId = n8lient-idea-catcher`이면 n8n Webhook path도 `n8lient-idea-catcher`.
    

## 주의사항

- n8n 서버 `.env`에 의존하는 구조는 피한다.
    
- n8n Cloud에서는 서버 `.env`를 직접 관리할 수 없으므로 Credential 기반이 더 안전하다.
    
- Callback URL을 하드코딩하지 말고 가능하면 `payload.callbackUrl`을 사용한다.
    
- Header Auth Credential 값과 Gateway `N8N_SERVER_MAIN_TOKEN`이 다르면 n8n 수신이 실패한다.
    
- Callback Secret이 다르면 submissions 상태가 갱신되지 않는다.

# 9. 초기 운영자 계정 생성

```text
N8Lient 초기 운영자 계정을 만들어야 한다.
```

# 11. N8N 워크플로우 마스터 등록

## 목적

운영자가 사용할 수 있는 n8n 워크플로우 명세를 N8Lient에 등록한다.

## 작업

운영자 콘솔에서 `workflowTemplates`를 등록한다.

필수값:

```text
workflowKey
name
shortName
status
n8nServerKey
webhookSecretId
inputSchema
configSchema
```

예시:

```json
{
  "workflowKey": "idea-catcher",

  "webhookSecretId": "n8lient-idea-catcher",
  "inputSchema": {
    "acceptedInputTypes": ["text", "audio"],
    "allowedFileTypes": ["txt", "md", "webm", "mp3", "m4a", "wav"],
    "maxFileSizeMB": 10
  },
  "configSchema": [
    {
      "key": "mdFolderId",
      "label": "마크다운 저장 폴더 ID",
      "type": "text",
      "required": true
    },
    {
      "key": "originalFileFolderId",
      "label": "원본 파일 저장 폴더 ID",
      "type": "text",
      "required": true
    },
    {
      "key": "reportEmailTo",
      "label": "결과 보고 수신 이메일",
      "type": "email",
      "required": false
    }

}
```

## AI 지시 예시

```text
아래 n8n 워크플로우 정보를 기준으로 N8Lient workflowTemplates 등록용 명세를 만들어줘.
settings로 받아야 할 값은 configSchema로 정리하고, 사용자 입력값은 inputSchema로 분리해줘.

워크플로우 이름:

필요 입력:
```

- `webhookSecretId`는 n8n Webhook path와 같게 둔다.
    
- Gateway는 `/webhook/{webhookSecretId}`로 자동 조합한다.
    
- 새 워크플로우를 추가할 때 `env.yaml` 수정은 기본적으로 필요 없다.
    

## 주의사항

- `configSchema.key`와 n8n에서 참조하는 `settings.xxx`는 반드시 같아야 한다.
    
- 예: n8n이 `settings.mdFolderId`를 쓰면 configSchema key도 `mdFolderId`.
    
- `status`가 `published`가 아니면 사용자에게 노출되지 않을 수 있다.

# 15. 일반 사용자 승인 및 개인 설정

## 목적

일반 사용자가 회사가 제공한 워크플로우를 자기 업무 환경에 맞게 실행할 수 있게 한다.

## 작업

1. 일반 사용자가 로그인한다.
    
2. 회사코드로 가입 요청한다.
    
3. 회사 관리자가 승인한다.
    
4. 사용자는 `/user/execute`에서 워크플로우를 선택한다.
    
5. 필요 시 `내 설정`에서 개인 설정값을 등록한다.
    

예시:

```json
{
  "mdFolderId": "개인_MD_폴더_ID",
  "originalFileFolderId": "개인_원본_폴더_ID",
  "reportEmailTo": "user@example.com"
}
```

## 우선순위

```text
사용자 개인 설정 > 회사 공용 설정
```

빈 값은 회사 공용 설정으로 fallback된다.

## AI 지시 예시

```text
일반 사용자 개인 설정 가이드를 만들려고 한다.
아래 워크플로우 configSchema를 기준으로 사용자가 개인 설정에 넣을 수 있는 값과, 비워두면 회사 기본값이 적용되는 항목을 설명해줘.
```

## 팁

- 개인 Google Drive 폴더를 쓰려면 그 폴더를 n8n 공용 Google 계정에 공유해야 한다.
    
- 개인 설정을 안 해도 회사 공용 설정으로 실행 가능해야 한다.
    

## 주의사항

- 개인 설정에 API Key, Token, Credential ID 같은 비밀값을 넣지 않는다.
    
- 개인 폴더 권한 공유를 안 하면 n8n이 파일을 저장하지 못한다.
    
- 사용자가 개인 설정을 비워둔 항목은 회사 설정을 사용한다.

# 16. 텍스트 실행 테스트

## 목적

파일 없이 기본 실행 흐름이 정상 작동하는지 확인한다.

## 작업

1. `/user/execute` 접속
    
2. 워크플로우 선택
    
3. 제목과 텍스트 입력
    
4. 실행 요청
    
5. Gateway 로그 확인
    
6. n8n 실행 이력 확인
    
7. N8Lient 실행 로그 확인
    

## Gateway 로그 확인

```powershell
gcloud logging read 'resource.type="cloud_run_revision" AND resource.labels.service_name="n8lient-gateway" AND (logName="projects/PROJECT_ID/logs/run.googleapis.com%2Fstderr" OR logName="projects/PROJECT_ID/logs/run.googleapis.com%2Fstdout")' `
  --limit 50 `
  --format="value(timestamp,textPayload)" `
  --project PROJECT_ID
```

정상 로그:

```text
[execute] n8n Webhook 호출 시작. URL: https://n8n-domain/webhook/webhookSecretId
[execute] n8n Webhook 호출 성공. Status: 200
[callback] submissionId=... 업데이트 완료. Status: success
```

## AI 지시 예시

```text
아래 Cloud Run 로그를 보고 N8Lient Gateway에서 n8n 호출과 callback이 정상인지 판단해줘.
문제가 있다면 원인을 Gateway, n8n, Firestore, 권한 문제로 나눠서 분석해줘.

로그:
```

## 팁

- 텍스트 실행이 먼저 성공해야 파일 실행을 테스트한다.
    
- Gateway가 n8n 호출 성공 200을 받았다고 해서 n8n 처리가 끝난 것은 아니다.
    
- 최종 완료는 callback success로 판단한다.
    

## 주의사항

- n8n Webhook이 200을 줘도 내부 노드에서 실패할 수 있다.
    
- callback이 실패하면 submissions 상태가 계속 processing 또는 queued로 남을 수 있다.

# 17. 파일/음성 실행 테스트

## 목적

파일 업로드, Gateway 임시 저장, n8n 전달, Google Drive 저장, callback까지 확인한다.

## 작업

1. 1~2MB 음성 파일로 먼저 테스트한다.
    
2. 성공하면 7~10MB 파일로 테스트한다.
    
3. Firestore submissions 문서를 확인한다.
    
4. n8n에서 binary `file_0`가 들어왔는지 확인한다.
    
5. Google Drive에 원본 파일이 저장됐는지 확인한다.
    
6. callback success를 확인한다.
    

## Firestore 확인

`submissions.input`에는 메타데이터만 있어야 한다.

```json
{
  "fileName": "audio.webm",
  "mimeType": "audio/webm",
  "sizeBytes": 8234412,
  "inputType": "audio"
}
```

## AI 지시 예시

```text
파일 실행 테스트 결과를 점검하려고 한다.
아래 Gateway 로그, n8n 실행 로그, Firestore submissions 문서를 보고 파일 업로드 흐름이 어디까지 성공했는지 분석해줘.

Gateway 로그:
n8n 로그:
submissions 문서:
```

## 팁

- 처음부터 10MB 파일로 테스트하지 않는다.
    
- 먼저 작은 파일로 성공한 뒤 큰 파일로 올린다.
    
- `MAX_UPLOAD_MB`와 workflowTemplates.inputSchema.maxFileSizeMB를 맞춰두는 것이 좋다.
    

## 주의사항

- Firestore/Firebase Storage에 파일 원본이 저장되면 안 된다.
    
- Gateway의 `/tmp` 파일은 n8n 전송 후 삭제되어야 한다.
    
- n8n 공용 Google 계정이 대상 Drive 폴더에 쓰기 권한을 가져야 한다.
    
- n8n에서 binary 필드명이 `file_0`인지 확인한다.

# 18. 운영 전 보안 점검

## 목적

실제 고객사 운영 전에 노출 위험이 있는 키와 권한을 점검한다.

## 점검 항목

```text
1. env.yaml이 Git에 올라가지 않았는가.
2. .env.local이 Git에 올라가지 않았는가.
3. Firebase Admin Key가 노출되지 않았는가.
4. N8N_SERVER_MAIN_TOKEN이 노출되지 않았는가.
5. N8N_CALLBACK_SECRET이 노출되지 않았는가.
6. operator 권한 사용자가 최소 인원인가.
7. Firestore Rules가 완화되어 있지 않은가.
8. n8n Credential이 회사별로 올바르게 분리되어 있는가.
9. Google Drive 공유 권한이 필요한 폴더에만 부여되어 있는가.
```

## AI 지시 예시

```text
운영 전 보안 점검을 하려고 한다.
아래 파일 목록, git status, Firestore Rules, Cloud Run 환경변수 목록을 보고 민감정보 노출 가능성이 있는지 점검해줘.

git status:
git show --name-only HEAD:
Firestore Rules:
Cloud Run env describe 결과:
```

## 팁

- 테스트 중 대화창, 로그, 캡처에 노출된 키는 운영 전 교체한다.
    
- 장기적으로는 Google Secret Manager를 사용한다.
    
- 고객사별 설치라면 토큰과 callback secret은 회사별로 다르게 만든다.
    

## 주의사항

- Cloud Audit Log에 환경변수 값이 노출될 수 있다.
    
- 실제 운영 전에는 Firebase Admin Key, n8n token, callback secret을 새로 발급하는 것이 안전하다.
    
- `--allow-unauthenticated`는 Cloud Run 접근을 공개하는 옵션이므로 Gateway 내부 인증이 반드시 작동해야 한다.

```text

서비스 URL
포트 8080
컨테이너 기동 로그
```

```text
Firebase Auth Provider
승인 도메인
.env.local Firebase Web SDK 값
Netlify 환경변수 반영 여부
```

## 실행 화면에 워크플로우가 안 보임

확인:

```text
clientContracts 매핑 여부
clientAutomations 존재 여부
enabled = true
configStatus = configured
사용자 approvalStatus = approved
사용자 clientId 일치
```

```text
N8N_SERVER_MAIN_TOKEN
workflowTemplates.n8nServerKey
workflowTemplates.webhookSecretId
n8n Webhook path
Header Auth Credential
```

```text
N8N_CALLBACK_SECRET
Callback HTTP Request Credential
Gateway /api/automation/callback 로그
submissions 문서 권한 및 submissionId
```
